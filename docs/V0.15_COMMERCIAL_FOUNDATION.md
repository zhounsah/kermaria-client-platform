# V0.15 Commercial Foundation

## Objectif

La V0.15 pose un socle commercial prudent pour le portail Kermaria :

- catalogue d'offres administrable ;
- documents commerciaux informatifs ;
- brouillons de devis ou de suivi commercial ;
- lignes de document ;
- affichage client ;
- suivi admin.

Cette version ne crée aucune facture légale définitive.

## Principes

- Mention centrale : `Document informatif - ne constitue pas une facture officielle.`
- La facturation réelle n'est pas encore activée.
- Aucun paiement n'est possible depuis cet espace.
- Ce document sert au suivi commercial interne et à l'information du client.

## Catalogue

Le catalogue repose sur `commercial_offers`.

Champs principaux :

- nom ;
- description ;
- catégorie ;
- unité ;
- prix indicatif en centimes ;
- statut `active|inactive` ;
- ordre d'affichage ;
- dates de création et mise à jour.

Le client peut consulter les offres via le portail. L'admin peut créer, mettre
à jour et désactiver une offre sans suppression définitive.

## Documents commerciaux informatifs

Le stockage repose sur `commercial_documents`.

Types autorisés :

- `quote_draft`
- `billing_draft`
- `informational_invoice`

Statuts autorisés :

- `draft`
- `pending_review`
- `shared_with_customer`
- `cancelled`

`shared_with_customer` signifie uniquement que le document devient visible côté
client. Il ne crée ni facture officielle ni obligation de paiement.

## Lignes

Les lignes sont stockées dans `commercial_document_lines`.

Champs principaux :

- offre liée optionnelle ;
- libellé ;
- description ;
- quantité ;
- unité ;
- prix unitaire en centimes ;
- taux indicatif optionnel en basis points ;
- total de ligne en centimes ;
- ordre d'affichage.

Les calculs sont faits côté API-INTERNAL. Les montants sont validés et stockés
en centimes entiers. Aucun flottant n'est utilisé pour la persistance.

## Affichage client

Le client retrouve les documents commerciaux informatifs sur :

- `/invoices`
- `/commercial-documents/[id]`

Le client ne voit que ses propres documents, uniquement lorsqu'ils ont été
partagés. Aucun bouton de paiement, aucun PDF légal et aucune promesse de
facturation réelle ne sont affichés.

## Affichage admin

L'admin utilise :

- `/admin/catalog`
- `/admin/commercial-documents`
- `/admin/commercial-documents/[id]`

Fonctions disponibles :

- créer une offre ;
- modifier une offre ;
- créer un brouillon ;
- ajouter ou modifier des lignes tant que le document est brouillon ;
- partager le document au client ;
- annuler un document.

Fonctions explicitement absentes :

- facture officielle ;
- numérotation fiscale définitive ;
- PDF légal ;
- paiement ;
- e-mail réel ;
- signature électronique ;
- provisioning ;
- action AD.

## API et sécurité

Le navigateur ne contacte jamais `API-INTERNAL` directement.

Flux imposé :

`Navigateur -> WEBPORTAL/BFF -> API-INTERNAL -> MariaDB`

Règles de sécurité :

- `INTERNAL_API_URL` reste strictement serveur ;
- `SERVICE_AUTH_TOKEN` reste strictement serveur ;
- aucune session dans `localStorage` ou `sessionStorage` ;
- aucune journalisation de token, cookie, mot de passe ou chaîne de connexion ;
- validation stricte des identifiants, statuts et montants ;
- isolement client par `customer_id` de session ;
- mutations commerciales réservées à `internal_admin`.

## Validation

La V0.15 étend :

- les smoke tests API mock et MariaDB ;
- le contrat web `verify-commercial-foundation-contract.mjs` ;
- `npm run validate` ;
- `npm run validate:mariadb`.

## Limites volontaires

La V0.15 reste volontairement non comptable et non transactionnelle :

- pas de facture officielle ;
- pas de paiement ;
- pas de TVA automatisée validée ;
- pas d'e-invoicing ;
- pas de Chorus Pro ;
- pas de signature électronique ;
- pas d'envoi d'e-mail ;
- pas de provisioning automatique.
