# V0.30 Test envoi e-mail réel — livraison partielle (allowlist)

Statut : **partiellement livré le 2026-07-02**, uniquement la brique
garde-fou (`EMAIL_LIVE_ALLOWLIST`). Brique livrée **validée en recette
utilisateur le 2026-07-02** avec envoi SMTP OVH réel (voir section
« Recette utilisateur » plus bas). Les autres briques du cadrage —
statuts `email_messages` étendus (`smtp_auth`/`smtp_relay_refused`/
`smtp_timeout`), SPF/DKIM/DMARC documentés, recette multi-fournisseurs
Gmail / Outlook, sous-domaine `tests-mail.*` — restent à faire avant la
V1.0 RC.

## Objet du lot partiel

Rendre possible un test contrôlé de `EMAIL_INTEGRATION_MODE=live` sans
attendre le R740xd. La contrainte : **aucun envoi accidentel vers un
vrai destinataire**. La solution : une liste blanche de destinataires
autorisés, **fail-closed par défaut**.

## Ce qui est livré

### Configuration

Deux nouvelles variables d'environnement :

```bash
EMAIL_LIVE_ALLOWLIST_ONLY=true        # défaut fail-closed
EMAIL_LIVE_ALLOWLIST=support@zacharyhounsa.ovh,@home.bzh
```

- `EMAIL_LIVE_ALLOWLIST_ONLY` : quand `true` (défaut), seuls les
  destinataires listés dans `EMAIL_LIVE_ALLOWLIST` peuvent recevoir un
  e-mail en mode `live`. Toute autre adresse est refusée par
  `LiveEmailService` **avant tout appel SMTP**.
- `EMAIL_LIVE_ALLOWLIST` : liste séparée par virgules, points-virgules
  ou sauts de ligne. Chaque entrée est :
  - une **adresse complète** (`zhounsah@home.bzh`) qui matche uniquement
    cette adresse, ou
  - un **motif de domaine** commençant par `@` (`@home.bzh`) qui matche
    tout destinataire de ce domaine.
- Comparaison insensible à la casse.
- **Liste vide + `AllowlistOnly=true` = tout envoi live est bloqué**
  (défense supplémentaire).

Quand `EMAIL_LIVE_ALLOWLIST_ONLY=false` (à réserver à la V1.0 RC après
validation juridique), l'allowlist est ignorée et tout destinataire est
autorisé.

### Blocage effectif

`LiveEmailService.SendAsync` vérifie l'allowlist **avant** la
construction du `SmtpClient`. Un blocage renvoie
`EmailDeliveryResult(false, "blocked_allowlist", ...)`, journalisé :

- dans les logs applicatifs : `LogWarning` avec template, destinataire et
  correlation_id (adresse en clair, mais pas de secret) ;
- dans `email_messages.status = "blocked_allowlist"` via le pipeline
  standard `EmailDispatchService.RecordAsync` (aucune insertion SMTP,
  aucune facture BPCE ni bascule de statut) ;
- code de retour BFF : dépend du template. Pour signup/contact form,
  l'erreur est traitée comme une défaillance d'envoi (le service
  fonctionnel continue).

### Modes non concernés

`EMAIL_INTEGRATION_MODE=disabled` et `=mock` ignorent l'allowlist
(aucun envoi réel de toute façon). Le mock reste la voie testable par
défaut.

## Configuration test avec OVH MX Plan

Pour la boîte `support@zacharyhounsa.ovh` (MX Plan OVH fourni avec le
domaine) :

```bash
EMAIL_INTEGRATION_MODE=live
SMTP_HOST=ssl0.ovh.net
SMTP_PORT=587
SMTP_USE_STARTTLS=true
SMTP_USERNAME=support@zacharyhounsa.ovh    # adresse complète en username
SMTP_PASSWORD=<mot de passe boîte OVH>
SMTP_FROM_ADDRESS=support@zacharyhounsa.ovh
SMTP_FROM_DISPLAY_NAME=Kermaria
SMTP_TIMEOUT_MS=10000

# Allowlist ciblée : seul le compte support et le domaine home.bzh
EMAIL_LIVE_ALLOWLIST_ONLY=true
EMAIL_LIVE_ALLOWLIST=support@zacharyhounsa.ovh,@home.bzh

# Facultatif : formulaire de contact V0.27
CONTACT_FORM_RECIPIENT=support@zacharyhounsa.ovh
```

**Note SmtpClient** : `System.Net.Mail.SmtpClient` ne supporte pas SMTPS
implicite (port 465). Le port 587 STARTTLS est le seul chemin
compatible avec le code actuel.

## Procédure de test suggérée

1. Configurer `.env` comme ci-dessus (mot de passe SMTP en dernier).
2. `dotnet run --project apps/api-internal` — vérifier que la
   configuration est valide (log au démarrage : `Email mode live`).
3. Envoyer un e-mail contrôlé au destinataire autorisé, au choix :
   - **formulaire de contact** `/contact` (V0.27) avec
     `CONTACT_FORM_RECIPIENT=support@zacharyhounsa.ovh`. La soumission
     déclenche un `contact_form` vers la boîte support.
   - **signup** `/signup` (V0.26) avec `SIGNUP_ENABLED=true` et un
     e-mail visitor `@home.bzh` : déclenche `signup_verification`.
   - **facture BPCE** en mode `mock` ou `live` : `send-reminder` sur un
     `commercial_document` dont `billing_email` est dans l'allowlist.
4. Vérifier :
   - **Réception** dans la boîte destinataire (Inbox, pas Spam).
   - **Journal `/admin/email-log`** : status `sent` avec timestamp
     `sent_at` renseigné.
   - **Blocage** : soumettre un destinataire hors allowlist ; status
     `blocked_allowlist`, aucun `sent_at`.
5. Si l'authentification SMTP échoue, la config OVH `SMTP_USERNAME`
   attend l'adresse complète (pas `support` seul). Le mot de passe est
   celui de la boîte, pas du compte OVH.
6. **Rebasculer `EMAIL_INTEGRATION_MODE=mock`** à la fin du test si le
   run n'est pas destiné à durer.

## Écarts vs cadrage initial

- Fail-closed par défaut (`EMAIL_LIVE_ALLOWLIST_ONLY=true`) alors que le
  cadrage ne précisait pas de sens de défaut. Choix : plus safe en phase
  de tests, évite qu'un `LIVE_ALLOWLIST_ONLY=false` involontaire ouvre
  les envois.
- Blocage journalisé en `email_messages.status = "blocked_allowlist"`
  (nouveau code) plutôt que `smtp_*`. Cohérent : ce n'est pas un échec
  SMTP mais un refus applicatif.
- Le format d'entrée `@domaine` a été ajouté (pas explicité dans le
  cadrage). Utile pour n'autoriser qu'un domaine sans lister tous ses
  comptes.

## Recette utilisateur (2026-07-02)

Bascule `EMAIL_INTEGRATION_MODE=live` reelle avec SMTP OVH MX Plan
(`ssl0.ovh.net:587` STARTTLS, boite `support@zacharyhounsa.ovh`),
allowlist ciblee `support@zacharyhounsa.ovh,@home.bzh`.

### Scenarios executes

- **Formulaire `/contact`** (V0.27, template `contact_form`) : e-mail
  recu **Inbox** dans la boite support, `email_messages.status = sent`
  avec `sent_at` renseigne.
- **Signup complet** (V0.26, templates `signup_verification` +
  `account_approved`) avec destinataire dans l'allowlist : les deux
  e-mails recus reellement, chaine `email_pending` -> `email_verified`
  -> `approved` -> `password_set` -> `auth.login` bouclee sur un
  nouveau compte reel (reference `CLI-T2BKVQ`).
- **Blocage allowlist** : signup avec destinataire hors liste
  (`boutique@uneclope.com`) ->
  `email_messages.status = blocked_allowlist`, `sent_at = NULL`, aucun
  trafic SMTP sortant.
- **Anti-rejeu mot de passe** : rechargement d'un lien `/set-password`
  deja consomme -> `signup.password_failed refused TOKEN_INVALID`,
  aucun impact.

### Observations

- **Deliverabilite Inbox** confirmee sur destinataire `@home.bzh` avec
  From `support@zacharyhounsa.ovh` **sans configuration DNS SPF/DKIM/
  DMARC dediee** (la reputation du domaine racine `zacharyhounsa.ovh`
  suffit pour ce test bilateral). Le comportement peut differer sur
  Gmail/Outlook grand public — validation multi-fournisseurs a rejouer
  dans le lot V0.30 final.
- **Aucune tentative SMTP** n'est faite quand la ligne est bloquee par
  l'allowlist : le blocage est bien anterieur a la construction du
  `SmtpClient`, confirme par l'absence de trafic sortant cote reseau et
  par le log `WARN Live email blocked by allowlist ...`.
- **Correlation e-mail <-> audit** : le `correlation_id` de la ligne
  `email_messages` correspond au `correlation_id` de l'entree
  `signup.submit` / `signup.approved` correspondante. Suivi complet
  reproductible depuis l'audit log.
