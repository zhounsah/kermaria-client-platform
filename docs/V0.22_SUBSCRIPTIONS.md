# V0.22 Abonnements recurrents PayPal

Statut : **implemente dans le depot** (2026-06-25).

## Objet

V0.22 ajoute la facturation automatique mensuelle des services
recurrents (acces VPN, RDS, support continu, hebergement) en s'appuyant
sur l'API PayPal Subscriptions, distincte de l'API Orders one-shot
utilisee en V0.21.

## Architecture

```text
browser
  -> WEBPORTAL/BFF (Next.js)
     -> API-INTERNAL (.NET) -> MariaDB
     -> PayPal Subscriptions (sandbox/live)
PayPal -> WEBPORTAL/BFF (POST /api/webhooks/paypal)
       -> API-INTERNAL (POST /internal/webhooks/paypal)
         -> Subscriptions / BPCE / Email
```

Tout reste cote serveur (BFF + API-INTERNAL). Aucun token PayPal n'est
expose au navigateur. La signature webhook est validee chez PayPal avant
le forward interne.

## Modeles MariaDB

- Migration `011_subscription_offers.sql` : ajoute
  `billing_cadence ENUM('one_time','monthly')` et `paypal_plan_id VARCHAR(64)`
  sur `commercial_offers`. Le `paypal_plan_id` est cree manuellement dans le
  dashboard PayPal et colle dans le formulaire admin du catalogue.
- Migration `012_subscriptions.sql` : nouvelle table `subscriptions`
  (`customer_id` FK, `commercial_offer_id` FK, `paypal_subscription_id` UNIQUE,
  `paypal_plan_id`, `status` ENUM, `started_at`, `next_billing_at`,
  `cancelled_at`).
- Migration `013_subscription_pending_activation.sql` : etend le ENUM avec
  `pending_activation` (etat entre approbation buyer et webhook ACTIVATED).
- Migration `014_paypal_webhook_events.sql` : table d'audit des webhooks
  PayPal, `UNIQUE event_id` pour idempotence stricte, statut ENUM
  `received|processed|failed|ignored` + `raw_payload JSON`.
- Migration `015_subscription_document_link.sql` : ajoute
  `subscription_id CHAR(36) NULL` (FK -> `subscriptions`) a
  `commercial_documents` pour lier chaque facture mensuelle a sa
  souscription.
- Migration `016_paypal_plan_per_mode.sql` (V0.22.1) : remplace
  `paypal_plan_id` par deux colonnes `paypal_plan_id_sandbox` et
  `paypal_plan_id_live`. Les valeurs existantes sont migrees vers la
  colonne sandbox. Permet de stocker les deux IDs PayPal en parallele
  parce qu'un plan cree en sandbox n'existe pas en live et vice-versa.

## Cycle de vie cote serveur

1. **Admin configure une offre mensuelle** depuis `/admin/catalog` (cadence
   `monthly`). Le `paypal_plan_id` est genere automatiquement via le bouton
   "Créer le plan PayPal" sur le formulaire d'offre (V0.22.1), qui appelle
   `POST /api/admin/catalog/{id}/paypal-plan`. Le BFF :
   - cree un product PayPal via `POST /v1/catalogs/products` (type=SERVICE,
     category=SOFTWARE),
   - cree le plan associe via `POST /v1/billing/plans` (cycle mensuel,
     prix fige a `priceAmountCents`, status ACTIVE),
   - persiste l'ID dans `paypal_plan_id_sandbox` ou `paypal_plan_id_live`
     selon `PAYPAL_MODE`.
   Le prix de l'offre devient immutable des qu'au moins un plan PayPal
   existe : pour changer le prix, l'admin doit creer une nouvelle offre.
2. **Client clique "Souscrire"** sur `/services`. Le BFF
   `POST /api/subscriptions/create` :
   - valide l'offre via `GET /internal/portal/catalog`,
   - cree la souscription chez PayPal via `POST /v1/billing/subscriptions`
     (statut `APPROVAL_PENDING`),
   - persiste une ligne `pending_approval` via
     `POST /internal/portal/subscriptions`,
   - retourne `approveUrl`.
3. **Buyer approuve sur PayPal**, qui redirige vers
   `GET /api/subscriptions/return?subscription_id=...`. Le BFF passe la
   row a `pending_activation` via
   `POST /internal/portal/subscriptions/{id}/return-approved`.
4. **PayPal POST le webhook `BILLING.SUBSCRIPTION.ACTIVATED`** sur
   `POST /api/webhooks/paypal`. Le BFF lit le body brut, verifie la
   signature (`verify-webhook-signature`), forward a
   `POST /internal/webhooks/paypal`. Le `PayPalWebhookService` passe la
   row a `active` et calcule `next_billing_at = J+1 mois`.
5. **Chaque echeance**, PayPal envoie `PAYMENT.SALE.COMPLETED`. Le service
   :
   - retrouve la souscription via `billing_agreement_id`,
   - cree un `informational_invoice` lie via
     `CreateBillingDocumentFromOfferAsync` (ligne copiee de l'offre),
   - emet la facture BPCE via `IInvoiceIssuingService.IssueInvoiceAsync`,
   - confirme le paiement via `ConfirmPaymentAsync` (mark paid + email
     `payment_confirmed`).
6. **Annulation/suspension/expiration** : `BILLING.SUBSCRIPTION.{SUSPENDED,
   CANCELLED, EXPIRED}` met a jour le statut local et audite.

## UI

- `/services` (client) : badge cadence + bouton "Souscrire" sur les offres
  `monthly` avec `paypal_plan_id` defini.
- `/profile/subscriptions` (client) : liste des souscriptions du compte
  avec statut, prochaine echeance, id PayPal. Flash messages
  approved/cancelled/error sur retour PayPal.
- `/admin/subscriptions` (admin) : liste filtrable par statut + nom/ref
  client, indicateurs souscriptions actives, MRR HT estime, total.
- `/admin/subscriptions/{id}` (admin) : infos generales + historique des
  factures BPCE generees + bouton "Annuler". L'annulation cote BFF appelle
  `POST /v1/billing/subscriptions/{id}/cancel` chez PayPal puis
  `POST /internal/admin/subscriptions/{id}/cancel` (audit
  `subscription.admin_cancel`).
- Lien "Abonnements" dans `AdminNavigation` entre Paiements et E-mails.

## Garde-fous

- `PAYPAL_MODE=sandbox` par defaut. `live` reste interdit avant V0.23b
  (cf. `roadmap-current`).
- Signature webhook : vraie par defaut. `PAYPAL_WEBHOOK_VERIFY=false`
  autorise pour debug local sandbox uniquement.
- Idempotence : UNIQUE sur `paypal_webhook_events.event_id`. Un meme
  event recu deux fois ne genere qu'une seule facture.
- Aucune annulation automatique cote DB si PayPal echoue : l'admin
  tranche manuellement.
- Toutes les datetimes en UTC : `UTC_TIMESTAMP(6)` cote SQL, `DateTime.UtcNow`
  cote C#, ISO 8601 en sortie.
- Audits emis : `subscription.created`, `subscription.return_approved`,
  `subscription.activated`, `subscription.payment_received`,
  `subscription.suspended`, `subscription.cancelled`,
  `subscription.expired`, `subscription.admin_cancel`.

## Variables d'environnement (`.env.example`)

| Variable | Role | Defaut |
|---|---|---|
| `PAYPAL_CLIENT_ID` / `PAYPAL_CLIENT_SECRET` | App PayPal (deja V0.21) | requis |
| `PAYPAL_MODE` | `sandbox` ou `live` | `sandbox` |
| `PAYPAL_WEBHOOK_ID` | ID du webhook configure sur developer.paypal.com | requis pour la verif signature |
| `PAYPAL_WEBHOOK_VERIFY` | `true` defaut, `false` autorise en sandbox local | `true` |
| `PUBLIC_PORTAL_URL` | Base URL portail (deja V0.21) pour return/cancel URLs | requis |

## Limites connues

- Le webhook necessite MariaDB en backend ;
  `MockCommercialRepository.CreateBillingDocumentFromOfferAsync` throw
  `NotSupportedException` (le mock store n'est pas concu pour le flux
  systeme).
- En dev local, exposer le portail via tunnel HTTPS (ngrok, cloudflare
  tunnel) pour que PayPal puisse pousser sur
  `https://.../api/webhooks/paypal`.
- L'activation du mode `live` reste gated V0.23b (durcissement + audit).
