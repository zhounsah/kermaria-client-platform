# V0.22 Abonnements recurrents PayPal

## Objet

La V0.22 ajoute la facturation automatique mensuelle des services
recurrents (acces VPN, RDS, support continu, hebergement). Elle s'appuie
sur l'API PayPal Subscriptions, distincte de l'API Orders utilisee en
V0.21 pour les paiements one-shot.

Statut : **a faire (cadrage)**.

## Pourquoi une API differente

L'API Orders (`/v2/checkout/orders`, V0.21) est un flux one-shot :
creation d'un ordre, approbation buyer, capture, fin. Elle ne maintient
aucun engagement de prelevement futur.

L'API Subscriptions (`/v1/billing/...`) introduit deux objets persistants
cote PayPal :

- **Plan** : decrit le produit recurrent (montant, devise, frequence,
  duree). Cree une fois par service, reutilise pour tous les abonnes ;
- **Subscription** : lien entre un buyer et un plan, avec son propre id,
  son statut et son historique de prelevements. PayPal declenche les
  paiements selon le plan et notifie l'application par webhook.

## Architecture cible

```text
browser -> WEBPORTAL / BFF -> API-INTERNAL -> PayPal Subscriptions API
                                          -> BPCE (creation facture mensuelle)
                                          -> MariaDB (table subscriptions)
PayPal -> API-INTERNAL (webhook) -> BPCE (mark_as_paid)
```

Le webhook PayPal est l'evenement declencheur du cycle mensuel : a chaque
`PAYMENT.SALE.COMPLETED` recu, une facture BPCE est generee
automatiquement, son PDF est cache localement et son statut passe a
`paid`.

## Perimetre

### Catalogue : one-shot vs recurrent

Ajout d'une colonne `billing_cadence` sur `commercial_offers` :

| Valeur | Sens |
|---|---|
| `one_time` (defaut) | Prestation ponctuelle, facturee a la commande |
| `monthly` | Abonnement mensuel preleve via PayPal Subscriptions |

Les offres `monthly` requierent un identifiant `paypal_plan_id` (cree
manuellement dans le dashboard developpeur au demarrage).

### Souscription cote portail

- nouvelle page `/services/{id}/subscribe` ;
- creation d'une `Subscription` PayPal via `POST /v1/billing/subscriptions` ;
- redirection vers l'URL d'approbation buyer ;
- callback `GET /api/subscriptions/paypal/return?subscription_id=...`
  persiste l'id, le plan, le statut initial dans `subscriptions`.

### Table `subscriptions`

Migration future, schema indicatif :

```sql
CREATE TABLE subscriptions (
  id                CHAR(36) PRIMARY KEY,
  customer_id       CHAR(36) NOT NULL,
  offer_id          CHAR(36) NOT NULL,
  paypal_plan_id    VARCHAR(80) NOT NULL,
  paypal_subscription_id VARCHAR(80) NOT NULL UNIQUE,
  status            ENUM('approval_pending', 'active', 'suspended',
                         'cancelled', 'expired') NOT NULL,
  subscribed_at     DATETIME NOT NULL,
  cancelled_at      DATETIME NULL,
  next_billing_at   DATETIME NULL,
  created_at        DATETIME NOT NULL,
  updated_at        DATETIME NOT NULL,
  INDEX idx_subscriptions_customer (customer_id),
  INDEX idx_subscriptions_status (status)
);
```

### Webhooks PayPal

Endpoint `POST /internal/webhooks/paypal` :

- **authentification** : verification de la signature PayPal
  (`PayPal-Auth-Algo`, `PayPal-Cert-Url`, `PayPal-Transmission-Sig`, ...) ;
- **idempotence** : table `paypal_webhook_events (event_id PK, received_at)`
  pour ignorer les rejouages ;
- **evenements geres** :
  - `BILLING.SUBSCRIPTION.ACTIVATED` ã `subscriptions.status = active` ;
  - `PAYMENT.SALE.COMPLETED` ã creation facture BPCE + cache PDF + statut
    `paid` (s'appuie sur `InvoiceIssuingService` existant) ;
  - `BILLING.SUBSCRIPTION.SUSPENDED` ã statut `suspended` ;
  - `BILLING.SUBSCRIPTION.CANCELLED` ã statut `cancelled` + date ;
  - `BILLING.SUBSCRIPTION.EXPIRED` ã statut `expired`.

### Interface admin

Nouvelle page `/admin/subscriptions` :

- liste des abonnements actifs par client ;
- detail : historique des factures mensuelles generees, statut PayPal,
  dates ;
- actions : suspendre, reactiver, annuler (appellent PayPal cote serveur,
  jamais cote navigateur) ;
- alerte sur les abonnements `suspended` ou `expired` necessitant un
  contact client.

## Variables d'environnement a ajouter

| Variable | Role |
|---|---|
| `PAYPAL_WEBHOOK_ID` | Identifiant du webhook PayPal pour la verification de signature |

(`PAYPAL_CLIENT_ID`, `PAYPAL_CLIENT_SECRET`, `PAYPAL_MODE` sont deja
declares en V0.21.)

## Garde-fous

- aucun cycle de prelevement reel tant que `PAYPAL_MODE=sandbox` ;
- aucune souscription active en environnement de tests sur un buyer reel ;
- l'admin reste capable de marquer une periode comme `unpaid` cote local
  si la banque refuse le prelevement (statut PayPal `SUSPENDED` ou
  `EXPIRED`) ;
- la facture BPCE generee par un webhook reste immuable, comme toutes les
  factures BPCE ;
- la signature de webhook est validee avant tout traitement ;
- les `subscription_id` exposes au navigateur sont consideres comme des
  identifiants opaques, jamais comme un secret.

## Limites volontaires

La V0.22 ne fait pas :

- de prelevement SEPA hors PayPal ;
- de gestion d'avoirs ou de remboursements partiels ;
- de proration en cas de changement de plan en cours de mois ;
- de provisioning AD/NAS/RDS/VPN automatique declenche par un encaissement.

Ces sujets restent hors sequence tant que V1.0 n'est pas atteinte.
