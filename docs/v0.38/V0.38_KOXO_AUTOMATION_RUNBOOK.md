# V0.38 - Runbook d'automatisation KoXo par CSV/XML et tache planifiee

Statut : **runbook cible differe**. A executer uniquement lorsque
l'infrastructure `clients.home.bzh`, KoXo Administrator et le serveur
R740xd seront prets.

## 1. Objet

Ce runbook decrit la chaine asynchrone obligatoire de reprise KoXo :

1. Kermaria cree le compte AD au `set-password`
2. Kermaria emet un artefact `CSV/XML`
3. une tache planifiee detecte, valide et traite l'artefact
4. KoXo reprend le compte pour l'administration quotidienne
5. le statut revient dans la supervision d'exploitation

Le signup ne doit pas attendre la fin de cette chaine pour etre considere
comme termine si la creation AD a reussi.

## 2. Principes de conception

- **Kermaria source de verite**
- **AD cree avant KoXo**
- **KoXo asynchrone**
- **idempotence obligatoire**
- **journalisation bout en bout**
- **rejeu operable sans reouvrir le signup**

## 3. Pre-requis techniques

Avant activation :

- domaine enfant `clients.home.bzh` deja cree et resolvable
- racine `OU=Clients,DC=clients,DC=home,DC=bzh` confirmee
- OUs clientes `Users` / `Disabled` creees selon le besoin
- groupes `GG_VPN`, `GG_RDS`, `GG_Radio` disponibles dans
  `OU=SecurityGroups,OU=Kermaria,DC=home,DC=bzh`
- delegations appliquees sur la racine clients et sur les groupes du
  domaine parent
- KoXo Administrator installe et configure pour le domaine cible
- partage ou dossier de depot des artefacts accessible
- compte d'execution de la tache planifiee provisionne
- politique d'archivage des artefacts et des logs definie

Pre-requis encore ouverts a confirmer avant execution :

- hote exact de la tache planifiee
- chemin exact du depot `CSV/XML`
- methode de confirmation de succes KoXo
- mode de transport du statut de retour vers Kermaria

## 4. Chaine cible

### Emission Kermaria

Apres succes AD, Kermaria doit :

- calculer un `batch_id`
- produire un artefact `CSV` ou `XML`
- l'ecrire dans un dossier de depot atomique
- marquer l'utilisateur `koxo_pending`
- journaliser l'emission

Depot recommande :

- dossier `incoming`
- ecriture dans un fichier temporaire
- renommage atomique en fichier final

### Detection par tache planifiee

La tache planifiee doit :

- s'executer periodiquement
- scanner le dossier `incoming`
- verifier le format et le checksum
- deplacer le lot vers `processing`
- appeler la chaine KoXo prevue
- classer le lot en `processed` ou `failed`

### Reprise KoXo

KoXo doit reprendre les comptes deja presents dans l'AD, sur la base des
donnees structurees emises par Kermaria.

Le mecanisme cible est un traitement automatise de lots `CSV/XML`, pas une
operation GUI manuelle dans le chemin nominal.

### Retour d'etat

La chaine doit produire un etat final :

- succes
- echec recuperable
- echec non recuperable

L'operateur doit pouvoir relier cet etat a :

- `customer_reference`
- `portal_user_id`
- `sam_account_name`
- `batch_id`

## 5. Arborescence recommandee

Modele cible :

```text
<KOXO_EXCHANGE_ROOT>\
  incoming\
  processing\
  processed\
  failed\
  archive\
  logs\
```

Regles :

- aucun traitement directement dans `incoming`
- `processing` reserve aux lots en cours
- `processed` pour les succes confirmes
- `failed` pour les lots a rejouer
- `archive` pour retention longue

## 6. Tache planifiee Windows

Caracteristiques recommandees :

- execution sous compte de service dedie
- lancement a intervalle fixe
- pas de dependance a une session interactive
- sortie standard et sortie erreur redirigees vers journaux
- verrou anti-concurrence pour eviter deux executions simultanees

Comportement attendu :

- si aucun fichier n'est disponible : sortir sans erreur
- si un fichier est mal forme : le classer `failed`
- si KoXo est indisponible : conserver un etat de rejeu
- si un lot a deja ete traite : ne pas dupliquer l'action

## 7. Idempotence et rejeu

Le rejeu doit etre normal, pas exceptionnel.

Regles minimales :

- ne jamais retraiter un lot `processed` sans action volontaire
- autoriser le rejeu d'un lot `failed`
- garder les metadonnees de traitement et l'horodatage de chaque tentative
- journaliser le nombre de tentatives

Cas a couvrir :

- crash pendant `processing`
- indisponibilite KoXo
- fichier incomplet
- doublon d'emission
- reprise apres redemarrage serveur

## 8. Controle operateur

L'operateur doit pouvoir verifier rapidement :

- le compte portail est actif
- le compte AD existe et est correctement renseigne
- l'artefact KoXo a bien ete emis
- la tache planifiee a bien pris le lot
- KoXo a bien repris le compte

Tableau de controle recommande :

| Controle | Source |
|---|---|
| portail actif | Kermaria |
| user AD cree | AD |
| batch present | depot de fichiers |
| batch traite | logs tache planifiee |
| reprise KoXo | logs KoXo / etat de retour |

## 9. Gestion d'erreurs

### Echec AD avant emission KoXo

- aucun artefact KoXo emis
- utilisateur non considere comme completement actif
- ticket ou alerte exploitable

### Echec emission artefact

- compte AD deja cree
- statut `koxo_pending` ou `koxo_failed`
- reprise manuelle ou reemission possible

### Echec KoXo

- compte portail et AD restent valides
- tache planifiee classe en `failed`
- rejet operable sans recreer le compte

## 10. Scenarios d'acceptation du runbook

- particulier : succes complet portail + AD + KoXo
- pro/association : plusieurs utilisateurs sur un meme `customer`
- collision `sAMAccountName` avec suffixe numerique
- echec KoXo sans perte de l'etat AD
- rejeu idempotent d'un lot deja vu
- reprise correcte apres indisponibilite temporaire KoXo

## 11. Hors perimetre

Ce runbook ne couvre pas :

- la creation GUI manuelle KoXo comme chemin nominal
- une API REST KoXo inexistante dans le cadrage actuel
- la remise en cause de Kermaria comme source de verite
- les traitements M365/Exchange annexes
