# V0.38 - Contrats de donnees Kermaria, AD et KoXo

Statut : **contrats cibles**, a utiliser comme base d'implementation et de
runbooks. Les formats exacts de colonnes, JSON et CSV restent a traduire en
schema reel plus tard, mais les decisions de contenu sont deja figees.

Note 2026-07-18 : ces contrats s'appliquent a la cible
`clients.home.bzh`, domaine enfant maintenant cree. Le point d'entree pour
le delta entre le depot actuel et cette cible est
[`V0.38_SITE_AD_ALIGNMENT.md`](V0.38_SITE_AD_ALIGNMENT.md).

## 1. Objet

Ce document fixe :

- les donnees client et utilisateur que Kermaria doit porter ;
- les attributs AD minimums a ecrire dans `clients.home.bzh` ;
- les informations a persister pour suivre la reprise KoXo ;
- les artefacts `CSV/XML` a produire pour l'automatisation asynchrone.

## 2. Modeles metier cibles dans Kermaria

### Customer

Le `customer` doit porter au minimum :

- `customerReference`
- `customerType` : `individual`, `professional`, `association`
- `displayName`
- `billingEmail`
- `phone`
- adresse structuree :
  - `addressLine1`
  - `addressLine2`
  - `postalCode`
  - `city`
  - `country`

L'adresse est une donnee `customer`, pas `portal_user`.

### Portal user

Chaque `portal_user` issu du signup doit porter au minimum :

- `customerId`
- `email`
- `status`
- `role`
- `personalTitle`
- `givenName`
- `sn`
- `displayName`
- `initials`
- `phone`
- `isPrimaryContact`

Contraintes :

- e-mail unique au niveau portail
- plusieurs `portal_users` possibles pour un meme `customer`
- un seul `portal_user` primaire de contact par signup

### Signup pending

La future structure de signup doit memoriser :

- donnees `customer`
- liste d'utilisateurs
- statut global du signup
- statut de verification de l'e-mail principal
- jetons `set-password` individuels
- snapshot de pack si applicable

Le modele V0.26 mono-utilisateur n'est plus suffisant pour V0.38.

## 3. Contrat d'identite AD

### Domaine et UPN

- domaine cible : `clients.home.bzh`
- UPN :
  `<sAMAccountName>@clients.home.bzh`

### Regle `sAMAccountName`

Entrants :

- `givenName`
- `sn`

Transformation :

1. initiale du prenom
2. six premiers caracteres du nom
3. normalisation ASCII
4. suppression des caracteres non compatibles AD
5. verification d'unicite dans le domaine
6. suffixe numerique si collision

Exemples :

| Utilisateur | Base | Resultat |
|---|---|---|
| Alexandre GUYOT | `aguyot` | `aguyot` |
| Amelie GUYOT | `aguyot` | `aguyot1` |

### Attributs AD minimums

| Attribut AD | Source Kermaria | Regle |
|---|---|---|
| `personalTitle` | `portal_user.personalTitle` | copie simple |
| `givenName` | `portal_user.givenName` | copie simple |
| `sn` | `portal_user.sn` | copie simple |
| `displayName` | `portal_user.displayName` | copie simple |
| `initials` | `portal_user.initials` | copie simple ou derivee |
| `cn` | `portal_user.displayName` | derive de l'identite affichee |
| `name` | `portal_user.displayName` | aligne sur `cn` |
| `sAMAccountName` | derive | voir regle plus haut |
| `userPrincipalName` | derive | suffixe `@clients.home.bzh` |
| `mail` | `portal_user.email` | unique par utilisateur |
| `telephoneNumber` | `portal_user.phone` | nullable |
| `employeeNumber` | `customer.customerReference` | obligatoire |

Attributs volontairement hors perimetre initial :

- attributs de licence M365
- attributs de facturation complexes
- home/profile path
- groupes metier KoXo specifiques

### Emplacement des comptes et groupes

- les comptes utilisateurs Kermaria vivent dans
  `OU=Users,OU=<CUSTOMER_REFERENCE>,OU=Clients,DC=clients,DC=home,DC=bzh`
- les comptes desactives vivent dans
  `OU=Disabled,OU=<CUSTOMER_REFERENCE>,OU=Clients,DC=clients,DC=home,DC=bzh`
- aucun groupe de securite n'est cree dans le domaine enfant
- les groupes de securite Kermaria restent dans
  `OU=SecurityGroups,OU=Kermaria,DC=home,DC=bzh`
- les appartenances `GG_VPN`, `GG_RDS`, `GG_Radio` sont gerees comme des
  rattachements a ces groupes du domaine parent

## 4. Persistance du lien AD

Kermaria doit garder une table ou extension de table permettant de suivre :

- `customerId`
- `portalUserId`
- `adDomain`
- `adDistinguishedName`
- `adObjectGuid`
- `adObjectSid`
- `sAMAccountName`
- `userPrincipalName`
- `adProvisioningStatus`
- `adProvisionedAt`
- `lastPasswordSyncAt`
- `lastPasswordSyncStatus`

Le modele historique `customer_ad_links` centre sur le `customer` doit etre
etendu ou complete pour suivre plusieurs utilisateurs sous un meme client.

## 5. Suivi KoXo dans Kermaria

Kermaria doit persister separement l'etat de reprise KoXo.

Etat minimum cible :

- `koxoExportStatus`
- `koxoExportAttemptCount`
- `koxoExportLastAttemptAt`
- `koxoExportLastSuccessAt`
- `koxoExportLastErrorCode`
- `koxoExportLastErrorMessage`
- `koxoArtifactType`
- `koxoArtifactPath`
- `koxoArtifactChecksum`
- `koxoCorrelationId`

Le suivi KoXo ne doit pas ecraser le suivi AD.

## 6. Artefact `CSV` cible

Format cible recommande pour la tache planifiee :

- un fichier par lot
- UTF-8
- delimiteur fixe et documente
- idempotence portee par un identifiant de lot et un identifiant de ligne

Colonnes minimales recommandees :

| Colonne | Usage |
|---|---|
| `batch_id` | identifiant unique du lot |
| `event_type` | `create_or_update_user` |
| `customer_reference` | rapprochement metier |
| `portal_user_id` | rapprochement Kermaria |
| `email` | identite portail |
| `personal_title` | identite |
| `given_name` | identite |
| `sn` | identite |
| `display_name` | identite |
| `initials` | identite |
| `phone` | identite |
| `employee_number` | reference client |
| `sam_account_name` | identite AD |
| `user_principal_name` | identite AD |
| `ad_distinguished_name` | emplacement AD |
| `customer_type` | routage KoXo eventuel |
| `is_primary_contact` | indicateur |
| `exported_at_utc` | audit |

## 7. Artefact `XML` cible

Le XML doit etre prevu pour les environnements ou KoXo consomme plus
facilement des fichiers de reponses ou des imports structures.

Structure logique cible :

- `batch`
  - `batchId`
  - `generatedAtUtc`
  - `sourceSystem=Kermaria`
  - `customer`
    - `customerReference`
    - `customerType`
    - `displayName`
  - `users`
    - `user`
      - `portalUserId`
      - `email`
      - `personalTitle`
      - `givenName`
      - `sn`
      - `displayName`
      - `initials`
      - `phone`
      - `employeeNumber`
      - `sAMAccountName`
      - `userPrincipalName`
      - `distinguishedName`

Le XML doit contenir assez d'information pour etre retraitable sans
recalcul de l'identite AD par KoXo.

## 8. Contrat d'idempotence

Le lot KoXo doit etre retraitable sans effet de bord non controle.

Regles minimales :

- un `batch_id` unique par emission
- une cle metier stable par utilisateur :
  `customer_reference + portal_user_id + sam_account_name`
- reemission autorisee du meme utilisateur si :
  - l'etat AD a change
  - une erreur KoXo precedente doit etre rejouee
- la reprise KoXo doit ignorer les doublons deja confirmes

## 9. Contrat de journalisation

Chaque traitement doit produire :

- un journal Kermaria cote emission
- un journal tache planifiee cote transport
- un journal KoXo cote reprise

Correlation minimale :

- `correlation_id` Kermaria
- `batch_id`
- `portal_user_id`
- `customer_reference`
- `sam_account_name`

## 10. Contrat d'etats

Etats documentes a conserver :

| Etat | Signification |
|---|---|
| `ad_pending` | compte portail pret, AD pas encore cree |
| `ad_succeeded` | compte AD cree et lie |
| `ad_failed` | creation AD echouee |
| `koxo_pending` | export KoXo a produire ou a rejouer |
| `koxo_exported` | artefact emis |
| `koxo_succeeded` | reprise KoXo confirmee |
| `koxo_failed` | reprise KoXo echouee |

La documentation et l'implementation future doivent toujours distinguer
`ad_succeeded` et `koxo_succeeded`.
