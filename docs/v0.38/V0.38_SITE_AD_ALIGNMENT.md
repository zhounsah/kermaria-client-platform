# V0.38 - Alignement des donnees site et Active Directory

Statut : **document d'alignement implementation-ready**, redige apres la
creation du domaine enfant `clients.home.bzh` le 2026-07-18. Ce document
part du **code actuel du depot** et fixe ce qu'il faut aligner dans les
donnees Kermaria avant de brancher proprement le signup et les comptes
portail sur l'AD cible.

## 1. Objet

L'objectif est de faire converger :

- les donnees saisies et affichees par le site ;
- les donnees persistees dans MariaDB ;
- les attributs minimaux attendus dans `clients.home.bzh` ;
- le suivi applicatif du lien portail <-> AD.

Ce document ne change pas le comportement actuel V0.26 : il prepare la
translation de ce comportement vers un modele de donnees compatible AD.

## 2. Verite actuelle dans le depot

### Customers

Le schema courant `customers` porte seulement :

- `external_reference`
- `display_name`
- `billing_email`
- `phone`
- `address`
- `city`
- `country`

Cela suffit pour V0.26/V0.25, mais pas pour une identite client structuree
avec adresse postale exploitable et type de client distinct.

### Portal users

Le schema courant `portal_users` porte surtout :

- `email`
- `display_name`
- `password_hash`
- `status`
- `role`

Il manque aujourd'hui les champs d'identite attendus pour l'AD :

- `personal_title`
- `given_name`
- `surname`
- `initials`
- `phone`
- `is_primary_contact`

### Signup pending

Le signup V0.26 reste mono-utilisateur. `signup_pending` memorise :

- `company_name`
- `contact_name`
- `email`
- `phone`
- `message`

Il ne memorise pas encore :

- un type de client ;
- une adresse structuree ;
- plusieurs utilisateurs ;
- des statuts par utilisateur ;
- des jetons `set-password` par utilisateur.

### Liens AD

Le schema `customer_ad_links` est encore centre sur le `customer` et sur
un objet AD, sans `portal_user_id` direct. Il ne suffit pas pour suivre un
compte AD par utilisateur portail dans un scenario multi-utilisateur.

### Ce que le code AD sait deja faire

Le depot a deja deux fondations importantes :

- `AdConfigurationResolver` accepte deja `AD_DOMAIN`,
  `AD_CLIENTS_OU_DN`, `AD_REQUIRED_OU_ROOT` et `AD_ALLOWED_ROOTS` ;
- `LdapActiveDirectoryService.CreateUserAsync(...)` sait deja pousser
  `sAMAccountName`, `userPrincipalName`, `givenName`, `sn`,
  `displayName` et `description`.

Le blocage principal n'est donc plus le domaine enfant lui-meme. Le blocage
est surtout le **modele de donnees du site**.

## 3. Decisions figees

- domaine AD cible : `clients.home.bzh`
- racine AD cible : `OU=Clients,DC=clients,DC=home,DC=bzh`
- structure cible : `OU=<CUSTOMER_REFERENCE>/Users|Disabled`
- groupes de securite centralises dans
  `OU=SecurityGroups,OU=Kermaria,DC=home,DC=bzh`
- Kermaria reste la source de verite metier
- creation AD au `set-password`, pas a l'approbation admin
- `employeeNumber = customer.customerReference`
- `userPrincipalName = <sAMAccountName>@clients.home.bzh`
- un e-mail portail unique par `portal_user`
- pour les pros/associations : un compte portail et un compte AD par
  utilisateur

## 4. Donnees site a aligner

### Cote customer

Le site doit porter de facon explicite :

- `customerType` : `individual`, `professional`, `association`
- `displayName`
- `billingEmail`
- `phone`
- `addressLine1`
- `addressLine2`
- `postalCode`
- `city`
- `country`

### Cote portal user

Le site doit porter de facon explicite :

- `personalTitle`
- `givenName`
- `sn`
- `displayName`
- `initials`
- `email`
- `phone`
- `isPrimaryContact`

### Cote identite AD

Le site ne doit pas recalculer l'identite AD a l'aveugle dans plusieurs
endroits. Une fois calculee, il faut pouvoir tracer :

- `sAMAccountName`
- `userPrincipalName`
- `adDomain`
- `adDistinguishedName`
- `adObjectGuid`
- `adObjectSid`
- `adProvisioningStatus`
- `adProvisionedAt`
- `lastPasswordSyncAt`
- `lastPasswordSyncStatus`

## 5. Evolutions de schema a preparer

### `customers`

Evolution cible :

- ajouter `customer_type`
- remplacer `address` libre par `address_line_1`, `address_line_2`,
  `postal_code`, `city`, `country`
- conserver `display_name`, `billing_email`, `phone`

### `portal_users`

Evolution cible :

- ajouter `personal_title`
- ajouter `given_name`
- ajouter `surname`
- ajouter `initials`
- ajouter `phone`
- ajouter `is_primary_contact`

### `signup_pending`

Evolution cible :

- ne plus supposer `1 signup = 1 seul utilisateur`
- separer les donnees `customer` des donnees `users`
- memoriser des jetons et statuts d'activation **par utilisateur**

Implementation recommandee :

- soit `signup_pending` + `signup_pending_users`
- soit `signup_pending` + `signup_pending_user_tokens`

Le JSON seul n'est pas recommande comme unique support, car il rendrait les
controles admin, les reprises et les migrations plus fragiles.

### `customer_ad_links`

Evolution cible :

- ajouter `portal_user_id`
- ajouter `ad_domain`
- ajouter `ad_provisioning_status`
- ajouter `ad_provisioned_at`
- ajouter `last_password_sync_at`
- ajouter `last_password_sync_status`

Si la table devient trop heterogene, un renommage plus explicite de type
`portal_user_ad_links` sera preferable a long terme.

## 6. Impact fonctionnel

### Signup public

Le futur formulaire `/signup` devra cesser d'envoyer seulement :

- `companyName`
- `contactName`
- `email`
- `phone`
- `message`

Il devra envoyer des blocs structures `customer` et `users`.

### Approbation admin

L'approbation admin doit continuer a creer les objets portail uniquement :

- `customer`
- `portal_user`
- jetons `set-password`

La creation AD reste reportee au `set-password`.

### Set-password

Le `set-password` devra devenir le moment ou :

1. le mot de passe portail est valide ;
2. l'identite AD cible est calculee ;
3. le compte AD est cree ou complete dans `clients.home.bzh` ;
4. le mot de passe AD est aligne ;
5. le lien AD applicatif est persiste.

## 8. Ordre recommande d'implementation

1. etendre le modele SQL `customers`, `portal_users` et les liens AD
2. etendre les contrats API internes et BFF
3. faire evoluer l'admin signup pour afficher les donnees structurees
4. faire evoluer le formulaire public `/signup`
5. brancher la creation AD au `set-password`
6. brancher ensuite la reprise KoXo asynchrone

## 9. Limites explicites

- la creation du domaine enfant `clients.home.bzh` ne suffit pas, a elle
  seule, a aligner les donnees Kermaria ;
- le code courant V0.26 reste mono-utilisateur et sans creation AD
  automatique ;
- l'alignement de code vers la nouvelle arborescence
  `OU=Clients/<CUSTOMER_REFERENCE>/Users|Disabled` reste un prerequis
  distinct de l'alignement de donnees.
