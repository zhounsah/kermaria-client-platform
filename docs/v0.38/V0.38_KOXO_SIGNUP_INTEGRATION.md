# V0.38 - Integration signup Kermaria, Active Directory et KoXo

Statut : **spec future feature**, a implementer plus tard. Ce document
decrit la cible voulue ; il ne decrit pas le code actuellement livre.

Note 2026-07-18 : le domaine enfant `clients.home.bzh` existe maintenant.
Le point dur n'est plus l'existence du domaine, mais l'alignement du
modele de donnees Kermaria et du workflow signup. Voir aussi
[`V0.38_SITE_AD_ALIGNMENT.md`](V0.38_SITE_AD_ALIGNMENT.md).

## 1. Objet

V0.38 prepare un flux dans lequel Kermaria :

- collecte des donnees client et utilisateur structurees au signup ;
- cree les comptes portail ;
- cree les comptes AD dans `clients.home.bzh` ;
- synchronise les mots de passe portail -> AD ;
- alimente une reprise KoXo asynchrone et obligatoire ;
- reste la source de verite metier de bout en bout.

La creation AD n'est **pas** declenchee a l'approbation admin. Elle est
declenchee au moment du `set-password`, afin d'aligner le mot de passe
initial du portail et de l'AD et d'eviter la creation de comptes AD jamais
actives.

## 2. Etat actuel a depasser

L'etat courant V0.26 est documente dans
[../V0.26_USER_GUIDE_SIGNUP.md](../V0.26_USER_GUIDE_SIGNUP.md) :

- le signup collecte `societe`, `contact`, `e-mail`, `telephone`,
  `besoin` ;
- l'approbation admin cree `customer + portal_user` ;
- aucun compte AD n'est cree automatiquement ;
- l'AD est gere ensuite par des actes admin separes.

V0.38 remplace cette hypothese par une cible multi-utilisateur, structuree
et integree a l'AD et a KoXo.

## 3. Perimetre cible

### Particuliers

Un signup particulier produit :

- un `customer` ;
- un `portal_user` principal ;
- un `user AD` principal ;
- un artefact de reprise KoXo pour ce compte.

### Pros et associations

Un signup pro/association devient **multi-utilisateur** des l'origine :

- un `customer` unique ;
- plusieurs `portal_users` rattaches a ce `customer` ;
- un `user AD` par `portal_user` ;
- un e-mail unique obligatoire par utilisateur ;
- un `set-password` et une activation independants par utilisateur ;
- une reprise KoXo obligatoire pour chaque compte cree.

Chaque utilisateur supplementaire a un compte portail et un compte AD. Il
n'y a pas, dans V0.38, de mode "AD only" pour les utilisateurs secondaires.

## 4. Donnees fonctionnelles a collecter

Le futur `/signup` ne repose plus sur `contactName` libre. Il devient
structure et separe les donnees **client** des donnees **utilisateur**.

### Donnees client

- type de client : `individual`, `professional`, `association`
- raison sociale ou nom du client
- adresse structuree
- telephone principal du client
- e-mail de contact principal
- besoin ou message libre facultatif
- eventuel snapshot de pack deja existant dans le modele courant

L'adresse est portee au niveau `customer`, pas repetee sur chaque
utilisateur.

### Donnees utilisateur

Pour chaque utilisateur :

- `personalTitle`
- `givenName`
- `sn`
- `displayName`
- `initials`
- `email`
- `phone`
- futurs identifiants techniques derives :
  `employeeNumber`, `sAMAccountName`, `userPrincipalName`

`employeeNumber` doit reprendre la reference client Kermaria.

## 5. Flux cible de bout en bout

### Etape 1 - Soumission `/signup`

Le formulaire public :

- collecte les donnees client ;
- collecte une liste d'utilisateurs ;
- valide les regles d'unicite e-mail ;
- reste protege par les garde-fous anti-bot et anti-abus deja en place ;
- cree une demande `signup_pending` enrichie, compatible multi-utilisateur.

### Etape 2 - Verification e-mail

Le mecanisme de verification e-mail reste present, mais il doit etre
etendu pour gerer un signup avec plusieurs utilisateurs.

Decision cible :

- le contact principal reste l'ancre initiale du workflow ;
- les utilisateurs supplementaires recoivent ensuite leur propre lien
  `set-password` apres approbation admin ;
- la validation de l'adresse principale n'active pas encore les comptes AD.

### Etape 3 - Approbation admin

L'approbation admin cree :

- le `customer` ;
- l'ensemble des `portal_users` rattaches au `customer` ;
- les jetons de `set-password` individuels ;
- l'audit complet de l'operation.

L'approbation n'ecrit pas encore dans `clients.home.bzh`.

### Etape 4 - Set-password individuel

Chaque utilisateur finalise son activation via son propre lien
`/set-password`.

Au moment du `set-password`, Kermaria doit :

1. valider le jeton et le mot de passe portail ;
2. calculer l'identite AD cible ;
3. creer ou completer le `user AD` dans `clients.home.bzh` ;
4. definir le mot de passe AD avec la meme valeur que le portail ;
5. activer le compte AD ;
6. enregistrer le lien AD dans la base Kermaria ;
7. marquer l'utilisateur portail comme actif ;
8. alimenter la file/artefact de reprise KoXo ;
9. journaliser les statuts de creation AD et de reprise KoXo.

Si la creation AD echoue, le portail ne doit pas presenter l'utilisateur
comme completement actif.

## 6. Domaine et structure AD cibles

### Domaine

Le domaine cible est **`clients.home.bzh`**.

V0.38 ne reutilise pas le schema V0.25/V0.31 qui etait centre sur
`home.bzh` et `OU=TEST_SITE_WEB`.

### Structure logique d'OU

La structure cible retenue pour les comptes clients est :

```text
OU=Clients,DC=clients,DC=home,DC=bzh
  +-- OU=<CUSTOMER_REFERENCE>
      +-- OU=Users
      +-- OU=Disabled
```

Les groupes de securite restent, eux, dans le domaine parent :

```text
OU=SecurityGroups,OU=Kermaria,DC=home,DC=bzh
  +-- CN=GG_VPN
  +-- CN=GG_RDS
  +-- CN=GG_Radio
```

Contraintes a respecter :

- OU racine dediee aux clients web Kermaria :
  `OU=Clients,DC=clients,DC=home,DC=bzh`
- segregation claire par `customer`
- OUs `Users` et `Disabled` previsibles
- aucune creation de groupes de securite dans le domaine enfant
- gestion des appartenances `GG_*` dans `home.bzh`
- aucun melange avec les populations non portail du domaine

## 7. Regles d'identite AD

### sAMAccountName

Regle figee :

- base = `initiale du prenom + 6 premiers caracteres du nom`
- normalisation ASCII
- suppression des espaces, apostrophes, tirets et caracteres non admis
- casse normalisee

Exemples :

- Alexandre GUYOT -> `aguyot`
- Amelie GUYOT -> `aguyot1`

Politique de collision :

- premier compte : suffixe absent
- collision suivante : suffixe numerique `1`, `2`, `3`, etc.
- la recherche d'un identifiant libre se fait dans `clients.home.bzh`

### userPrincipalName

Le `userPrincipalName` doit etre :

```text
<sAMAccountName>@clients.home.bzh
```

### Attributs AD minimums

Le compte AD doit au minimum porter :

- `personalTitle`
- `givenName`
- `sn`
- `displayName`
- `initials`
- `cn`
- `name`
- `sAMAccountName`
- `userPrincipalName`
- `employeeNumber`
- `mail`
- `telephoneNumber`

`employeeNumber` doit contenir la reference client Kermaria.

## 8. Synchronisation continue du mot de passe

Decision V0.38 : la synchronisation portail -> AD est continue.

Cela implique :

- alignement initial du mot de passe au `set-password`
- propagation des futurs changements de mot de passe portail vers l'AD
- aucune synchronisation inverse AD -> portail
- journalisation et statut d'echec distincts pour le changement de mot de
  passe AD

Kermaria reste la source de verite. L'AD n'impose pas son propre mot de
passe au portail.

## 9. Place de KoXo

KoXo Administrator :

- n'est pas la source de verite ;
- n'est pas dans le chemin bloquant du signup ;
- reprend les comptes deja crees dans AD ;
- sert a l'administration quotidienne, a la supervision et aux operations
  de maintenance courantes.

La reprise KoXo est :

- **obligatoire**
- **asynchrone**
- **non bloquante** pour la fin du `set-password`

En cas d'echec KoXo, le compte portail et le compte AD peuvent etre actifs
si la creation AD a reussi, mais l'etat KoXo doit rester visible et
reessayable.

## 10. Etats et erreurs a rendre visibles

Le systeme doit distinguer au minimum :

- `portal_user_created`
- `portal_password_set`
- `ad_user_created`
- `ad_password_synced`
- `ad_link_recorded`
- `koxo_export_pending`
- `koxo_export_sent`
- `koxo_replay_succeeded`
- `koxo_replay_failed`

Il faut explicitement separer :

- **compte AD cree**
- **compte repris par KoXo**

Un echec KoXo ne doit jamais etre confondu avec un echec AD.
