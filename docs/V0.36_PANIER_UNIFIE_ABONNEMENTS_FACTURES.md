# V0.36 - Panier unifié et abonnements facturés

Statut : **implémenté dans le dépôt (2026-07-09)**.

Cette V0.36 complète la V0.35. La V0.35 introduisait un panier client
strictement one-shot. La V0.36 ajoute un **checkout unifié** pour
prévisualiser ensemble les achats ponctuels et les sélections récurrentes,
mais conserve volontairement **deux confirmations métier distinctes** :

- les achats ponctuels restent confirmés comme une commande one-shot ;
- les abonnements récurrents sont confirmés comme des souscriptions locales
  d'abord en attente de paiement, puis activées au règlement.

Le dépôt contient donc désormais un tunnel client cohérent de bout en bout :

- sélection depuis `/souscrire` ;
- mini-panier dans le header ;
- récapitulatif `/panier` ;
- génération du document commercial ;
- choix explicite du règlement Stripe / PayPal / virement bancaire ;
- déclenchement automatique de l'activation ou du provisioning une fois le
  document réellement payé.

## Ce que livre V0.36

- un **drawer panier** en haut à droite de l'espace client authentifié,
  accessible soit au survol continu, soit au clic pour l'épingler ;
- une page `/souscrire` qui alimente un **même récapitulatif visuel** pour :
  - les achats ponctuels `one_time` ;
  - les packs/offres récurrentes `monthly` ;
- une page `/panier` qui montre les deux familles sur un seul écran, avec
  deux cartes de confirmation distinctes ;
- un tunnel de **souscriptions facturées** :
  - création locale de souscriptions `pending_payment`,
  - création d'une facture initiale groupée,
  - choix du rail de paiement sur la page document,
  - activation automatique après paiement ;
- le choix **explicite du virement bancaire** côté client sur la page document,
  en plus de Stripe et PayPal ;
- la réutilisation du flux admin existant **Marquer comme payé** pour déclencher
  automatiquement les effets métier d'un virement reçu ;
- un worker de renouvellement qui émet les prochaines factures des
  souscriptions `billing` à échéance, puis suspend en cas d'impayé prolongé ;
- une remontée d'erreur portail plus utile :
  - les erreurs 400 / 404 / 409 propagent le message serveur ;
  - les erreurs connues de résiliation/paiement (`STRIPE_NOT_CONFIGURED`,
    `PAYPAL_NOT_CONFIGURED`, `PERSIST_ERROR`, `SQL_UNAVAILABLE`,
    `INTERNAL_ERROR`) sont montrées avec leur référence de corrélation.

## Parcours utilisateur

### 1. Entrée `/souscrire`

La page [`apps/webportal/app/souscrire/page.tsx`](../apps/webportal/app/souscrire/page.tsx)
devient le point d'entrée unique de sélection :

- les **packs récurrents** passent par `PublicPackCard` puis
  `AddRecurringCheckoutButton` ;
- les **achats ponctuels** passent par `AddToCartButton` ;
- les **prestations sur devis** conservent le tunnel séparé
  `/request-service`.

Le haut de page affiche un bandeau de synthèse si le client a déjà des
éléments en cours, avec un lien direct vers `/panier`.

### 2. Drawer panier dans le header

Le composant [`apps/webportal/components/HeaderCartDrawer.tsx`](../apps/webportal/components/HeaderCartDrawer.tsx)
charge `/api/checkout/summary` et prévisualise en direct :

- le sous-total des achats ponctuels ;
- le sous-total des abonnements récurrents ;
- un total immédiat estimatif ;
- quelques lignes récentes de chaque tunnel.

Le drawer s'ouvre :

- au survol ;
- au focus clavier ;
- au clic, ce qui l'épingle jusqu'à clic extérieur ou `Escape`.

Une **zone de pont invisible** (`header-cart-hover-bridge`) supprime la
rupture entre le bouton et le drawer pour permettre un déplacement de souris
 continu sans fermeture intempestive.

### 3. Récapitulatif `/panier`

La page [`apps/webportal/app/panier/page.tsx`](../apps/webportal/app/panier/page.tsx)
reste volontairement unique visuellement, mais sépare les confirmations :

- **Achats ponctuels**
  - liste tabulaire ;
  - suppression par ligne ;
  - confirmation via `CartConfirmButton` ;
- **Abonnements récurrents**
  - cartes détaillées ;
  - engagement, cadence, mise en service, première échéance ;
  - suppression par ligne ;
  - confirmation via `RecurringCheckoutConfirmButton`.

Le client voit donc un seul panier, mais ne mélange jamais les règles métier
de validation.

## Tunnel achat ponctuel

Le tunnel V0.35 reste en place :

1. les lignes `cart_items` sont confirmées par `ICartService` ;
2. `CreateCartDocumentAsync(...)` matérialise un document multi-lignes
   `origin='client_cart'` ;
3. `IssueInvoiceAsync(...)` essaie de l'émettre immédiatement ;
4. le client est redirigé vers `/commercial-documents/{id}` ;
5. il choisit ensuite Stripe, PayPal ou virement bancaire.

Le provisioning lié au panier reste **best-effort** et borné au mécanisme
existant `CartProvisioningTrigger`. Sur le catalogue courant, ce déclencheur
est principalement inertiel tant qu'aucune offre one-shot n'est mappée vers
une ressource provisionnable.

## Tunnel abonnements facturés

La V0.36 introduit un second tunnel, distinct des abonnements Stripe/PayPal
créés directement chez le PSP.

### Confirmation

`RecurringCheckoutService.ConfirmAsync(...)` :

1. recharge les sélections persistées du client ;
2. crée **une souscription locale par ligne** via
   `SubscriptionService.CreateBilledPendingAsync(...)` ;
3. crée ensuite un document commercial groupé via
   `CreateRecurringCheckoutDocumentAsync(...)` ;
4. tente de l'émettre immédiatement ;
5. vide la sélection récurrente du client.

Les souscriptions sont créées avec :

- `rail = 'billing'` ;
- `status = 'pending_payment'` ;
- `paid_cycles_count = 0` ;
- `started_at = null`, `next_billing_at = null`,
  `commitment_ends_at = null` tant qu'aucun paiement n'est enregistré.

### Document initial

Le document initial :

- porte `origin='recurring_checkout'` ;
- contient une ligne de récurrence par souscription ;
- ajoute une ligne de mise en service si `setup_fee_amount_cents > 0` ;
- est relié aux souscriptions via
  `commercial_document_line_subscriptions`.

Cette table de liaison est essentielle : elle permet au paiement d'un document
de retrouver **toutes** les souscriptions concernées, y compris lorsque le
document regroupe plusieurs lignes récurrentes.

## Paiement et choix explicite du virement bancaire

### Choix côté client

La page document client
[`apps/webportal/app/commercial-documents/[id]/page.tsx`](../apps/webportal/app/commercial-documents/[id]/page.tsx)
affiche désormais un choix explicite de rail via
[`apps/webportal/components/PayButton.tsx`](../apps/webportal/components/PayButton.tsx) :

- Stripe ;
- PayPal ;
- Virement bancaire.

Le rail `manual` n'est pas un paiement en ligne. Il enregistre simplement le
choix du client via :

- `POST /api/commercial-documents/{id}/payment-method`
- puis `POST /internal/portal/commercial-documents/{id}/payment-method`

avec les garde-fous suivants :

- seul `manual` est accepté ;
- le document doit être déjà `issued` ;
- le document reste **non payé** tant qu'aucune confirmation de règlement
  n'est reçue.

Une fois le choix enregistré, la page affiche les coordonnées bancaires
existantes (IBAN / BIC / bénéficiaire / référence à indiquer).

### Effet métier attendu

Choisir le virement bancaire signifie :

- la commande est validée côté document ;
- le document reste à régler ;
- les souscriptions facturées restent `pending_payment` ;
- aucun provisioning n'est déclenché avant encaissement.

### Encaissement

Le passage réel à `paid` peut venir :

- d'un retour Stripe ;
- d'un retour PayPal ;
- de l'action admin **Marquer comme payé** pour un virement reçu.

Les trois chemins convergent vers
`InvoiceIssuingService.ConfirmPaymentAsync(...)`, qui :

1. marque la facture BPCE comme payée ;
2. marque le document commercial local comme `paid` ;
3. déclenche `CartProvisioningTrigger` ;
4. déclenche `BilledSubscriptionPaymentTrigger` ;
5. envoie l'e-mail `payment_confirmed`.

## Activation automatique des souscriptions facturées

[`apps/api-internal/Services/Provisioning/BilledSubscriptionPaymentTrigger.cs`](../apps/api-internal/Services/Provisioning/BilledSubscriptionPaymentTrigger.cs)
reçoit l'événement "document payé" et recharge les souscriptions liées.

Cas principal, premier paiement :

- `pending_payment` -> `pending_activation`
- puis `ActivateAsync(...)`
- puis `RecordPaymentAsync(...)`

Effets calculés :

- `started_at` = date de paiement du premier terme ;
- `next_billing_at` = `started_at + billing_interval_months` ;
- `commitment_ends_at` = `started_at + commitment_months` ;
- `paid_cycles_count` incrémenté ;
- provisioning réconcilié via `SubscriptionProvisioningManager`.

Cas suivant :

- si la souscription est déjà `active`, `RecordPaymentAsync(...)` avance
  simplement la prochaine échéance ;
- si elle est `suspended` et qu'un paiement revient, le trigger peut la
  réactiver et relancer le provisioning.

## Renouvellement automatique

[`apps/api-internal/Services/BillingSubscriptionRenewalWorker.cs`](../apps/api-internal/Services/BillingSubscriptionRenewalWorker.cs)
tourne en tâche de fond toutes les 10 minutes.

Il cible les souscriptions :

- `rail='billing'` ;
- non `cancelled` / non `expired` ;
- dont `next_billing_at <= now`.

Comportement :

- s'il existe déjà un document non payé pour la souscription, aucun nouveau
  document n'est créé ;
- si ce document reste ouvert plus de 24 heures, la souscription passe
  `suspended` ;
- sinon, un nouveau document de renouvellement est créé via
  `CreateBillingDocumentForSubscriptionAsync(...)` puis émis ;
- si la souscription est `pending_cancellation` avec `cancel_at_term_end`,
  elle bascule `cancelled` à la fin du terme au lieu de régénérer une facture.

### Migration 029

### Migration 030

`030_recurring_offer_cadence_backfill.sql` corrige les offres historiques
mensuelles qui avaient conservé par héritage la cadence `one_time`, afin que
la page `/souscrire` et le checkout récurrent affichent le bon tunnel.

### Correctif MariaDB panier

`MariaDbCartRepository` ne caste plus directement `offer_id` comme une chaîne
par un `GetString(...)` fragile. La lecture passe par
`MariaDbIdentifierReader.ReadRequired(...)`, ce qui évite le
`InvalidCastException` observé avec certains retours MariaDB.

## Résumé métier

V0.36 ne fusionne pas les règles métier, elle fusionne la **lecture du panier**.
Le client compose librement ses achats sur un écran unifié, puis chaque famille
continue sur le tunnel qui lui correspond :

- achat ponctuel -> document one-shot payable ;
- abonnement facturé -> souscription locale `pending_payment` + facture
  initiale groupée + activation automatique après règlement.
