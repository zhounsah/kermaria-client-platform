# V0.19 Active Directory Security Hardening

## Objet

La V0.19 prolonge la V0.18 et durcit la securite et la coherence des flux
admin autour de l'integration Active Directory controlee. Elle ne change pas
l'architecture et reste strictement bornee a l'OU de test :

`OU=TEST_SITE_WEB,DC=home,DC=bzh`

Le flux autorise reste :

```text
browser -> WEBPORTAL / BFF -> API-INTERNAL -> Active Directory
```

`WEBPORTAL` ne contacte jamais Active Directory directement.

## Renforts

- mutations BFF admin sensibles protegees par un jeton CSRF cote serveur
  (`apps/webportal/lib/csrf-server.ts`, `apps/webportal/lib/csrf.ts`), sans
  stockage du jeton en `localStorage` ou `sessionStorage` ;
- `X-Service-Auth` exige sur `/internal/*` dans tout environnement non
  `Development` et plus seulement en Production ;
- `RUN_MARIADB_TESTS=true` explicitement refuse hors `Development` par
  `RuntimeConfigurationValidator` ;
- validateur d'entrees Active Directory strict cote `API-INTERNAL`
  (`ActiveDirectoryInputValidator`) : DN normalises, scope client verifie et
  operations cross-customer refusees avant toute action ;
- routes admin BFF reorganisees autour de mutations bornees auditables
  plutot qu'une simple lecture seule ;
- suite de tests `API-INTERNAL` etendue sur les flux AD controles
  (`tests/api-internal`).

## Compatibilite avec la V0.18

La V0.19 ne retire aucun garde-fou de la V0.18 :

- les modes `disabled`, `mock`, `read_only` et `controlled_write` restent les
  seuls modes autorises ;
- les ecritures autorisees, les interdictions et les garde-fous decrits dans
  `docs/V0.18_ACTIVE_DIRECTORY_CONTROLLED_WRITE.md` restent integralement
  applicables ;
- aucune OU de production n'est activee.

## Interdictions reaffirmees

- aucune mutation admin sans jeton CSRF cote serveur ;
- aucun appel `/internal/*` accepte sans `X-Service-Auth` hors
  `Development` ;
- aucun `RUN_MARIADB_TESTS=true` accepte hors `Development` ;
- aucun DN libre fourni par le navigateur pour choisir le scope d'ecriture ;
- aucun hard delete AD, reset de mot de passe ou authentification portail
  contre AD ;
- aucune extension implicite vers la production.

## Garde-fous additionnels

- `API-INTERNAL` reste l'unique composant autorise a parler a AD et a
  MariaDB ;
- les jetons CSRF, sessions, mots de passe et chaines de connexion ne sont
  ni journalises ni affiches ;
- les actions AD sont auditees sans secret ;
- `WEBPORTAL` n'expose ni `INTERNAL_API_URL` ni `SERVICE_AUTH_TOKEN` au
  navigateur.
