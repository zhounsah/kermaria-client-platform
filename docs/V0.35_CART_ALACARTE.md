# V0.35 — Panier / commande groupee a la carte

Statut : **implemente dans le depot (2026-07-08)**. Le socle add / remove /
synthese fonctionne des le mode mock ; la confirmation (materialisation du
document + emission BPCE) est un chemin **MariaDB**, comme la facturation
d'abonnement (V0.22/V0.29). Reste borne a la phase de tests : mode `live`
interdit avant V1.0 beta 1. Recette a rejouer sur MariaDB + BPCE mock +
Stripe `test` / PayPal `sandbox`.

Cadrage initial : echange du 2026-07-07. Decisions tranchees et
implementation : 2026-07-08 (voir « Decisions » et « Implementation livree »
plus bas). La dependance a **V0.28** (packs) s'est averee **conceptuelle et
non bloquante** : V0.35 reutilise directement `commercial_document_lines`
sans les tables de bundles, donc V0.35 est livre avant V0.28.

## Decisions (2026-07-08)

- **Le panier produit directement un document commercial payable** (option 1
  du point ouvert #1), **sans approbation admin prealable**. Le client
  compose son panier, voit une synthese, confirme, puis regle.
- **Trois rails au reglement** : Stripe, PayPal **et virement bancaire**
  (bloc IBAN/BIC existant V0.21, pointe « paye » par l'admin). Aucun nouveau
  rail ni code de paiement : on reutilise `PayButton` + le bloc virement du
  document genere.
- **Provisioning automatique « le cas echeant » au reglement** : ce point
  **revise le garde-fou initial** (« aucun provisioning declenche par le
  panier »). Voir la section « Provisioning » ci-dessous.
- **Panier one-shot strict** : seules les offres `commercial_offers` de
  cadence `one_time`, actives et a prix > 0 sont eligibles ; toute offre
  recurrente est refusee (`CART_OFFER_NOT_ELIGIBLE`).
- **Etat du panier** : table dediee `cart_items` cote API-INTERNAL (une ligne
  par offre, quantite cumulee), rattachee au client — pas d'etat navigateur
  non maitrise.
- **Option indisponible entre l'ajout et la lecture** : auto-nettoyage
  silencieux (l'offre desactivee ou passee en recurrent est retiree du panier
  a la lecture suivante).

## Contexte : etat actuel de `/souscrire`

La page [`apps/webportal/app/souscrire/page.tsx`](../apps/webportal/app/souscrire/page.tsx)
gere deux flux, tous les deux **mono-article** :

1. **Packs grand public** — rendus via `PublicPackCard mode="subscribe"`,
   qui branche `SubscribeButton`. Le choix du rail de paiement
   **Stripe / PayPal est deja present** (radio affiche des que les deux
   rails sont actifs, defaut Stripe — livre en V0.29).
2. **Options a la carte** — chaque prestation affiche un bouton
   « Prendre cette option » qui renvoie vers
   `/request-service?service=<id>`. C'est une **demande** etudiee avant
   activation, **sans paiement direct** et **une option a la fois**.

Il n'existe aujourd'hui aucun moyen de selectionner plusieurs options a
la carte et de les traiter ensemble.

## Objectif V0.35

Introduire un **panier** cote portail client permettant de regrouper N
options a la carte en **une seule commande**, sans changer les rails de
paiement existants.

## Perimetre vise

- **Selection multiple** des prestations a la carte depuis `/souscrire`
  (ajouter / retirer une option, voir le recapitulatif).
- **Persistance du panier** rattachee a la session client cote
  API-INTERNAL (pas de nouvel etat global navigateur non maitrise, dans
  l'esprit des contraintes existantes du portail).
- **Materialisation en un seul document commercial multi-lignes** :
  reutiliser `commercial_documents` + `commercial_document_lines` (une
  ligne par option retenue) plutot que d'emettre N demandes
  `/request-service` distinctes. Idempotent.
- **Reglement via le flux existant** `PayButton` sur le document genere :
  le choix du rail Stripe vs PayPal se fait au niveau du document, comme
  aujourd'hui. **Aucun nouveau rail de paiement.**
- **Garde-fou cadence** : un panier ne regroupe que des options
  compatibles avec un reglement **one-shot**. Tout melange avec des
  offres `recurring` (abonnement) est refuse ou isole, coherent avec la
  regle V0.28 (« un pack ne peut etre recurring que si toutes ses offres
  filles ont la meme cadence »).

## Hors perimetre (explicitement)

- Aucun nouveau rail de paiement (le choix Stripe/PayPal est deja livre
  en V0.29 sur `SubscribeButton` et `PayButton`).
- Aucun checkout Stripe/PayPal **multi-panier natif** (panier envoye
  d'un bloc au PSP) : on passe par un document commercial regle en une
  fois, pas par un panier PSP.
- Aucun prelevement recurrent multi-articles.
- Aucune fonctionnalite hardware-gated ; aucun passage en mode `live`.

## Dependances et sequencement

- **V0.28** (packs / offres groupees) partage la meme mecanique
  « exploser une selection en N lignes filles » et les memes garde-fous
  de cadence. V0.35 doit venir **apres ou avec** V0.28 pour reutiliser
  ce socle plutot que de le dupliquer.
- Reutilise sans modification : `commercial_document_lines`, `PayButton`,
  les rails Stripe (V0.29) et PayPal (V0.21).

## Implementation livree (2026-07-08)

- **Migration `028_alacarte_cart.sql`** : table `cart_items`
  (`UNIQUE(customer_id, offer_id)`, FK vers `customers` et
  `commercial_offers`, `ON DELETE CASCADE`) + colonne
  `commercial_documents.origin` (tag `client_cart`).
- **API-INTERNAL** :
  - `CartService` porte les regles metier (eligibilite one-shot, quantite,
    jointure catalogue, auto-nettoyage, confirmation) ;
  - `ICartRepository` (Maria + Mock singleton store) : stockage
    (offre, quantite) rattache au client ;
  - `ICommercialRepository.CreateCartDocumentAsync` materialise **un** document
    `informational_invoice` multi-lignes en `shared_with_customer`, tag
    `origin = 'client_cart'`, totaux recalcules ; la confirmation l'emet
    aussitot (`IssueInvoiceAsync`, BPCE mock en phase de tests) pour le rendre
    payable ;
  - endpoints portail : `GET /internal/portal/cart`,
    `POST /internal/portal/cart/items`,
    `POST /internal/portal/cart/items/remove`,
    `POST /internal/portal/cart/confirm` (session client requise, bornes au
    customer de la session), audit `cart.*`.
- **WEBPORTAL** : routes BFF `/api/cart/*` (helpers `portal-bff`), getter
  serveur `getCart()`, composants `AddToCartButton`, `CartItemRemoveButton`,
  `CartConfirmButton`, page synthese `/panier`, section « Options a la carte »
  (offres one-shot) + barre d'acces panier sur `/souscrire`. La confirmation
  redirige vers `/commercial-documents/{id}` ou vivent deja `PayButton`
  (Stripe/PayPal) et le bloc virement (V0.21).
- **Tests** : `npm run test:cart`
  (`scripts/verify-cart-contract.mjs`, contrat statique de bout en bout —
  migration, contrats, repo, service, endpoints, BFF, UI, garde-fous).

## Provisioning « le cas echeant »

Ce point **revise le garde-fou de cadrage** (« aucun provisioning declenche
par le panier »). Au passage `issued -> paid` d'un document `client_cart`
(quel que soit le rail : Stripe / PayPal / virement, tous convergent sur
`InvoiceIssuingService.ConfirmPaymentAsync`), `CartProvisioningTrigger`
reconcilie le provisioning du client via le mecanisme existant
(`ISubscriptionProvisioningManager.ReconcileAsync`), en best-effort (n'echoue
jamais le paiement).

**Etat reel du catalogue courant** : le provisioning du projet est pilote par
les **abonnements** actifs (V0.31 : offre recurrente -> groupe AD). Le panier
V0.35 etant strictement **one-shot**, et aucune offre one-shot du catalogue ne
mappant de ressource provisionnable, ce declencheur est **inerte
aujourd'hui** (audit `cart.provisioning.skipped`). Il est neanmoins cable :
si une offre one-shot provisionnable est ajoutee au catalogue plus tard, le
reglement d'un panier la provisionnera automatiquement sans code
supplementaire.

## Reste / evolutions possibles

- Recette MariaDB complete du chemin confirm -> emission -> reglement ->
  provisioning sur les 3 rails.
- Mapping eventuel d'offres one-shot vers des ressources provisionnables (hors
  perimetre V0.35, dependant du catalogue).
- Champ quantite editable cote synthese (aujourd'hui : +1 par ajout, retrait
  par ligne).

## Rappel garde-fous projet

Conforme a `AGENTS.md` et `docs/SECURITY.md` : architecture
`browser -> WEBPORTAL/BFF -> API-INTERNAL -> MariaDB` inchangee, aucun
acces SQL direct cote WEBPORTAL, aucun mode `live` active. Le provisioning
declenche par le panier reste **borne au mecanisme AD existant** (reconcile
des abonnements, `OU=TEST_SITE_WEB`) et n'ouvre aucune ecriture nouvelle.
