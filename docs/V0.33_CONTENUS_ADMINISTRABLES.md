# V0.33 Contenus administrables

Statut : **implante dans le depot**. Cette documentation decrit l'etat
reel du code et de la base apres l'introduction du module de contenus
administrables pour les pages legales et les fiches techniques packs.

## Lecture rapide

Si vous reprenez le sujet sans contexte, l'ordre de lecture le plus
rentable est :

1. ce document pour comprendre le perimetre et le flux complet ;
2. `docs/ARCHITECTURE.md` pour la vue systeme ;
3. `docs/DATA_MODEL.md` pour les tables `managed_content_entries` et
   `public_pack_catalog_content` ;
4. `docs/API_CONTRACT.md` pour les surfaces BFF et internes ;
5. `docs/GUIDE_ADMIN.md` pour l'usage back-office ;
6. puis les fichiers clefs listes en fin de document.

## 1. Objectif

La V0.33 supprime le besoin de modifier manuellement des fichiers TSX
pour mettre a jour :

- les CGV ;
- les mentions legales ;
- les fiches techniques des packs publics ;
- et, dans l'implementation actuelle, la page `/a-propos`.

Le contenu est persiste en MariaDB, edite depuis le back-office,
affiche sur le site public sans redeploiement manuel, et rendu en
Markdown securise.

## 2. Perimetre reel livre

### Contenus geres

Le registre est **ferme** : il n'y a ni creation libre, ni suppression
libre, ni mini-CMS generic.

Cles actuellement supportees :

| Cle | Type | URL publique | Source initiale |
|---|---|---|---|
| `legal:cgv` | `legal` | `/cgv` | fichier seed UTF-8 |
| `legal:mentions-legales` | `legal` | `/mentions-legales` | fichier seed UTF-8 |
| `page:a-propos` | `page` | `/a-propos` | fichier seed UTF-8 |
| `pack-sheet:pack-dossier-securise` | `pack_sheet` | `/offres/dossier-securise` | generation applicative |
| `pack-sheet:pack-acces-distance` | `pack_sheet` | `/offres/acces-distance` | generation applicative |
| `pack-sheet:pack-bureau-windows-distance` | `pack_sheet` | `/offres/bureau-windows-distance` | generation applicative |
| `pack-sheet:pack-pro-association` | `pack_sheet` | `/offres/pro-association` | generation applicative |

### Ce qui reste hors perimetre

- `/politique-confidentialite` reste hors module V0.33 ;
- les titres, types et URLs publiques sont figes par la cle ;
- seul le couple `bodyMarkdown` + `versionLabel` est editable ;
- il n'y a pas d'editeur WYSIWYG : l'admin saisit du Markdown brut.

## 3. Architecture fonctionnelle

La V0.33 repose sur **trois couches distinctes** qu'il ne faut pas
confondre :

1. `PUBLIC_PACKS` dans `packages/shared` porte le manifeste fonctionnel
   des packs publics : slug, libelle, highlights, audience, references
   techniques, variantes 1/6/12 mois.
2. `public_pack_catalog_content` porte le contenu marketing de la page
   comparative `/offres` : hero, cartes, comparatif, textes de vitrine.
3. `managed_content_entries` porte les contenus editoriaux
   administrables : CGV, mentions legales, a-propos et fiche detaillee
   de chaque pack.

Consequence importante :

- la page `/offres` reste la vitrine comparative / marketing ;
- la page `/offres/[slug]` devient la fiche technique detaillee d'un
  pack ;
- les composants techniques lies aux packs ne sont pas dupliques dans
  le texte libre : ils sont recalcules depuis le manifeste et le
  catalogue commercial actif.

## 4. Source de verite partagee

Le module s'appuie sur des types communs exposes par
`packages/shared/src/index.ts` :

- `ManagedContentType = "legal" | "pack_sheet" | "page"`
- `ManagedContentKey`
- `ManagedContentSummary`
- `ManagedContentDetail`
- `ManagedContentPayload`
- `ManagedContentMutationResponse`
- `buildPackSheetContentKey(packCode)`
- `getManagedContentRegistry()`

Ce registre partage sert a garder les memes cles, libelles et URLs des
deux cotes :

- validation BFF Next ;
- navigation admin ;
- routing public ;
- contrats d'API ;
- documentation et tests statiques.

## 5. Persistance MariaDB

### 5.1 Table des contenus administrables

Migration : `apps/api-internal/Migrations/MariaDb/027_managed_content_entries.sql`

```sql
CREATE TABLE IF NOT EXISTS managed_content_entries (
    content_key VARCHAR(128) NOT NULL,
    content_type VARCHAR(32) NOT NULL,
    title VARCHAR(200) NOT NULL,
    public_path VARCHAR(255) NOT NULL,
    body_markdown LONGTEXT NOT NULL,
    version_label VARCHAR(160) NULL,
    created_at DATETIME(6) NOT NULL,
    updated_at DATETIME(6) NOT NULL,
    PRIMARY KEY (content_key)
);
```

Points d'attention :

- `content_key` est la cle stable et unique ;
- `title` et `public_path` sont aussi stockes en base, mais restent
  derives du registre ferme ;
- `body_markdown` contient le texte editable ;
- `version_label` est optionnel et visible publiquement ;
- tous les horodatages sont stockes en UTC.

### 5.2 Table de vitrine packs adjacente

Migration : `apps/api-internal/Migrations/MariaDb/026_public_pack_catalog_content.sql`

Cette table existe a cote du module :

- `public_pack_catalog_content(content_key, content_json, created_at, updated_at)`

Elle ne remplace pas `managed_content_entries` :

- `public_pack_catalog_content` pilote la vitrine `/offres` ;
- `managed_content_entries` pilote les pages legales, `a-propos` et
  les fiches techniques `/offres/[slug]`.

## 6. Backend API-INTERNAL

### 6.1 Service et repository

Fichiers principaux :

- `apps/api-internal/Contracts/ManagedContentContracts.cs`
- `apps/api-internal/Data/Repositories/IManagedContentRepository.cs`
- `apps/api-internal/Data/Repositories/MariaDbManagedContentRepository.cs`
- `apps/api-internal/Services/ManagedContentService.cs`

Responsabilites du service :

- valider la cle contre le registre ferme ;
- seed idempotent des entrees manquantes ;
- lecture publique ;
- lecture admin ;
- upsert admin ;
- normalisation du Markdown et des metadonnees.

Regles de validation actuelles :

- `bodyMarkdown` requis ;
- texte normalise et trimme ;
- longueur max `120000` caracteres ;
- `versionLabel` optionnel ;
- longueur max `160` caracteres ;
- aucune creation de cle non connue ;
- aucune suppression de contenu.

### 6.2 Seed idempotent

Le seed est **applicatif**, pas SQL :

- la migration cree la table ;
- `EnsureSeededAsync()` insere uniquement les cles manquantes ;
- un contenu deja edite en back-office n'est jamais ecrase par le seed ;
- le seed est rejoue automatiquement lors des lectures si necessaire.

#### Fichiers seed copies a la sortie API

Le projet `apps/api-internal/Kermaria.ApiInternal.csproj` copie :

- `SeedContent/cgv.md`
- `SeedContent/mentions-legales.md`
- `SeedContent/a-propos.md`

L'application les lit explicitement en `UTF-8`.

Regles de parsing :

- une premiere ligne `# ...` est ignoree ;
- une ligne `**Version du : ...**` ou
  `**Derniere mise a jour : ...**` alimente `versionLabel` ;
- le reste est stocke dans `bodyMarkdown`.

#### Fiches techniques seed

Les fiches techniques packs ne viennent pas d'un fichier brut. Elles
sont generees a partir :

- du manifeste `PUBLIC_PACKS` ;
- des `technicalServiceReferences` du pack ;
- du catalogue commercial courant.

Le seed de base cree les sections :

- `## Presentation`
- `## Composants techniques lies`
- `## Pre-requis`
- `## Limites`
- `## Support`

Le bloc "Composants techniques lies" dans `bodyMarkdown` reste
informatif ; la liste detaillee visible publiquement est recalculee a
l'affichage pour eviter les duplications.

## 8. Back-office webportal

### 8.1 Liste des contenus

Fonctions :

### 8.2 Edition d'un contenu

Page : `apps/webportal/app/admin/content/[key]/page.tsx`

Fonctions :

- `requireAdminSession()` ;
- normalisation de la cle avec `decodeURIComponent()` ;
- validation de la cle avec `isManagedContentKey()` ;
- affichage des metadonnees fixes ;
- lien "Voir la page publique" en nouvel onglet ;
- formulaire Markdown + version + apercu.

Composant client :

- `apps/webportal/components/AdminManagedContentForm.tsx`

Le formulaire appelle :

- `PATCH /api/admin/content/{key}`

Payload :

```json
{
  "bodyMarkdown": "## Titre\n\nContenu...",
  "versionLabel": "Version du : 07 juillet 2026"
}
```

### 8.3 BFF admin

Routes Next :

## 9. Site public

### 9.1 Pages legales et editoriales

Pages actuellement alimentees par le module :

- `apps/webportal/app/cgv/page.tsx`
- `apps/webportal/app/mentions-legales/page.tsx`
- `apps/webportal/app/a-propos/page.tsx`

Ces pages chargent `getPublicManagedContent(...)` puis rendent
`PublicManagedContentArticle`.

Presentation publique commune :

- eyebrow fixe ;
- titre fixe derive de la cle ;
- `versionLabel` si renseigne ;
- `updatedAt` visible ;
- rendu Markdown via le meme composant partage que l'admin.

### 9.2 Fiches techniques packs

Page : `apps/webportal/app/offres/[slug]/page.tsx`

Cette page combine :

1. le manifeste du pack (`PUBLIC_PACKS`) ;
2. le contenu public de la vitrine packs ;
3. le contenu administrable `pack-sheet:<code>` ;
4. le catalogue commercial public actif.

Blocs rendus :

- entete de la fiche technique ;
- lien retour vers `/offres` ;
- carte pack/CTA via `PublicPackCard` ;
- bloc calcule "Composants techniques lies" ;
- bloc Markdown editable "Details operationnels".

Les liens "Voir la fiche technique" sont exposes depuis :

- `PublicPackCard`
- `PublicPackComparisonTable`
- `/admin/public-pack-catalog` pour l'edition rapide cote admin

### 9.3 Rendu Markdown securise

Composant partage :

- `apps/webportal/components/ManagedMarkdown.tsx`

Choix retenus :

- `react-markdown`
- pas de `rehype-raw`
- pas de `dangerouslySetInnerHTML`
- HTML brut non interprete
- liens externes ouverts avec `target="_blank"` et
  `rel="noreferrer noopener"`

Le meme composant est utilise :

- en public ;
- dans l'apercu admin ;
- dans les fiches techniques packs ;
- dans les pages legales.

## 10. Routing public / shell

Point important pour ne pas se tromper en reprise :

- le choix `PublicShell` vs shell securise n'est plus documente comme
  dependant de `proxy.ts` ;
- la source de verite fonctionnelle est maintenant
  `apps/webportal/components/AppShell.tsx` ;
- `AppShell` utilise `usePathname()` + `isPublicRoute()` depuis
  `apps/webportal/lib/public-route-config.ts` ;
- `proxy.ts` peut encore exister, mais il n'est pas la reference
  principale pour ce split UX.

Cette nuance est importante pour les futurs correctifs UX lorsque
l'admin ouvre un lien public depuis le back-office.

## 13. Ajouter un nouveau contenu plus tard

Exemple : ajouter plus tard `politique-confidentialite`.

Ordre recommande :

1. Ajouter la cle dans `packages/shared/src/index.ts`.
2. L'ajouter au registre partage `getManagedContentRegistry()`.
3. L'ajouter au registre backend `ManagedContentService.Definitions`.
4. Ajouter un seed fichier ou une generation applicative.
5. Creer la page publique consommatrice dans `apps/webportal/app/...`.
6. Etendre au besoin les liens admin/public.
7. Etendre `verify-managed-content-contract.mjs`.
8. Verifier les smoke tests API si la surface change.

Le schema SQL n'a normalement pas besoin d'etre modifie pour ce type
d'extension.

## 14. Limites connues

- pas d'historique de versions editoriales au-dela de `updated_at` ;
- pas de workflow brouillon/publication ;
- pas de WYSIWYG ;
- pas d'import direct runtime depuis les fichiers `.txt` originaux :
  la source de seed runtime est `SeedContent/` dans `api-internal` ;
- la page `/politique-confidentialite` n'est pas encore migree vers ce
  module ;
- les URLs et titres publics restent verrouilles par conception.

## 15. Checklist de reprise / diagnostic

Quand quelque chose ne colle plus, l'ordre de verification le plus
rentable est en general :

1. verifier la cle concernee dans `packages/shared/src/index.ts` et dans
   `getManagedContentRegistry()` ;
2. verifier la definition miroir dans
   `apps/api-internal/Services/ManagedContentService.cs` ;
3. verifier la route publique attendue (`/cgv`, `/mentions-legales`,
   `/a-propos`, `/offres/[slug]`) ;
4. verifier que la page admin decode bien la cle URL-encodee
   (`decodeURIComponent`) ;
5. verifier que le lien public admin s'ouvre en nouvel onglet et que le
   shell public est bien decide par `AppShell.tsx` +
   `public-route-config.ts` ;
6. verifier en base la presence de l'entree
   `managed_content_entries.content_key` attendue ;
7. verifier `X-Data-Source` et les smoke tests si le doute porte sur le
   runtime reel ;
8. pour un pack, verifier separement :
   - le manifeste `PUBLIC_PACKS` ;
   - le contenu `public_pack_catalog_content` de `/offres` ;
   - le contenu `managed_content_entries` de `/offres/[slug]`.

Pour les sources legales initiales, les originaux fournis a l'origine
restent utiles comme reference humaine, mais le **runtime** seed
desormais depuis `apps/api-internal/SeedContent/`.
