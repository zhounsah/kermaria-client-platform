# V0.12 - Notifications portail

## Objectif

La V0.12 ajoute un centre d'activité client sans transport externe. Les
notifications sont consultées uniquement dans le portail authentifié.

```text
Navigateur -> WEBPORTAL/BFF -> API-INTERNAL -> MariaDB
```

## Événements visibles

Une notification est créée lorsque :

- le statut d'une demande support change réellement ;
- le statut d'une demande de service change réellement ;
- un administrateur ajoute un message public support ;
- un administrateur ajoute un message public service.

Une note interne, une consultation admin ou un changement vers le statut déjà
présent ne crée aucune notification.

## Isolation

Chaque notification possède un `customer_id`. API-INTERNAL résout ce client
depuis la session et l'utilise pour toutes les lectures et mutations.

Le client peut :

- consulter au maximum les 100 notifications les plus récentes ;
- marquer une notification lui appartenant comme lue ;
- marquer toutes ses notifications comme lues ;
- suivre le lien interne vers la demande concernée.

Une notification appartenant à un autre client reste inaccessible et est
retournée comme introuvable.

## Interface

- `/notifications` : centre d'activité complet ;
- `/dashboard` : aperçu des trois dernières notifications ;
- navigation client : accès direct « Notifications ».

Les messages sont courts et synthétiques. Ils ne recopient jamais le contenu
d'une note interne ou d'un message public complet.

## Migration

La migration additive
`apps/api-internal/Migrations/MariaDb/005_portal_notifications.sql` crée la
table `portal_notifications` et ses index. Elle ne supprime ni ne transforme
les données existantes.

Application contrôlée en développement :

```powershell
dotnet run --project .\apps\api-internal\Kermaria.ApiInternal.csproj -- --apply-migrations
```

Effectuer une sauvegarde MariaDB avant application sur un environnement
partagé.

## Validation


Avec les variables MariaDB locales déjà injectées :

```powershell

```

## Limites volontaires

La V0.12 n'ajoute aucun e-mail, SMTP, SMS, push, WebSocket, SignalR, worker,
queue, cron, AD, paiement, facturation réelle ou provisioning. Les
notifications nécessitent un rafraîchissement ou une navigation normale du
portail pour être visibles.
