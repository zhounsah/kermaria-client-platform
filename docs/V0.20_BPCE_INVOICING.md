# V0.20 Facturation reelle BPCE controlee

## Objet

La V0.20 introduit la premiere facturation fiscale reelle du portail via
l'API de gestion de factures de la Banque Populaire (groupe BPCE). Elle
remplace les "documents informatifs" de la V0.15 par des factures
numerotees et immuables emises cote banque.

L'integration reste entierement bornee par le principe de phase-de-tests :
le mode `live` est desactive par defaut et n'est jamais active sans
validation explicite.

Le flux autorise reste :

```text
browser -> WEBPORTAL / BFF -> API-INTERNAL -> BPCE
                                          -> MariaDB
```

`WEBPORTAL` ne contacte jamais BPCE directement.

## Modes d'integration

`BPCE_INTEGRATION_MODE` accepte trois valeurs :

- `disabled` (defaut) : aucune dependance BPCE, aucun appel sortant, les
  endpoints admin renvoient 503 ;
- `mock` : implementation en memoire pour les tests d'integration, conserve
  l'etat entre requetes via un singleton DI ;
- `live` : appels reels a l'API BPCE, emet de vraies factures fiscales cote
  banque. **Ne jamais activer sans validation explicite.**

## Variables d'environnement

| Variable | Role |
|---|---|
| `BPCE_INTEGRATION_MODE` | `disabled` / `mock` / `live` |
| `BPCE_BASE_URL` | `https://www.gestion-factures.banquepopulaire.fr` |
| `BPCE_REFRESH_TOKEN` | JWT long-lived BPCE, **secret strict** |
| `BPCE_SENDER_ID` | Profil de facturation BPCE, renseigne via `--verify-bpce-sender` |
| `BPCE_REQUEST_TIMEOUT_MS` | Timeout par defaut, 10 000 ms |

Le `BPCE_REFRESH_TOKEN` ne doit jamais etre commite ni journalise. Il vit
uniquement dans le fichier de secrets local et sera roule en V0.23b.

## Commande CLI `--verify-bpce-sender`

Commande lecture seule (aucune ecriture cote banque) qui :

1. echange le refresh token contre un access token JWT court ;
2. liste les profils de facturation visibles cote BPCE ;
3. permet de retrouver le `BPCE_SENDER_ID` a injecter dans la configuration.

La commande contourne le `RuntimeConfigurationValidator` car son objet
meme est de configurer la valeur que le validateur exige.

## Synchronisation client

Avant chaque emission, le client local est synchronise cote BPCE via
`POST /customers/` avec :

- `external_id` = reference externe client (idempotente) ;
- adresse, ville et `country: "FR"` (toujours en code ISO, jamais un texte
  libre) ;
- champs optionnels (e-mail, telephone) omis s'ils sont vides.

Le mapping local est persiste dans la table `bpce_customers` (migration
`008_bpce_invoicing`).

## Emission de facture

1. **Draft** : `POST /invoices/` avec les lignes (label, description,
  quantite, prix unitaire en chaine, taux TVA). Le titre est tronque a 30
  caracteres pour respecter la limite BPCE.
2. **Validate** : `POST /invoices/{id}/validate/`. BPCE alloue alors une
  numerotation fiscale immuable et la facture devient non modifiable.
3. **PDF** : `GET /invoices/{id}/pdf` retourne le binaire. Le PDF est
  archive en LONGBLOB cote API-INTERNAL avec son hash SHA-256 dans la
  table `bpce_invoices`. Tout acces ulterieur est servi depuis le cache
  local, jamais en re-tirant l'API BPCE.
4. **Document local** : `commercial_documents.status` passe a `issued`.

## Marquage comme payee

`POST /invoices/{id}/mark_as_paid/` est appele par
`InvoiceIssuingService.ConfirmPaymentAsync` lorsqu'un paiement est
confirme (cf. V0.21 PayPal). Le statut local passe a `paid`.

## Double persistance

L'API-INTERNAL maintient toujours la trace locale des emissions :

- `bpce_customers` : mapping reference externe ↔ identifiant BPCE ;
- `bpce_invoices` : id BPCE, numero fiscal, dates, PDF, hash, statut.

Cette double persistance protege contre une indisponibilite temporaire de
l'API BPCE et permet de continuer a servir les factures aux clients meme
si BPCE est en maintenance.

## Endpoints exposes

Cote admin via le BFF :

- `POST /internal/admin/commercial-documents/{id}/issue` : declenche le
  workflow draft→validate→cache PDF ;
- `GET /internal/admin/commercial-documents/{id}/invoice` : retourne les
  metadonnees de la facture BPCE associee ;
- `GET /internal/admin/commercial-documents/{id}/invoice/pdf` : retourne
  le PDF depuis le cache local.

Cote portail :

- `POST /internal/portal/commercial-documents/{id}/payment-confirm` :
  appele par le retour PayPal pour acter le paiement (V0.21).

Le telechargement PDF cote client est prevu en finition de V0.21 et
necessite un endpoint portail dedie avec controle d'ownership.

## Securite

- `BPCE_REFRESH_TOKEN` jamais journalise, jamais expose dans une reponse
  API, jamais commite ;
- les tokens d'acces sont mis en cache en memoire jusqu'a expiration moins
  60 secondes, avec verrou (`SemaphoreSlim`) pour eviter les rafraichissements
  concurrents ;
- en cas de 401, le cache est invalide et un seul retry est tente ;
- les corps d'erreur sont lus avant levee d'exception pour faciliter le
  diagnostic sans exposer de secret cote portail ;
- les PDF sont servis uniquement aux roles autorises et leur hash permet de
  detecter toute corruption stockage.

## Limites volontaires

La V0.20 ne fait pas :

- d'envoi e-mail (V0.21) ;
- de paiement en ligne (V0.21) ;
- de prelevement recurrent (V0.22) ;
- d'integration Chorus Pro ou EDI ;
- de rapprochement bancaire automatique ;
- de gestion des avoirs ou notes de credit.

Le mode `live` reste desactive tant que la cible R740xd n'est pas
validee (V0.23b).
