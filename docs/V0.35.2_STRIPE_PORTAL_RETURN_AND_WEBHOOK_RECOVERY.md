# V0.35.2 Correctif retours portail + reprise webhook Stripe

Statut : **implemente, deployee sur staging, et valide par rejeu webhook**
(SRV-01 / SRV-02, premiere execution le 2026-07-06, documentation de
reprise consolidee le 2026-07-09).

## Objet

Ce document explique de bout en bout :

- pourquoi un paiement Stripe/PayPal pouvait renvoyer vers `localhost`
  au lieu du domaine public ;
- pourquoi un abonnement Stripe pouvait rester en `pending_activation`
  alors que le paiement etait bien accepte par Stripe ;
- quels correctifs code, config, base de donnees et exploitation ont ete
  appliques ;
- comment rejouer un incident similaire sans repartir de zero.

Il complete `docs/V0.29_STRIPE_PAYMENTS.md`, qui decrit l'architecture
Stripe generale. Ici, on documente surtout le **correctif de terrain** et
le **runbook de reprise**.

## Resume executif

Le probleme ne venait pas d'une seule cause :

1. le portail staging pouvait encore utiliser une URL publique stale ou
   un fallback trop local, ce qui provoquait des retours vers
   `http://localhost:3000` ;
2. le webhook Stripe d'abonnement ne reconnaissait pas tous les variants
   utiles (`invoice.payment_succeeded` en plus de `invoice.paid`) ;
3. certains payloads Stripe modernes portent l'identifiant de souscription
   dans `data.object.parent.subscription_details.subscription` au lieu de
   `data.object.subscription` ;
4. l'API renvoyait `200 OK` meme quand le traitement webhook echouait, ce
   qui empechait Stripe de signaler clairement l'echec et de retenter dans
   de bonnes conditions ;
5. le deploiement sur SRV-02 a aussi revele un drift de config
   (`AD_REQUIRED_OU_ROOT`) et un drift d'historique de migrations
   (`021_subscription_provisioning` absent de `schema_migrations`).

Apres correction :

- l'URL canonique du portail est `https://portail.home.bzh` ;
- les retours Stripe/PayPal utilisent bien le domaine public ;
- le webhook Stripe peut retraiter un evenement en echec ;
- le paiement rejoue a active la souscription locale et genere la facture
  attendue ;
- le seul point restant observe sur le cas de test est un warning de
  provisioning (`PROVISIONING_NO_TARGET_USERS`) qui est separe du paiement.

## Symptomatique initiale

### 1. Redirection vers localhost

Apres approbation d'un paiement ou d'une souscription, le navigateur
pouvait etre renvoye vers une URL locale au lieu du portail public.

Symptomes vus ou attendus :

- ouverture de `http://localhost:3000/...` ;
- retour sur une URL non resolvable publiquement ;
- flux qui semble "bloque" alors que Stripe a bien encaisse.

### 2. Paiement accepte, mais rien ne se passe cote abonnement

Apres paiement Stripe, la fiche client montrait encore un etat proche de
"approuvee, activation en cours" ou "en attente", sans activation locale
ni facture BPCE attendue.

Le portail n'etait pas necessairement faux : Stripe avait bien paye, mais
le traitement serveur-a-serveur n'avait pas abouti correctement.

## Valeurs canoniques a retenir

### Domaine portail

Pour l'application, il ne faut **pas** injecter "les deux noms de domaine"
dans les variables runtime.

La valeur canonique a utiliser pour les retours, emails et callbacks
publics est :

`PUBLIC_PORTAL_URL=https://portail.home.bzh`

Et pour les scripts/outils de verification bases sur l'URL web :

`WEBPORTAL_BASE_URL=https://portail.home.bzh`

Important :

- `zacharyhounsa.ovh` peut exister comme domaine vitrine ou domaine
  annexe, mais **pas** comme deuxieme base URL applicative concurrente ;
- si un second domaine doit rester utilisable, il doit rediriger vers
  `https://portail.home.bzh` au niveau DNS / reverse proxy / IIS, pas via
  deux variables applicatives contradictoires ;
- en pratique, le portail doit avoir **un seul host canonique**.

### Staging AD

Sur SRV-02, tant que `AD_INTEGRATION_MODE` vaut `read_only` ou
`controlled_write`, la variable suivante est obligatoire :

`AD_REQUIRED_OU_ROOT=OU=TEST_SITE_WEB,DC=home,DC=bzh`

Sans elle, l'API refuse de demarrer.

## Causes racines

### A. Resolution d'URL publique trop permissive

Le portail utilisait deja `PUBLIC_PORTAL_URL` comme base publique, mais le
comportement effectif dependait encore du contexte runtime et du host
percu par le BFF. Derriere IIS / ARR, un fallback base uniquement sur
`nextUrl.origin` ou sur un host local peut reinjecter du `localhost`.

Le correctif final consiste a :

- lire proprement `x-forwarded-proto` et `x-forwarded-host` ;
- ne jamais privilegier une origine locale si une origine publique est
  disponible ;
- retomber sur `PUBLIC_PORTAL_URL` si les headers ne suffisent pas ;
- ne garder `http://localhost:3000` qu'en ultime fallback de dev.

### B. Couverture webhook Stripe incomplete

Le flux d'abonnement depend du webhook Stripe, pas du retour navigateur.
Or le traitement cote `api-internal` ne couvrait pas tout ce qui etait
necessaire pour le cas reel observe :

- `invoice.payment_succeeded` devait etre accepte au meme titre que
  `invoice.paid` ;
- l'ID de souscription devait etre lu dans les nouveaux payloads
  `parent.subscription_details.subscription` ;
- les evenements precedemment marques `failed` devaient pouvoir etre
  rejoues proprement.

### C. Idempotence sur `resource_id` pas assez robuste

Le dedoublonnage par ressource webhook etait trop fragile pour les payloads
longs ou heterogenes. La correction appliquee :

- normalise `resource_id` dans un helper commun ;
- tronque de facon uniforme a 255 caracteres ;
- aligne le schema MariaDB sur `VARCHAR(255)` pour `paypal_webhook_events`
  et `stripe_webhook_events`.

### D. Mauvais signal HTTP en cas d'echec webhook

Le endpoint `/internal/webhooks/stripe` devait renvoyer un code d'erreur
quand le traitement metier echoue, sinon Stripe et les operateurs voient
un "succes" HTTP alors que le traitement a en fait rate.

Le correctif renvoie maintenant `500` quand `result.Status == "failed"`.

### E. Drifts annexes revelees par le redeploiement

Le redeploiement sur staging a fait remonter deux problemes
independants du bug fonctionnel initial :

1. `AD_REQUIRED_OU_ROOT` etait desormais requis par
   `RuntimeConfigurationValidator` ;
2. la base possedait deja le schema de `021_subscription_provisioning`,
   mais l'entree correspondante etait absente de `schema_migrations`,
   ce qui bloquait l'application de `022_webhook_resource_id_length`.

## Correctifs implementes dans le depot

### 1. WEBPORTAL - URL publique portail

### Fichier cle

`apps/webportal/lib/public-routes.ts`

Le helper central :

- introduit `getOriginFromHeaders(headers)` ;
- introduit `getPortalPublicUrlFromHeaders(headers)` pour les usages sans
  `NextRequest` complet ;
- garde `getPortalPublicUrl(request?)` comme point d'entree pour les routes
  applicatives ;
- priorise une origine publique issue des headers proxifies ;
- fallback ensuite sur `PUBLIC_PORTAL_URL` ;
- ne revient a `localhost` qu'en dernier ressort.

### Routes deja alignees sur `getPortalPublicUrl(request)`

Les routes de retour / creation de session utilisent la base portail
publique issue du helper :

- `apps/webportal/app/api/payment/paypal/create/route.ts`
- `apps/webportal/app/api/payment/paypal/return/route.ts`
- `apps/webportal/app/api/payments/stripe/create-intent/route.ts`
- `apps/webportal/app/api/payments/stripe/return/route.ts`
- `apps/webportal/app/api/subscriptions/create/route.ts`
- `apps/webportal/app/api/subscriptions/return/route.ts`
- `apps/webportal/app/api/subscriptions/stripe/return/route.ts`

Ce point est critique : pour Stripe, le retour navigateur n'active pas la
souscription, mais il doit quand meme ramener le client sur une URL
publique valide et coherente.

### 2. API-INTERNAL - Traitement webhook Stripe

### Fichier cle

`apps/api-internal/Services/StripeWebhookService.cs`

Le service :

- autorise le rejeu d'un evenement Stripe precedemment marque `failed` ;
- gere `invoice.paid` **et** `invoice.payment_succeeded` ;
- ignore proprement un `payment_intent.succeeded` sans
  `metadata.document_id` au lieu de faire echouer tout le webhook ;
- dedoublonne les succes de facture d'abonnement par `resource_id`
  normalise ;
- lit l'ID de souscription soit dans `data.object.subscription`, soit dans
  `data.object.parent.subscription_details.subscription` ;
- active la souscription locale si elle est encore en
  `pending_approval` / `pending_activation` ;
- cree puis emet la facture BPCE de cycle ;
- confirme le paiement localement avec `payment_method = "stripe"` ;
- enregistre l'audit `subscription.payment_received`.

### Repository Stripe

Fichiers :

- `apps/api-internal/Data/Repositories/IStripeWebhookRepository.cs`
- `apps/api-internal/Data/Repositories/MariaDbStripeWebhookRepository.cs`
- `apps/api-internal/Data/Repositories/MockStripeWebhookRepository.cs`

Le repository expose `HasProcessedInvoiceSuccessEventAsync(...)` pour
eviter les doubles traitements sur la meme facture Stripe, quel que soit
le nom exact de l'evenement de succes (`invoice.paid` ou
`invoice.payment_succeeded`).

### Normalisation partagee des resource ids

Fichiers :

- `apps/api-internal/Data/Repositories/WebhookResourceIdNormalizer.cs`
- `apps/api-internal/Data/Repositories/MariaDbPayPalWebhookRepository.cs`
- `apps/api-internal/Data/Repositories/MockPayPalWebhookRepository.cs`
- `apps/api-internal/Data/Repositories/MariaDbStripeWebhookRepository.cs`
- `apps/api-internal/Data/Repositories/MockStripeWebhookRepository.cs`

Le helper centralise la regle suivante :

- `Trim()`
- `null` si vide
- tronquage a 255 caracteres max

L'objectif est double :

- eviter les divergences de format entre Stripe et PayPal ;
- garantir que la persistance et la recherche d'evenements utilisent la
  meme representation.

### 3. Route webhook HTTP

Fichier :

`apps/api-internal/Program.cs`

La route `POST /internal/webhooks/stripe` :

- retourne `200` si le traitement aboutit ou est ignore ;
- retourne `500` si `StripeWebhookService` signale `failed`.

Ce comportement est indispensable pour qu'un echec metier ne soit plus
masque par un "succes" transport.

### 4. Migration MariaDB

Fichier :

`apps/api-internal/Migrations/MariaDb/022_webhook_resource_id_length.sql`

Migration appliquee :

- `paypal_webhook_events.resource_id -> VARCHAR(255)`
- `stripe_webhook_events.resource_id -> VARCHAR(255)`

### 5. Verification de contrat cote webportal

Fichier :

`apps/webportal/scripts/verify-stripe-contract.mjs`

La verification automatisee couvre explicitement `invoice.payment_succeeded`
en plus de `invoice.paid`, pour eviter une regression silencieuse.

## Rejeu webhook Stripe

### Principe

Quand un evenement Stripe a deja ete recu mais marque `failed`, il ne faut
pas necessairement refaire tout le paiement si le payload brut est encore
stocke.

Le chemin de reprise recommande est :

1. extraire `raw_payload` depuis `stripe_webhook_events` ;
2. rejouer ce payload tel quel vers
   `POST /internal/webhooks/stripe` ;
3. verifier que le statut local passe a `processed` ;
4. verifier la souscription, la facture BPCE et les audits.

### Methode qui a fonctionne

`Invoke-RestMethod` s'est revele moins fiable sur ce cas de payload brut.
Le rejeu concluant a ete fait avec `curl.exe --data-binary`, ce qui
preserve mieux le JSON exact.

Forme type :

```powershell
curl.exe `
  -X POST "http://192.168.100.202:5000/internal/webhooks/stripe" `
  -H "Content-Type: application/json" `
  -H "X-Service-Auth: <token_service>" `
  --data-binary "@payload.json"
```

Le log a encore montre :

`SubscriptionProvisioningManager -> PROVISIONING_NO_TARGET_USERS`

pour le client demo `CLI-DEMO-0060`.

Ce point ne bloque pas l'encaissement Stripe ni l'activation locale de la
souscription. Il concerne la cible de provisioning technique apres
paiement.

## Runbook court pour une prochaine fois

Si un paiement "ne fait rien" apres retour Stripe :

1. verifier d'abord le domaine public dans
   `C:\ProgramData\Kermaria\webportal.config.json` ;
2. verifier ensuite l'etat du webhook dans `stripe_webhook_events` ;
3. si `failed`, extraire `raw_payload` et rejouer ;
4. si le service ne redemarre pas, controler en priorite
   `AD_REQUIRED_OU_ROOT` ;
5. si une migration refuse de passer alors que le schema semble deja la,
   comparer `schema_migrations` au schema reel avant d'appliquer un nouveau
   correctif.

## Ce qu'il ne faut plus refaire

- ne pas mettre plusieurs noms de domaine concurrents dans
  `PUBLIC_PORTAL_URL` ;
- ne pas supposer que le retour navigateur Stripe active la souscription ;
- ne pas considerer un `200 OK` transport comme une preuve de succes
  metier sans verifier `status` et la base locale ;
- ne pas relancer des migrations a l'aveugle sans verifier l'historique
  `schema_migrations`.

## Conclusion

Le correctif ne se limite pas a "changer une URL".

Le sujet combine :

- une URL publique canonique ;
- un comportement BFF correct derriere proxy ;
- un traitement webhook Stripe tolerant aux variants reels ;
- une idempotence robuste ;
- une procedure de rejeu exploitable ;
- une hygiene de deploiement / migration suffisante pour staging.

Tant que ces six pieces restent alignees, le flux Stripe d'abonnement
redevient comprehensible, rejouable et maintenable.
