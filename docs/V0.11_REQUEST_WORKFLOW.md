# V0.11 - Workflow des demandes

## Objectif

La V0.11 transforme les demandes support et de service en objets de suivi
simples. Le client consulte leur état et les messages qui lui sont destinés.
Un administrateur interne peut changer leur statut, ajouter une note interne
ou publier un message dans le portail.

Le flux reste :

```text
Navigateur -> WEBPORTAL/BFF -> API-INTERNAL -> MariaDB
```

Le navigateur ne connaît ni `INTERNAL_API_URL`, ni `SERVICE_AUTH_TOKEN`, ni
les identifiants MariaDB.

## Statuts

Support :

| Code | Libellé |
|---|---|
| `open` | Ouverte |
| `in_progress` | En cours |
| `waiting_for_customer` | En attente client |
| `resolved` | Résolue |
| `closed` | Clôturée |
| `cancelled` | Annulée |

Demande de service :

| Code | Libellé |
|---|---|
| `received` | Reçue |
| `under_review` | En étude |
| `accepted` | Acceptée |
| `rejected` | Refusée |
| `cancelled` | Annulée |
| `completed` | Terminée |

Tous les passages entre statuts valides sont autorisés en V0.11. Un passage
vers le statut courant est un no-op réussi et ne crée pas d'événement en
double. Une demande de service acceptée reste à traiter manuellement.

## Visibilité

- `request_events` : historique de création et de statut visible du client.
- `request_public_messages` : messages admin visibles du client sous
  « Équipe Kermaria ».
- `request_internal_notes` : notes strictement réservées à l'administration.

Les trois ensembles sont append-only. Les textes sont affichés sans
interprétation HTML. Une note ou un message contient entre 3 et 2 000
caractères.

## Administration

Les pages `/admin/support-requests/[id]` et
`/admin/service-requests/[id]` proposent trois formulaires distincts :

1. changement de statut ;
2. ajout d'une note interne ;
3. ajout d'un message public.

Cette séparation réduit le risque de publier une information destinée à
l'interne. L'interface rappelle de ne jamais saisir de mot de passe, token ou
secret.

Les listes restent limitées à 100 lignes et acceptent des filtres simples par
statut, priorité pour le support et ordre chronologique.

## Côté client

Les pages `/support/[id]` et `/request-service/[id]` sont protégées par la
session. API-INTERNAL vérifie que la demande appartient au `customer_id` issu
de cette session. Aucun identifiant client fourni par le navigateur n'est
utilisé comme autorité.

Le client voit :

- les métadonnées utiles ;
- le statut et sa description en français ;
- l'historique de création et de statut ;
- les messages publics.

Il ne reçoit jamais les notes internes ni l'identité de l'administrateur.

## Migration

La migration additive
`apps/api-internal/Migrations/MariaDb/004_request_workflow.sql` crée les trois
tables du workflow et ajoute un événement `created` aux demandes existantes.
Elle ne supprime aucune donnée et ne modifie pas les statuts déjà enregistrés.

Application contrôlée :

```powershell
dotnet run --project .\apps\api-internal\Kermaria.ApiInternal.csproj -- --apply-migrations
```

Une sauvegarde MariaDB doit être réalisée avant application sur un
environnement partagé.

## Limites volontaires

La V0.11 n'ajoute aucun commentaire client entrant, e-mail, suppression,
provisioning, action AD, paiement ou facturation réelle. Un changement de
statut reste une opération de suivi et ne déclenche aucune automatisation.
