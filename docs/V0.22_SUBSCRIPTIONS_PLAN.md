# V0.22 — Plan d'implémentation détaillé

Ce document complète [`V0.22_SUBSCRIPTIONS.md`](V0.22_SUBSCRIPTIONS.md)
(cadrage fonctionnel) avec un découpage opérationnel en phases commitables
indépendamment. Rédigé en clôture V0.21 le **2026-06-25** pour servir de
point d'entrée à une session de développement future.

## Contexte d'entrée

Au démarrage de V0.22, l'état du dépôt inclut :

- V0.21 entièrement livré : PayPal Orders one-shot, IBAN, PDF client, vue
  admin paiements, marquage manuel, canal e-mail transactionnel mock/live,
  journal `email_messages`.
- Helper `lib/paypal.ts` : OAuth2 client credentials avec cache, créa
  d'ordre `intent: CAPTURE`. À étendre pour l'API Subscriptions.
- Tables existantes pertinentes : `commercial_offers`, `commercial_documents`,
  `bpce_invoices`, `email_messages`, `customers`.
- Modes : `PAYPAL_MODE=sandbox` par défaut, `BPCE_INTEGRATION_MODE=disabled`
  par défaut. Le mode `live` reste interdit jusqu'à V0.23b.

## Découpage en 6 phases

Chaque phase = 1 commit cohérent + 1 testable. Ordre strict (chaque phase
dépend de la précédente).

### Phase A — Schema catalogue + cadence (~30 min)

**But** : permettre à un article du catalogue d'être `one_time` ou
`recurring_monthly`.

Livrables :

- migration `011_subscription_offers.sql` : ajoute à `commercial_offers`
  - `billing_cadence ENUM('one_time','monthly') NOT NULL DEFAULT 'one_time'`
  - `paypal_plan_id VARCHAR(64) NULL`
- mise à jour `CommercialOfferSummary` (shared) et `CommercialOfferPayload`
- form admin catalog : sélecteur cadence + champ texte `paypal_plan_id`
  (rempli manuellement depuis le dashboard PayPal)
- filtre/badge visuel sur la page catalogue (mensuel vs ponctuel)

Validation : un admin peut créer un article "VPN mensuel - 50€/mois" et y
associer un identifiant `P-xxx` PayPal Plan créé à la main.

### Phase B — Table `subscriptions` + service (~45 min)

**But** : modèle de données pour stocker les souscriptions actives.

Livrables :

- migration `012_subscriptions.sql` :
  ```sql
  subscriptions (
    id CHAR(36) PRIMARY KEY,
    customer_id CHAR(36) NOT NULL FK -> customers,
    commercial_offer_id CHAR(36) NOT NULL FK -> commercial_offers,
    paypal_subscription_id VARCHAR(64) UNIQUE,
    paypal_plan_id VARCHAR(64) NOT NULL,
    status ENUM('pending_approval','active','suspended','cancelled','expired')
      NOT NULL DEFAULT 'pending_approval',
    started_at DATETIME(6) NULL,
    next_billing_at DATETIME(6) NULL,
    cancelled_at DATETIME(6) NULL,
    created_at DATETIME(6) NOT NULL,
    updated_at DATETIME(6) NOT NULL
  )
  ```
- `ISubscriptionRepository` + `MariaDbSubscriptionRepository` + `MockSubscriptionRepository`
- `ISubscriptionService` : `GetClientSubscriptionsAsync(session)`,
  `GetAdminSubscriptionsAsync()`, `GetSubscriptionAsync(id)`
- enregistrement DI dans `Program.cs`

Validation : `npm run test:bpce` ne casse pas. Pas d'UI à ce stade.

Important : utiliser `UTC_TIMESTAMP(6)` partout, pas `NOW(6)` (cf. commits
d29bacf et le fix V0.20 BPCE timestamps).

### Phase C — Flux de souscription client (~1h)

**But** : un client peut souscrire à une offre `recurring_monthly` depuis
le portail.

Livrables :

- helper `lib/paypal.ts` : `createPayPalSubscription(planId, subscriberEmail,
  returnUrl, cancelUrl)` via `POST /v1/billing/subscriptions`
- BFF `POST /api/subscriptions/create` :
  - lit l'offre, valide qu'elle a `billing_cadence='monthly'` et
    `paypal_plan_id` non null
  - crée la souscription PayPal en `APPROVAL_PENDING`
  - persiste la row locale en `pending_approval`
  - retourne l'`approveUrl`
- BFF `GET /api/subscriptions/return?subscription_id=...` : callback après
  approbation, met à jour la row en `pending_activation` (l'activation
  finale arrive via webhook)
- page client `/profile/subscriptions` : liste des souscriptions actives
  avec statut, lien vers les factures générées
- bouton "Souscrire" sur la page d'offre client (visible uniquement si
  `billing_cadence='monthly'` et `paypal_plan_id` configuré)

Validation : un client peut cliquer "Souscrire à VPN mensuel" → PayPal
sandbox → retour portail avec subscription en `pending_approval`.

### Phase D — Webhook PayPal (~1h30, morceau le plus délicat)

**But** : recevoir les événements PayPal et orchestrer la facturation BPCE
mensuelle.

Livrables :

- endpoint `POST /api/webhooks/paypal` (public, **pas de CSRF**, validation
  par signature PayPal `PAYPAL-AUTH-ALGO/CERT-URL/TRANSMISSION-ID/TRANSMISSION-SIG`)
- helper `verifyPayPalWebhookSignature` — appelle
  `POST /v1/notifications/verify-webhook-signature` côté PayPal
- migration `013_paypal_webhook_events.sql` :
  ```sql
  paypal_webhook_events (
    id CHAR(36) PRIMARY KEY,
    event_id VARCHAR(64) UNIQUE NOT NULL,
    event_type VARCHAR(64) NOT NULL,
    resource_id VARCHAR(64) NULL,
    received_at DATETIME(6) NOT NULL,
    processed_at DATETIME(6) NULL,
    status ENUM('received','processed','failed','ignored') NOT NULL,
    error_message TEXT NULL,
    raw_payload JSON NOT NULL
  )
  ```
- switch sur `event_type` :
  - `BILLING.SUBSCRIPTION.ACTIVATED` → `subscription.status = active`,
    calcule `next_billing_at` (J+1 mois)
  - `PAYMENT.SALE.COMPLETED` → crée un `commercial_document` (lignes copiées
    de l'offre) → émet la facture BPCE → `mark_as_paid` → envoie email
    `payment_confirmed`
  - `BILLING.SUBSCRIPTION.SUSPENDED` → `status = suspended`
  - `BILLING.SUBSCRIPTION.CANCELLED` → `status = cancelled`,
    `cancelled_at = UTC_TIMESTAMP(6)`
  - `BILLING.SUBSCRIPTION.EXPIRED` → `status = expired`
  - autres → ignoré, status=`ignored`
- env vars : `PAYPAL_WEBHOOK_ID`, `PAYPAL_WEBHOOK_VERIFY` (`true` défaut,
  `false` autorisé seulement en dev pour debug)

Validation : envoi simulé via `curl` avec payload sandbox, OU déclenchement
réel via PayPal sandbox webhook simulator du dashboard developer.

Risque principal : la validation de signature webhook est piégeuse en
PayPal sandbox. Avoir un mode `PAYPAL_WEBHOOK_VERIFY=false` permet
d'avancer sans bloquer. **Doit être `true` en mode `live`** (V0.23b).

### Phase E — Admin UI (~1h)

**But** : l'admin peut suivre, suspendre et annuler les abonnements.

Livrables :

- page `/admin/subscriptions` : liste filtrable (statut, customer), totaux
  (souscriptions actives, MRR estimé en €)
- page détail `/admin/subscriptions/{id}` :
  - infos générales (client, offre, plan PayPal, statut, prochaine échéance)
  - historique des factures BPCE générées (jointure sur `commercial_documents`
    via webhook `PAYMENT.SALE.COMPLETED`)
  - bouton "Annuler" → `POST /api/admin/subscriptions/{id}/cancel`
- endpoint API-INTERNAL `POST /internal/admin/subscriptions/{id}/cancel` :
  - appelle `POST /v1/billing/subscriptions/{id}/cancel` chez PayPal
  - met à jour le statut local
  - audit `subscription.admin_cancel`
- onglet "Abonnements" dans la nav admin entre Paiements et E-mails

### Phase F — Closure : docs, tests, ROADMAP (~30 min)

Livrables :

- `apps/webportal/scripts/verify-subscriptions-contract.mjs` (style V0.21),
  ajoute `npm run test:subscriptions`
- mise à jour `docs/V0.22_SUBSCRIPTIONS.md` : remplacer le cadrage par la
  doc d'implémentation effective
- mise à jour `docs/ROADMAP.md` : V0.22 passe à `implemente dans le depot`
- mise à jour `.env.example` : `PAYPAL_WEBHOOK_ID`, `PAYPAL_WEBHOOK_VERIFY`
- mise à jour mémoire `roadmap-current.md` (snapshot post-V0.22)
- supprimer ce fichier (`V0.22_SUBSCRIPTIONS_PLAN.md`) car obsolète

## Variables d'environnement à prévoir

| Variable | Rôle | Défaut |
|---|---|---|
| `PAYPAL_WEBHOOK_ID` | ID du webhook configuré dans le dashboard PayPal, sert à la validation de signature | vide (verify off) |
| `PAYPAL_WEBHOOK_VERIFY` | `true` / `false` — désactivable en dev local pour debug | `true` |
| `PUBLIC_PORTAL_URL` | déjà existant en V0.21, sert aux return/cancel URLs du flow subscribe | aucun |

## Garde-fous transverses (rappel)

- `PAYPAL_MODE=sandbox` par défaut, **interdiction d'activer `live` avant
  V0.23b**
- webhook signature vérifiée en mode `live`, exigée même en sandbox sauf
  override explicite
- idempotence stricte sur `paypal_webhook_events.event_id` UNIQUE — un
  même event reçu 2× ne génère qu'une seule facture BPCE
- pas d'annulation automatique côté DB sur erreur PayPal — l'admin tranche
  manuellement
- audit `subscription.created`, `subscription.activated`,
  `subscription.cancelled`, `subscription.payment_received`,
  `subscription.admin_cancel` avec `correlation_id`
- toutes les datetimes en UTC (`UTC_TIMESTAMP(6)` côté SQL, `DateTime.UtcNow`
  côté C#)

## Ordre d'attaque recommandé

A → B → C → D → E → F en séquence. **Ne sauter aucune phase** : C dépend
de B (modèle), D dépend de C (souscription approuvée à activer), E dépend
de D (statut alimenté par webhook).
