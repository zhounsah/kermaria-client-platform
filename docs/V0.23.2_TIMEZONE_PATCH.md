# V0.23.2 — Patch harmonisation horodatages (Europe/Paris)

## Motivation

Jusqu'a V0.23.1, tout le processus `api-internal` fonctionne en UTC :
`DateTime.UtcNow` cote C#, sortie JSON ISO 8601 avec suffixe `Z`, timestamps
de logs en UTC. Cote front, `formatDate` / `formatDateTime`
(`apps/webportal/lib/formatters.ts`) utilisent `Intl.DateTimeFormat("fr-FR", …)`
**sans** parametre `timeZone`. Resultat en Server Components :
`new Date("2026-07-02T13:00:00Z")` est formate dans le fuseau du conteneur
Node, qui est UTC en production. L'utilisateur voit alors `13:00` la ou il
attend `15:00` heure de Paris (ete = UTC+2). Un decalage identique de +1h
apparait en hiver (UTC+1). Les logs partagent le meme travers cote
exploitant.

V0.23.2 corrige la chaine d'affichage pour rendre tous les horodatages en
**Europe/Paris**, avec bascule automatique heure d'ete / heure d'hiver.
MariaDB continue a stocker en UTC (choix delibere : les timestamps
persistes restent stables et comparables independamment du fuseau).

Pas de migration, pas de refactor : c'est un patch cross-cutting borne aux
points de rendu et de sortie texte.

## Portee

- **Affichage portail et admin** : `formatDate` + `formatDateTime` acceptent
  desormais un `timeZone: DISPLAY_TIME_ZONE` (constante = `"Europe/Paris"`,
  IANA — la bascule DST est native). Toutes les 33 pages qui consomment ces
  helpers heritent du fix sans modification. Aucun `toLocaleString`
  contournant les helpers n'a ete trouve.
- **Templates email** (`invoice_issued`, `payment_reminder`,
  `payment_confirmed`, `contact_form`) : verifies un a un — aucune date
  inline n'est rendue dans le corps, donc rien a patcher cote template.
- **Date fiscale envoyee a BPCE** : `InvoiceIssuingService.cs:138`
  emmettait `DateTime.UtcNow.ToString("yyyy-MM-dd")`. Un emission entre
  00h00 et 02h00 heure de Paris ete aurait envoye la date de la veille a
  BPCE. Corrige en `KermariaTimeZone.Now.ToString("yyyy-MM-dd")`.
- **Logs api-internal** :
  - `Program.cs` `AddJsonConsole` : `UseUtcTimestamp = false` + format
    `yyyy-MM-ddTHH:mm:ss.fffzzz` (offset ISO 8601 explicite `+02:00` ete /
    `+01:00` hiver).
  - `FileLoggerProvider.cs` : rollover du nom de fichier (`api-internal-{date}.log`)
    et timestamp d'entree passes en `KermariaTimeZone.Now` avec le meme
    format `zzz`. La purge (`GetCreationTimeUtc` + `UtcNow.AddDays(-N)`)
    reste en UTC — c'est une comparaison de duree, non un affichage.
- **Audit events** : rendus via `formatDateTime` sur
  `/admin/audit-logs` et `/admin/activity`, donc couverts par le fix front.

## Implementation

### Helper C# partage

Nouveau `apps/api-internal/Infrastructure/KermariaTimeZone.cs` :

```csharp
public static class KermariaTimeZone
{
    public const string IanaId = "Europe/Paris";
    private const string WindowsId = "Romance Standard Time";

    private static readonly TimeZoneInfo _tz = ResolveTimeZone();

    public static TimeZoneInfo TimeZone => _tz;
    public static DateTime Now => TimeZoneInfo.ConvertTimeFromUtc(DateTime.UtcNow, _tz);
    public static DateTime ToLocal(DateTime utc) => TimeZoneInfo.ConvertTimeFromUtc(
        DateTime.SpecifyKind(utc, DateTimeKind.Utc), _tz);

    private static TimeZoneInfo ResolveTimeZone()
    {
        try { return TimeZoneInfo.FindSystemTimeZoneById(IanaId); }
        catch (TimeZoneNotFoundException)
        {
            return TimeZoneInfo.FindSystemTimeZoneById(WindowsId);
        }
    }
}
```

Le fallback `Romance Standard Time` couvre les hotes Windows anciens ou
la table IANA n'est pas installee (.NET 8+ l'a nativement mais la
robustesse coute deux lignes).

### Formatters front

`apps/webportal/lib/formatters.ts` expose `DISPLAY_TIME_ZONE = "Europe/Paris"`
et le passe aux deux `Intl.DateTimeFormat`. C'est le seul point de
transformation cote client — tous les composants passent par la.

## Tests

Nouveau script `apps/webportal/scripts/verify-timezone-contract.mjs`
enregistre comme `npm run test:timezone` (racine + apps/webportal). Il
combine :

- assertions statiques sur `formatters.ts`, `KermariaTimeZone.cs`,
  `InvoiceIssuingService.cs`, `FileLoggerProvider.cs`, `Program.cs` (les
  chaines exactes attendues doivent etre presentes) ;
- assertions runtime `Intl.DateTimeFormat` sur trois inputs UTC connus :
  - `2026-07-02T13:00:00Z` -> `15:00` (ete UTC+2) ;
  - `2026-01-15T13:00:00Z` -> `14:00` (hiver UTC+1) ;
  - `2026-03-29T01:30:00Z` -> `03:30` (bascule ete : 02:00 -> 03:00
    heure Paris, donc 01:30 UTC saute a 03:30 Paris).

Ces trois cas valident que la conversion DST est bien automatique.

## Prerequis de deploiement

Aucune migration DB. Le container `api-internal` doit avoir la table
timezone IANA disponible. Sur les images Docker officielles Microsoft
`aspnet:8.0`+ c'est le cas depuis 2023 ; sur images Alpine minimales,
installer `tzdata`. Le fallback Windows est un filet de securite, pas une
strategie de deploiement.

## Non-scope

- Pas de conversion cote MariaDB (stockage reste UTC).
- Pas d'ajout de configuration `TZ` env-var — le helper C# ecrase le
  besoin, et cote Node/Next les Intl formatters sont deterministes une
  fois `timeZone` fourni explicitement.
- Pas de retrofit des tests existants (verify-*-contract.mjs autres que
  timezone) : ils operent sur des strings UTC `Z` en ISO 8601, le patch
  ne change pas les payloads.

## Points d'attention (a garder en tete)

- Toute nouvelle date affichee cote front doit passer par `formatDate` /
  `formatDateTime`. Un `new Date().toLocaleString("fr-FR")` sans
  `timeZone` reintroduirait le bug en Server Component.
- Toute nouvelle date-only fiscale ou d'exploitation cote C# doit
  utiliser `KermariaTimeZone.Now` (jamais `DateTime.UtcNow.ToString("yyyy-MM-dd")`).
- Les timestamps ISO renvoyes en JSON cote api-internal restent en UTC
  avec `Z` — ne pas les convertir a la source, sinon on decale deux
  fois.
