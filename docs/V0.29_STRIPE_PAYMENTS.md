# V0.29 Stripe comme rail de paiement parallele

Statut : **implemente dans le depot** (2026-07-02).

Addendum important : pour le correctif de terrain sur les retours portail,
la reprise webhook et le redeploiement staging SRV-01 / SRV-02, voir
`docs/V0.35.2_STRIPE_PORTAL_RETURN_AND_WEBHOOK_RECOVERY.md`.

## Objet

V0.29 ajoute Stripe en parallele de PayPal, sans le remplacer : paiement
one-shot (V0.21) et abonnements mensuels (V0.22). Le client choisit son
rail au moment de regler une facture ou de souscrire un service, avec un
defaut sur Stripe des que `STRIPE_MODE != disabled`.

## Architecture

```text
browser
  -> WEBPORTAL/BFF (Next.js)
     -> API-INTERNAL (.NET) -> MariaDB
     -> Stripe Checkout Sessions (test/live)
Stripe -> WEBPORTAL/BFF (POST /api/webhooks/stripe)
       -> API-INTERNAL (POST /internal/webhooks/stripe)
         -> Subscriptions / BPCE / Email
```

Meme principe de layering que PayPal : toutes les requetes HTTP sortantes
vers l'API Stripe vivent dans `apps/webportal/lib/stripe.ts` (`fetch` brut,
aucun SDK). Le backend C# ne parle jamais a Stripe directement, il ne
connait que le mode (`test`/`live`) et l'etat local.

**Choix Checkout Sessions plutot que PaymentIntent+Elements** : le
cadrage roadmap parle de PaymentIntent, mais une Checkout Session Stripe
cree un PaymentIntent en interne (one-shot) ou une Subscription (mode
`subscription`) et reste 100% redirect-based, sans SDK JS cote client —
coherent avec la convention "raw fetch, pas de SDK" deja en place pour
PayPal et avec l'UX PayPal existante (redirection puis retour).

## Modeles MariaDB

- Migration `017_stripe_webhook_events.sql` : table `stripe_webhook_events`,
  miroir exact de `paypal_webhook_events` (`UNIQUE event_id`, statut ENUM
  `received|processed|failed|ignored`, `raw_payload JSON`).
- Migration `018_subscriptions_stripe_rail.sql` : generalise `subscriptions`
  par rail plutot que de creer une table separee — `rail ENUM('paypal',
  'stripe') NOT NULL DEFAULT 'paypal'`, `paypal_plan_id` relache en `NULL`,
  ajout `stripe_subscription_id VARCHAR(64) NULL UNIQUE` +
  `stripe_price_id VARCHAR(64) NULL`. Necessaire pour que `/admin/subscriptions`
  et le MRR restent une seule requete cross-rail.
- Migration `019_stripe_offers_and_payment_method.sql` : ajoute
  `stripe_price_id_test` / `stripe_price_id_live` sur `commercial_offers`
  (miroir de `paypal_plan_id_sandbox`/`_live`, nommage `test`/`live` pour
  matcher `STRIPE_MODE`) ; ajoute `payment_method ENUM('paypal','stripe',
  'manual') NULL` sur `commercial_documents`. Cette derniere colonne ferme
  une lacune preexistante : le flow PayPal V0.21 ne persistait jamais le
  rail utilise pour regler une facture (`payment-confirm` ignorait
  totalement le corps de la requete), ce qui rendait impossible la
  colonne "Rail" exigee sur `/admin/payments`.

## Cycle de vie cote serveur

### Paiement one-shot

1. Client clique "Payer via Stripe" sur `/commercial-documents/{id}`. Le
   BFF `POST /api/payments/stripe/create-intent` valide le document
   (`status === "issued"`, montant > 0) puis cree une Checkout Session
   `mode=payment` via `createStripeOneShotCheckoutSession` (metadata
   `document_id` propagee sur la session ET sur le PaymentIntent sous-jacent),
   retourne `approveUrl`.
2. Le client complete le paiement sur la page hostee Stripe. Stripe
   redirige vers `GET /api/payments/stripe/return` — cette route est
   **purement informative** (redirection vers `payment-success`), elle ne
   confirme rien : contrairement a PayPal ou la capture se fait au retour,
   Stripe a deja confirme le paiement serveur-a-serveur avant la
   redirection.
3. Stripe POST le webhook `payment_intent.succeeded` sur
   `POST /api/webhooks/stripe`. Le BFF verifie la signature HMAC locale
   (`verifyStripeSignature`), forward a `POST /internal/webhooks/stripe`.
   `StripeWebhookService` ignore l'evenement si `payment_intent.invoice`
   est non-null (facture d'abonnement, geree par `invoice.paid`), sinon
   extrait `metadata.document_id` et appelle
   `ConfirmPaymentAsync(documentId, correlationId, "stripe", ct)` — mark
   paid + email `payment_confirmed`, `payment_method = 'stripe'`.

### Abonnements

1. **Admin configure une offre mensuelle** : bouton "Créer le prix Stripe"
   sur le formulaire d'offre (`POST /api/admin/catalog/{id}/stripe-price`,
   miroir V0.22.1) — cree un product + price Stripe (`recurring[interval]
   =month`), persiste dans `stripe_price_id_test` ou `_live` selon
   `STRIPE_MODE`.
2. **Client clique "Souscrire"** avec le rail Stripe selectionne. Le BFF
   `POST /api/subscriptions/create` cree une Checkout Session
   `mode=subscription` (`line_items[0][price]=<stripe_price_id>`). A la
   difference de PayPal, **aucune ligne locale n'est creee a cette
   etape** : Stripe ne connait l'ID de la Subscription qu'une fois le
   checkout complete (contrairement a PayPal Billing Subscriptions qui
   cree la ressource `APPROVAL_PENDING` en amont).
3. **Client complete le paiement**, Stripe redirige vers
   `GET /api/subscriptions/stripe/return?offerId=...&session_id=...`. Le
   BFF relit la Checkout Session (`GET /v1/checkout/sessions/{id}`) pour
   obtenir le `stripe_subscription_id`, cree la ligne locale
   `pending_approval` via `POST /internal/portal/subscriptions` puis la
   fait passer a `pending_activation` via `return-approved` (meme
   endpoint que PayPal).
4. **Stripe POST `invoice.paid`** (premiere echeance ou renouvellement) —
   ce seul evenement joue le role de PayPal `BILLING.SUBSCRIPTION.ACTIVATED`
   ET `PAYMENT.SALE.COMPLETED` combines : `StripeWebhookService` active la
   souscription si elle est encore `pending_approval`/`pending_activation`
   (`next_billing_at = J+1 mois`), puis cree systematiquement le document
   de facturation (`CreateBillingDocumentFromOfferAsync`), emet la facture
   BPCE et confirme le paiement (`payment_method = 'stripe'`).
5. **`customer.subscription.deleted`** met le statut local a `cancelled`.

Le set d'evenements webhook geres est volontairement limite a
`payment_intent.succeeded` / `invoice.paid` / `customer.subscription.deleted`
(cf. `docs/ROADMAP.md`) — c'est un choix idiomatique Stripe : `invoice.paid`
est le signal "argent recu" pour un cycle de facturation, qu'il s'agisse
du premier ou d'un renouvellement, ce qui evite de devoir gerer
`customer.subscription.created`/`updated` separement.

## UI

- `/commercial-documents/{id}` (client) : `PayButton` (composant
  generalise, remplace l'ancien `PayPalPayButton`) affiche un radio de
  rail seulement si PayPal ET Stripe sont actifs, defaut Stripe.
- `/services` (client) : `SubscribeButton` recoit `paypalEnabled` /
  `stripeEnabled` calcules a partir de `paypal_plan_id_*` /
  `stripe_price_id_*` de l'offre, meme logique de radio.
- `/admin/payments` : colonne "Rail" (PayPal / Stripe / Manuel / —).
- `/admin/subscriptions` (liste + detail) : badge "Rail" a cote du statut,
  ligne plan/prix conditionnelle sur le rail. MRR HT reste une somme
  cross-rail inchangee (`commercial_offers.price_amount_cents`
  independant du rail).
- Formulaire catalogue admin : champs `stripePriceIdTest`/`stripePriceIdLive`
  en lecture seule + bouton "Créer le prix Stripe pour le mode actif".

## Garde-fous

- `STRIPE_MODE=disabled` par defaut. `live` interdit avant V1.0 beta 1 —
  **contrairement a PayPal**, ce garde-fou est code en dur dans
  `RuntimeConfigurationValidator` (mirroir du bloc BPCE existant) : passer
  `STRIPE_MODE=live` sans `STRIPE_SECRET_KEY`/`STRIPE_PUBLISHABLE_KEY`/
  `STRIPE_WEBHOOK_SECRET` valides fait echouer le demarrage de l'API. Le
  garde-fou equivalent manque aujourd'hui pour `PAYPAL_MODE=live` (jamais
  code, seulement discipline de process) — non retrofit dans ce lot.
- Signature webhook : verification HMAC locale par defaut (pas d'appel
  reseau, contrairement a la verification serveur PayPal), avec fenetre
  de tolerance de replay de 5 minutes. `STRIPE_WEBHOOK_VERIFY=false`
  autorise pour debug local uniquement.
- Idempotence : UNIQUE sur `stripe_webhook_events.event_id`, meme pattern
  insert-avant-dispatch que PayPal.
- Toutes les datetimes en UTC.
- Audits emis (partages avec PayPal, `TargetType: "subscription"`) :
  `subscription.activated`, `subscription.payment_received`,
  `subscription.cancelled`.

## Variables d'environnement (`.env.example`)

| Variable | Role | Defaut |
|---|---|---|
| `STRIPE_SECRET_KEY` | Cle secrete Stripe (`sk_test_...`/`sk_live_...` selon le mode) | requis |
| `STRIPE_PUBLISHABLE_KEY` | Cle publique Stripe | requis |
| `STRIPE_MODE` | `disabled`, `test` ou `live` | `disabled` |
| `STRIPE_WEBHOOK_SECRET` | Secret de signature configure sur le dashboard developer | requis pour la verif signature |
| `STRIPE_WEBHOOK_VERIFY` | `true` defaut, `false` autorise en debug local | `true` |
| `PUBLIC_PORTAL_URL` | Base URL portail (deja V0.21/V0.22) pour success/cancel URLs | requis |

## Limites connues

- Le webhook necessite MariaDB en backend (meme limite que PayPal —
  `MockCommercialRepository.CreateBillingDocumentFromOfferAsync` throw
  `NotSupportedException`).
- En dev local, exposer le portail via tunnel HTTPS pour que Stripe
  puisse pousser sur `https://.../api/webhooks/stripe`, ou utiliser
  `STRIPE_WEBHOOK_VERIFY=false` + requetes synthetiques locales.
- L'activation du mode `live` reste gated V1.0 beta 1 (durcissement + audit),
  garde-fou code en dur cette fois (voir Garde-fous).
