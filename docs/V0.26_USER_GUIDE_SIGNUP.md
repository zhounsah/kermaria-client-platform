# V0.26 Guide utilisateur — inscription self-service

Ce guide decrit l'activation, le parcours visiteur et le workflow admin
de l'inscription self-service livree en V0.26. Le cadrage technique est
dans [V0.26_SELF_SERVICE_SIGNUP.md](V0.26_SELF_SERVICE_SIGNUP.md).

Note 2026-07-18 : le domaine enfant `clients.home.bzh` existe, mais ce
guide decrit toujours le comportement actuel V0.26. L'inscription ne cree
pas encore de compte AD automatiquement ; l'alignement cible est documente
dans [`v0.38/V0.38_SITE_AD_ALIGNMENT.md`](v0.38/V0.38_SITE_AD_ALIGNMENT.md).

## 1. Activation

L'inscription est un **kill switch global** desactive par defaut. Pour
l'activer :

```bash
SIGNUP_ENABLED=true
```

Tant que `SIGNUP_ENABLED=false` :

- le lien « Inscription » n'apparait pas dans l'en-tete du site vitrine ;
- les CTA `/offres` pointent vers `/contact` au lieu de `/signup` ;
- une navigation directe vers `/signup` affiche un message « inscriptions
  fermees » ;
- le BFF `POST /api/signup` repond `403 SIGNUP_DISABLED`.

**Recommandation** : n'activer qu'en test interne tant que
`EMAIL_INTEGRATION_MODE=mock`, et attendre la validation juridique avant
l'ouverture publique (cf. jalon V1.0 RC de la roadmap).

### hCaptcha (obligatoire des l'ouverture)

```bash
HCAPTCHA_SITE_KEY=...        # cle publique, injectee dans le formulaire
HCAPTCHA_SECRET_KEY=...      # cle privee, server-only, jamais exposee
```

- En **Development sans cle**, la verification anti-robot est **sautee**
  (pratique pour tester en local).
- En **production**, l'absence de `HCAPTCHA_SECRET_KEY` fait **echouer**
  chaque soumission (fail-closed) : configurer les deux cles avant
  d'ouvrir les inscriptions.

### Autres variables

```bash
SIGNUP_RATE_LIMIT_PER_IP_PER_HOUR=3
SIGNUP_RATE_LIMIT_PER_EMAIL_PER_24H=1
SIGNUP_VERIFICATION_TOKEN_TTL_HOURS=24
SIGNUP_PASSWORD_SETUP_TOKEN_TTL_HOURS=24
SIGNUP_AUTO_APPROVE=false   # reserve, doit rester false
```

`PUBLIC_PORTAL_URL` doit pointer vers l'URL publique du portail : elle est
utilisee pour construire les liens de verification e-mail et de definition
de mot de passe envoyes par courriel.

## 2. Parcours visiteur

1. **Formulaire** `/signup` : societe, contact, e-mail, telephone
   (facultatif), besoin (facultatif). Un controle hCaptcha valide la
   soumission.
2. **E-mail de verification** : le visiteur recoit un lien
   `signup_verification` valable 24h. En cliquant, son adresse passe en
   `email_verified`. Le message affiche « en attente de validation par
   l'equipe ».
3. **Attente** : aucun acces n'est possible tant qu'un administrateur n'a
   pas approuve la demande.
4. **Approbation** : le visiteur recoit un e-mail `account_approved`
   contenant un lien one-shot vers `/set-password` (valable 24h).
5. **Definition du mot de passe** : le visiteur choisit un mot de passe
   d'au moins 12 caracteres. Son compte est alors actif et il peut se
   connecter via `/login`.

En cas de refus, le visiteur recoit un e-mail `account_rejected` avec le
motif eventuel saisi par l'administrateur.

**Reponse volontairement identique (non-leak)** : que l'adresse soit deja
connue ou non, le formulaire repond toujours « demande enregistree,
verifiez votre boite mail ». Aucune information ne fuite sur l'existence
d'un compte.

## 3. Workflow administrateur

1. Menu `Administration > Relation client > Demandes d'inscription`
   (`/admin/signups`). La liste est filtrable par statut (en attente
   e-mail, verifiees, approuvees, refusees).
2. Ouvrir une demande (`/admin/signups/[id]`) pour voir les informations
   transmises, l'adresse IP source et l'etat.
3. **Approuver** (disponible uniquement apres verification e-mail) : cree
   le compte client (`customer` + `portal_user`, statut `active` sans mot
   de passe) et envoie le lien de definition de mot de passe. **Aucun
   compte Active Directory n'est cree** : c'est un acte admin distinct
   (V0.18 / V0.25) a lancer si necessaire.
4. **Refuser** : saisir un motif facultatif (transmis par e-mail) et
   confirmer.
5. Tant que la demande est `approved` et que le mot de passe n'a pas encore
   ete defini, la fiche detail permet aussi :
   - **Initialisation par mes soins** : l'admin saisit directement un mot de
     passe portail pour le client ;
   - **Renvoi du mail de reinitialisation** : un nouveau lien one-shot
     `/set-password` est genere et renvoye.

Chaque etape est tracee dans le journal d'audit (`signup.submit`,
`signup.verify_success`, `signup.approved`, `signup.rejected`,
`signup.password_set`, ...).

## 4. Test manuel de bout en bout (mode mock)

Avec `EMAIL_INTEGRATION_MODE=mock` et `SIGNUP_ENABLED=true` :

1. Soumettre `/signup`. Recuperer le lien de verification dans le journal
   `/admin/email-log` (l'e-mail mock y est consigne avec son corps).
2. Ouvrir le lien `/signup/verify?token=...` : la demande passe en
   `email_verified`.
3. Dans `/admin/signups/[id]`, cliquer « Approuver ». Recuperer le lien
   `/set-password?token=...` dans `/admin/email-log`.
4. Ouvrir le lien, definir un mot de passe (>= 12 caracteres).
5. Se connecter via `/login` avec l'e-mail et le mot de passe choisis.
6. Variante admin : avant l'etape 4, verifier que la fiche detail propose
   bien soit une initialisation directe du mot de passe, soit un renvoi du
   mail tant que le compte reste sans mot de passe.

## 5. Securite

- Jetons (verification et mot de passe) : aleatoires 32 octets, stockes
  **uniquement en hash SHA-256**, TTL 24h, one-shot (hash efface apres
  usage pour empecher tout rejeu).
- Anti-bot en defense en profondeur : honeypot, controle de temps de
  remplissage minimal, hCaptcha serveur, rate limit IP.
- Aucun mot de passe en clair n'est envoye par e-mail ni journalise.
- Aucun acces au portail authentifie tant que le mot de passe n'est pas
  defini.

## 6. Limites connues

- Purge automatique des demandes expirees non incluse dans ce lot (a
  ajouter en script ops avant l'ouverture publique).
- Un signup = un `customer` + un `portal_user` ; les utilisateurs
  supplementaires d'un meme client restent geres par l'admin.
- Pas de federation SSO, pas de modification self-service apres signup.
