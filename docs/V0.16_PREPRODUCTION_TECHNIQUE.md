# V0.16 Preproduction Technique

## Objectif

La V0.16 prepare une preproduction technique bornee autour de trois axes :

- deploiement et validation avant mise en trafic ;
- observabilite exploitable sans fuite d'information sensible ;
- sauvegarde et restauration MariaDB sous Windows PowerShell.

Le perimetre reste strict :

- architecture inchangee `browser -> WEBPORTAL/BFF -> API-INTERNAL -> MariaDB` ;
- aucun acces direct WEBPORTAL -> MariaDB ;
- `INTERNAL_API_URL` strictement server-only ;
- aucune session dans `localStorage` ou `sessionStorage` ;
- aucun secret journalise ;
- `AD_INTEGRATION_MODE=disabled` conserve ;
- aucun paiement reel, aucune facturation legale, aucun e-mail, SMS, push,
  WebSocket, provisioning ou action admin destructive.

## Variables de preproduction

Variables obligatoires communes :

| Variable | Attendu V0.16 | Notes |
|---|---|---|
| `NODE_ENV` | `production` | WEBPORTAL |
| `ASPNETCORE_ENVIRONMENT` | `Production` | API-INTERNAL |
| `DOTNET_ENVIRONMENT` | `Production` | Doit rester coherent avec ASP.NET Core |
| `SERVICE_AUTH_TOKEN` | secret fort injecte hors Git | Identique sur WEBPORTAL et API-INTERNAL |
| `LOG_LEVEL` | `Information` ou plus strict | Ne jamais logger les secrets |

Variables WEBPORTAL :

| Variable | Attendu V0.16 | Notes |
|---|---|---|
| `INTERNAL_API_URL` | URL privee http(s) valide | Jamais `NEXT_PUBLIC_*` |
| `ALLOW_LOCAL_INTERNAL_API_URL` | `false` par defaut | `true` seulement avec derogation explicite |
| `SESSION_COOKIE_NAME` | nom de cookie dedie | Cookie `HttpOnly`, `SameSite=Lax` |
| `SESSION_COOKIE_SECURE` | `true` | Obligatoire en preproduction |

Variables API-INTERNAL / MariaDB :

| Variable | Attendu V0.16 | Notes |
|---|---|---|
| `SQL_PROVIDER` | `mariadb` | |
| `SQL_HOST` | hote MariaDB prive | |
| `SQL_PORT` | port MariaDB prive | Typiquement `3306` |
| `SQL_DATABASE` | base preproduction dediee | |
| `SQL_USERNAME` | compte SQL dedie | Privileges minimaux |
| `SQL_PASSWORD` | secret fort injecte hors Git | Jamais de chaine de connexion suivie |
| `SESSION_DURATION_MINUTES` | valeur bornee | Exemple `60` |
| `LOGIN_MAX_FAILURES` | valeur bornee | Exemple `5` |
| `LOGIN_LOCKOUT_MINUTES` | valeur bornee | Exemple `10` |
| `AD_INTEGRATION_MODE` | `disabled` | Integration reelle desactivee |

Variables interdites en preproduction :

- `DEMO_PORTAL_EMAIL`
- `DEMO_PORTAL_PASSWORD`
- `DEMO_INTERNAL_ADMIN_EMAIL`
- `DEMO_INTERNAL_ADMIN_PASSWORD`
- toute variante `NEXT_PUBLIC_INTERNAL_API_URL`
- toute variante `NEXT_PUBLIC_SERVICE_AUTH_TOKEN`

Variables optionnelles pour les scripts operatoires :

| Variable | Usage |
|---|---|
| `API_INTERNAL_BASE_URL` | cible de `npm run check:health` |
| `WEBPORTAL_BASE_URL` | cible de `npm run check:health` |
| `KERMARIA_BACKUP_DIR` | dossier de sortie de `backup-mariadb.ps1` |

## Checklist de predeploiement

1. `git status` est propre et aucun dump ou secret n'est present.
2. Les secrets exposes auparavant ont ete tournes hors depot.
3. Les variables de preproduction ont ete injectees hors Git.
4. `AD_INTEGRATION_MODE=disabled`.
5. `SESSION_COOKIE_SECURE=true`.
6. `INTERNAL_API_URL` pointe vers API-INTERNAL cote serveur uniquement.
7. `SERVICE_AUTH_TOKEN` est identique sur WEBPORTAL et API-INTERNAL.
8. Les flux reseau restent limites a `WEBPORTAL -> API-INTERNAL -> MariaDB`.
9. Une sauvegarde MariaDB horodatee a ete preparee avant migration.
10. Les artefacts ne contiennent aucun secret, dump ou fichier `.env` suivi.

## Scripts V0.16

Validation statique preproduction :

```powershell
npm run validate:preprod
```

Le script controle :

- la presence des variables obligatoires ;
- l'identification correcte de l'environnement ;
- le caractere server-only de `INTERNAL_API_URL` ;
- l'absence de `localStorage` et `sessionStorage` dans le code web ;
- l'absence de secrets evidents dans les fichiers suivis ;
- la coherence minimale WEBPORTAL -> API-INTERNAL.

Verification active des health checks :

```powershell
$env:API_INTERNAL_BASE_URL="https://api-internal-preprod.example.invalid"
$env:WEBPORTAL_BASE_URL="https://portal-preprod.example.invalid"
npm run check:health
```

Le script exige HTTP 200, `X-Correlation-Id`, un payload JSON coherent et
refuse un contenu sensible.

Validation existante complete :

```powershell
npm run validate
npm run build
npm run check:secrets
npm run validate:mariadb
git diff --check
```

`validate:mariadb` reste opt-in et suppose des variables SQL plus un jeu de
donnees de test explicitement fourni hors Git.

## Observabilite

### API-INTERNAL

- `GET /health` : confirme que l'API repond.
- `GET /health/live` : confirme que le processus est vivant.
- `GET /health/ready` : verifie la configuration et MariaDB si configuree.
- `GET /ready` : alias court de readiness pour la supervision.
- `X-Correlation-Id` et `X-Request-Id` : exposes dans les reponses API.
- logs JSON structures avec debut et fin de requete, code HTTP et duree.

Les payloads de sante ne doivent jamais inclure :

- mot de passe SQL ;
- token interservice ;
- cookie ;
- chaine de connexion ;
- secret AD ;
- stack trace brute.

### WEBPORTAL / BFF

- readiness `GET /api/health/ready` : valide sa configuration serveur puis la
  readiness de l'API interne ;
- messages clients neutres en cas d'erreur interne ;
- journalisation serveur structuree des echecs BFF avec `correlation_id`,
  `status` et `code`, sans secret ;
- aucune exposition navigateur de `INTERNAL_API_URL` ni `SERVICE_AUTH_TOKEN`.

## Diagnostic des incidents frequents

### API indisponible

Symptomes :

- `npm run check:health` echoue sur `API-INTERNAL /health` ou `/ready` ;
- WEBPORTAL retourne `INTERNAL_API_UNAVAILABLE` ;
- logs BFF structures avec `category=internal_api_transport`.

Actions :

1. verifier que le processus API est demarre ;
2. verifier l'URL privee `INTERNAL_API_URL` ;
3. verifier pare-feu, reverse proxy prive et certificat ;
4. relancer `npm run check:health` ;
5. corriger avant mise en trafic.

### MariaDB indisponible

Symptomes :

- `GET /ready` ou `/health/ready` retourne HTTP 503 ;
- check `mariadb=unhealthy` ;
- logs API indiquent un echec de readiness MariaDB sans details de connexion.

Actions :

1. verifier `SQL_HOST`, `SQL_PORT`, `SQL_DATABASE`, `SQL_USERNAME` ;
2. verifier le secret injecte hors Git ;
3. verifier les ACL reseau entre API-INTERNAL et MariaDB ;
4. verifier l'espace disque et l'etat du serveur MariaDB ;
5. retester `/ready` avant de remettre le trafic.

### Variable manquante

Symptomes :

- `npm run validate:preprod` echoue ;
- readiness WEBPORTAL ou API retourne `unhealthy` ;
- l'application refuse de demarrer en Production.

Actions :

1. identifier la variable mentionnee ;
2. corriger l'injection hors Git ;
3. reexecuter `npm run validate:preprod` ;
4. reexecuter `npm run check:health`.

### Erreur de migration

Symptomes :

- migration manuelle en echec ;
- `schema_migrations` incomplet ;
- API indisponible ou comportements incoherents apres migration.

Actions :

1. arreter la mise en trafic ;
2. restaurer la sauvegarde pre-migration si necessaire ;
3. verifier la derniere migration appliquee ;
4. corriger le SQL ou l'ordre d'execution ;
5. rejouer sur une base de test avant nouvelle tentative.

### Timeout BFF -> API

Symptomes :

- erreurs WEBPORTAL cote serveur sans details sensibles cote client ;
- lenteur sur `/api/*` ;
- checks API vivants mais BFF en echec applicatif.

Actions :

1. verifier la latence reseau privee ;
2. verifier la latence MariaDB et les logs de readiness ;
3. verifier la charge API et les codes HTTP finalises ;
4. corriger la cause cote API ou reseau avant de retester le BFF.

## Sauvegarde MariaDB

Sauvegarde horodatee :

```powershell
npm run backup:mariadb
```

Comportement :

- lit `SQL_HOST`, `SQL_PORT`, `SQL_DATABASE`, `SQL_USERNAME` ;
- utilise `SQL_PASSWORD` depuis l'environnement ou une saisie locale ;
- genere un dump `kermaria_mariadb_YYYYMMDD_HHMMSS.sql` ;
- calcule et affiche le hash SHA-256 ;
- recommande un stockage hors depot.

Stockage recommande :

- partage de sauvegarde dedie hors depot Git ;
- retention geree par l'infrastructure ;
- acces limite aux operateurs autorises ;
- chiffrement selon la politique de l'organisation.

## Restauration MariaDB

Restauration sur base de test ou de secours :

```powershell
npm run restore:mariadb -- -DumpPath C:\Backups\Kermaria\kermaria_mariadb_<DATE>.sql -TargetDatabase TEST_WEB_RESTORE -VerifySchema
```

Verification post-restauration recommandee :

1. verifier le code retour de la restauration ;
2. verifier `schema_migrations` ;
3. executer `npm run check:health` contre l'environnement restaure ;
4. tester login client, login admin et isolation de lecture ;
5. comparer les volumes attendus sur `portal_users`, `portal_sessions`,
   `support_requests`, `service_requests`, `audit_logs`.

## Checklist de test de restauration

1. restaurer vers une base distincte, jamais sur la source en premier ;
2. verifier que le dump choisi est le bon et qu'il n'est pas vide ;
3. verifier le hash SHA-256 du dump ;
4. verifier `schema_migrations` ;
5. verifier `/ready` et `/api/health/ready` ;
6. verifier un login client et un login admin ;
7. verifier qu'un `client_user` est refuse sur `/admin` ;
8. verifier qu'aucune donnee sensible n'apparait dans les logs ;
9. documenter la date, la source du dump et le resultat du test.
