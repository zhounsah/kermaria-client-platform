# V0.17 Recette Preproduction

## Objectif

Valider la V0.17 dans un environnement de staging ou de preproduction controlee
avant tag, sans changer l'architecture ni activer de fonctionnalite sensible
hors perimetre.

Architecture attendue :

```text
browser -> WEBPORTAL / BFF -> API-INTERNAL -> MariaDB
```

## Prerequis

- `AD_INTEGRATION_MODE=disabled`
- secrets injectes hors Git
- `npm run validate` execute
- `npm run validate:staging` ou `npm run validate:preprod` execute
- `npm run check:health` execute sur les URLs cibles
- sauvegarde MariaDB disponible avant toute migration

## Scenarios recette client

1. Connexion client valide.
   Attendu : session ouverte, cookie `HttpOnly`, aucune valeur sensible cote
   navigateur.
2. Consultation dashboard.
   Attendu : services, demandes, documents et activite charges sans fuite
   d'URL interne.
3. Consultation profil.
   Attendu : identite et statut du client corrects.
4. Consultation des services.
   Attendu : uniquement les services du client courant.
5. Consultation des factures et documents commerciaux.
   Attendu : seulement les documents du client ; aucun paiement disponible.
6. Consultation d'une demande support puis d'une demande de service.
   Attendu : aucun contenu interne ; statuts et historique coherents.
7. Reponse client sur une demande lui appartenant.
   Attendu : succes sans double soumission ni fuite de secret.
8. Tentative d'acces a une ressource d'un autre client.
   Attendu : ressource introuvable ou acces refuse, jamais un detail etranger.

## Scenarios recette admin

1. Connexion admin valide.
   Attendu : role `internal_admin`, aucune reference client exposee dans la
   session.
2. Consultation `/admin`.
   Attendu : synthese globale disponible, aucune action destructive.
3. Consultation `/admin/customers`.
   Attendu : liste coherente et lien vers la fiche detail.
4. Consultation d'une fiche client admin.
   Attendu : identite, statut, services, demandes, documents commerciaux,
   factures, activite recente et audits filtres pour ce client uniquement.
5. Consultation d'une demande support admin.
   Attendu : notes internes et messages publics separes.
6. Consultation d'une demande de service admin.
   Attendu : aucun provisioning reel ; preparation commerciale informative
   uniquement.
7. Creation ou mise a jour d'un document commercial informatif.
   Attendu : aucun paiement, aucune facture fiscale officielle, aucun e-mail.
8. Consultation des audits.
   Attendu : correlation id, action et resultat sans token, cookie, mot de
   passe ni chaine de connexion.

## Scenarios securite

1. Verifier les headers WEBPORTAL.
   Attendu : `nosniff`, `DENY`, CSP, `Referrer-Policy`,
   `Permissions-Policy`, `Cross-Origin-Opener-Policy`,
   `Cross-Origin-Resource-Policy`, `X-Robots-Tag`.
2. Verifier le cookie de session.
   Attendu : `HttpOnly`, `Secure` hors developpement local, `SameSite`
   conforme.
3. Verifier qu'aucun secret n'apparait dans le bundle client.
4. Verifier que `INTERNAL_API_URL` reste server-only.
5. Tenter un acces admin avec une session client.
   Attendu : refus.
6. Tenter un acces avec un identifiant mal forme sur une route admin detail.
   Attendu : `INVALID_REQUEST`.
7. Tenter un acces croise entre deux clients.
   Attendu : pas de fuite de donnees.

## Scenarios erreurs

1. Login invalide.
   Attendu : message neutre.
2. Session expiree ou revoquee.
   Attendu : refus propre sans detail interne.
3. API-INTERNAL indisponible.
   Attendu : erreur controlee cote WEBPORTAL, pas d'URL interne exposee.
4. Readiness en echec.
   Attendu : HTTP 503 et correlation id.
5. Payload JSON invalide.
   Attendu : `INVALID_REQUEST`.

## Criteres d'acceptation

- architecture inchangée ;
- `WEBPORTAL` sans acces SQL direct ;
- isolation `customer_id` confirmee ;
- pas de secret cote client ni dans les logs consultes ;
- cookies et headers conformes ;
- AD reelle toujours desactivee ;
- pas de paiement, e-mail, SMS, push, WebSocket, provisioning ou action
  destructive ajoutes ;
- validations techniques et recette fonctionnelle reussies.

## Checklist finale avant tag

1. `git status`
2. `npm run check:secrets`
3. `npm run validate`
4. `npm run validate:staging` ou `npm run validate:preprod`
5. `npm run validate:mariadb` si disponible
6. `npm run build`
7. `npm run check:health`
8. `git diff --check`
9. revue des logs sans secret
10. verification recette client et admin
11. verification restauration MariaDB sur base dediee de test
