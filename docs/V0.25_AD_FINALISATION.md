- `ux: messages 404 specifiques pour PORTAL_DATA_NOT_FOUND et codes AD`
  — texte explicite pour `PORTAL_DATA_NOT_FOUND` (cas typique :
  client cible inexistant), `AD_OBJECT_NOT_FOUND` et
  `AD_NO_LINK_FOR_USER`.
- `ux: auto-scroll vers le message d'action sur la fiche AD admin` —
  `useEffect` + `scrollIntoView({ behavior: "smooth", block: "center" })`

## Objet

La V0.18 a ouvert le mode `controlled_write` borne a
`OU=TEST_SITE_WEB,DC=home,DC=bzh`. La V0.19 a durci la securite
(validation DN, CSRF). La V0.25 finalise le perimetre AD avant la
bascule production :

1. reactiver le **changement de mot de passe AD cote client**, prepare
   en V0.9 puis desactive ;
2. completer le **provisioning AD admin** dans l'OU de test (creation
   d'utilisateurs/groupes au-dela de ce qui est expose en V0.18) ;
3. ecrire et documenter la **procedure de sortie progressive de
   `OU=TEST_SITE_WEB`** vers une OU de production reelle (prerequis
   technique de la V1.0 RC, mais aucune execution avant l'arrivee du
   R740xd).

La V0.25 reste sur SRV-01/02. **Aucune ecriture hors
`OU=TEST_SITE_WEB`** n'est ouverte dans ce jalon : seul le cadre
operationnel et la procedure sont prepares.

## Ordre compte portail / compte AD

Decision 2026-06-28 : **le compte portail est cree avant le compte AD**.

Sequence applicable a V0.26 (creation de compte self-service) et a
toute creation manuelle admin :

1. Admin approuve la demande -> creation `customers` + `portal_users`
   (statut `active`), email `account_approved` envoye avec lien de
   definition de mot de passe.
2. Le client peut se connecter au portail sans compte AD (acces aux
   pages publiques d'achat, devis, support, paiements one-shot).
3. Quand l'usage justifie un compte AD (VPN, RDS, partage fichier),
   l'admin lance separement le provisioning AD via
   `/admin/customers/[ref]` (brique 2 V0.25), qui cree l'utilisateur
   sous `OU=Users,OU=<CUSTOMER_REFERENCE>,...` et lie l'objet via
   `customer_ad_links`.
4. Le client active alors son acces AD :
   - soit l'admin lui communique un mot de passe initial hors bande,
   - soit le mecanisme V0.25 brique 1 (page `/password`) permet au
     client de definir son mot de passe AD lui-meme apres une
     premiere authentification portail validee.

Cette sequence **portail d'abord, AD ensuite** evite tout couplage
fort : la V0.26 n'a pas a savoir parler a AD, et l'AD n'est jamais un
prerequis pour ouvrir un compte client.

## Architecture

Inchangee :

```text
browser -> WEBPORTAL / BFF -> API-INTERNAL -> Active Directory
                                              (OU=TEST_SITE_WEB)
```

Aucune nouvelle dependance externe. Pas de migration MariaDB attendue
(les tables `customer_ad_links` et l'audit `audit_log` existant
suffisent).

## Brique 1 : changement de mot de passe AD client

**Statut : livree 2026-06-30.**

Implementation :
- Nouveau flag `AD_PASSWORD_CHANGE_ENABLED=true|false` (defaut `false`)
  + `AD_PASSWORD_RATE_LIMIT_PER_15MIN=3` (defaut). Lus dans
  [`AdPasswordRuntimeConfiguration.cs`](../apps/api-internal/Data/Configuration/AdPasswordRuntimeConfiguration.cs).
- Rate limiter in-memory singleton
  [`AdPasswordRateLimiter.cs`](../apps/api-internal/Services/ActiveDirectory/AdPasswordRateLimiter.cs) :
  3 echecs en 15 min -> lock 15 min (utilisateur cible, pas IP a ce
  stade). Pas de persistance — un restart reset le compteur, acceptable
  pour V0.25.
- `IActiveDirectoryService.ChangeUserPasswordAsync` + impls :
  - Mock : simule succes (sauf si current==new -> AD_PASSWORD_POLICY_VIOLATION) ;
  - Ldap : `DirectoryEntry.Invoke("ChangePassword", current, new)`,
    catch `TargetInvocationException` -> message generique
    AD_PASSWORD_CHANGE_FAILED (pas de leak du motif AD : mot de passe
    actuel incorrect vs policy violation vs historique restent
    indistinguables cote client) ;
  - Disabled : AD_INTEGRATION_DISABLED.
- Endpoint API-INTERNAL `POST /internal/profile/password` :
  - Resolve session client (refuse admin) ;
  - Refuse si `AD_PASSWORD_CHANGE_ENABLED=false` (audit
    `ad.password_change` outcome=refused reason=AD_PASSWORD_CHANGE_DISABLED) ;
  - Refuse si rate limiter locked (audit reason=AD_PASSWORD_CHANGE_LOCKED,
    HTTP 429) ;
  - Resolve AD link via nouvelle methode
    `IActiveDirectoryLinkRepository.FindUserLinkByEmailAsync` : match
    `customer_ad_links.user_principal_name == portal_users.email` (case
    insensitive). Si pas de match -> 404 AD_NO_LINK_FOR_USER ;
  - Appelle `ChangeUserPasswordAsync(link.CustomerReference, link.SamAccountName, ...)` ;
  - Sur succes : reset rate limiter, audit success ;
  - Sur echec : `RegisterFailure`, audit avec code AD reel + verif si
    nouveau lock -> remappe en AD_PASSWORD_CHANGE_LOCKED 429 si oui ;
  - Aucun mot de passe (actuel ou nouveau) n'est journalise.
- BFF route [`/api/profile/password`](../apps/webportal/app/api/profile/password/route.ts)
  via `handlePortalPayloadMutation` (pas de CSRF cote portail mutations,
  le mot de passe actuel sert d'anti-CSRF + cookie sameSite).
- Page [`/password`](../apps/webportal/app/password/page.tsx) :
  server-render verifie `process.env.AD_PASSWORD_CHANGE_ENABLED` —
  si `false` -> `DisabledActionNotice` comme avant (kill switch
  observable cote page) ; si `true` -> rend
  [`PasswordChangeForm`](../apps/webportal/components/PasswordChangeForm.tsx)
  (client component : 3 inputs masques avec autoComplete adequat, valide
  confirmation == nouveau et nouveau != actuel cote client, POST JSON,
  affiche message serveur generique).
- Tests : `scripts/verify-ad-security-contract.mjs` etend les
  assertions (route passe par handlePortalPayloadMutation, ne journalise
  pas le mot de passe, formulaire en `type="password"` + pas de
  `localStorage`, page lit le flag).

### Ecart vs cadrage initial brique 1

- Pas de lock IP separe : seul l'utilisateur est tracke. Suffisant tant
  que SMTP / VPN ne sont pas exposes publiquement. A re-evaluer en
  V1.0 RC si exposition publique.
- Liaison portal_user -> AD object via convention email == UPN au lieu
  de colonne `portal_users.ad_object_guid`. Choix pragmatique pour
  eviter une migration en V0.25. A migrer si la convention email != UPN
  apparait en prod.
- Pas de policy mot de passe cote API (regle du cadrage respectee :
  seul AD du domaine valide).

## Brique 1 (specification originale) — historique du cadrage 2026-06-28

### Contexte

La page `/password` cote portail existe deja (preparee V0.9). Elle est
desactivee depuis : le bouton "Changer le mot de passe" affiche un
message "Indisponible". Cote API-INTERNAL, l'endpoint correspondant est
soit absent, soit court-circuite. A verifier dans
`apps/webportal/app/password/page.tsx` et `Program.cs`.

### Perimetre V0.25

- reactivation conditionnee par `AD_INTEGRATION_MODE=controlled_write`
  et un nouveau flag `AD_PASSWORD_CHANGE_ENABLED=true|false` (defaut
  `false` pour ne pas casser le comportement actuel) ;
- formulaire `/password` actif uniquement si le client a un lien AD
  valide (`customer_ad_links` row presente, `account_disabled` = false) ;
- re-authentification obligatoire (mot de passe actuel + nouveau mot
  de passe + confirmation) AVANT envoi a l'API ;
- **politique de mot de passe deleguee a AD** : aucune regle de
  longueur/complexite/historique cote API. La policy du domaine AD est
  l'unique source de verite. Le formulaire affiche le message
  d'erreur AD generique ("Le mot de passe ne respecte pas la politique
  du domaine") sans details specifiques pour ne pas leaker les regles ;
- aucun stockage du mot de passe cote portail, ni en log, ni en cache ;
- audit ecrit (`ad.password_change.attempt` / `.success` / `.failed`
  avec motif AD), pas de payload sensible ;
- rate limit dur cote API (anti-brute-force, indepedant de la policy
  domaine) : 3 echecs en 15 minutes -> blocage temporaire de l'IP
  **ET** du compte, audit `ad.password_change.locked`.

### Routes / contrats

- BFF `POST /api/profile/password` (nouveau) : reception du formulaire,
  forward `X-Service-Auth` vers API-INTERNAL avec corrwlation_id.
- API-INTERNAL `POST /internal/profile/password` (nouveau) : verifie la
  session, verifie le lien AD, appelle l'AD (`DirectoryEntry.Invoke`
  ou `Set-ADAccountPassword` selon le pilote retenu en V0.18),
  ecrit l'audit, ne retourne **aucun message d'erreur AD detaille** au
  client (mapping interne -> message generique).

### Bypass / kill switch

- `AD_PASSWORD_CHANGE_ENABLED=false` desactive le formulaire ET
  l'endpoint cote API. Toute requete refusee avec audit
  `ad.password_change.disabled`.
- En mode `disabled`/`mock`/`read_only`, l'endpoint refuse meme si le
  flag est `true`.

## Brique 2 : provisioning AD admin etendu (OU de test)

**Statut : livree 2026-06-30** (sous-briques 2a / 2b / 2c).

### Sous-brique 2a livree 2026-06-30 — lecture groupes effectifs

- `IActiveDirectoryService.GetUserEffectiveGroupsAsync` ajoute, avec
  implementations Mock / Ldap / Disabled.
- Implementation Ldap utilise
  `(member:1.2.840.113556.1.4.1941:=<userDn>)`
  (LDAP_MATCHING_RULE_IN_CHAIN) pour recuperer l'appartenance directe
  ET transitive aux groupes.
- Endpoint API-INTERNAL :
  `GET /internal/admin/customers/{customerReference}/ad/users/{samAccountName}/groups`
  (audit `admin.customers.ad_users.groups_read`, code succes
  `AD_USER_GROUPS_FOUND`).
- BFF route :
  `GET /api/admin/customers/{customerReference}/ad/users/{samAccountName}/groups`
  (passe par `handleAdminGet` qui exige session + CSRF + role
  `internal_admin`).
- UI : nouvelle SectionCard "Groupes effectifs" au bas de
  `AdminCustomerAdManager`, activee uniquement si un utilisateur est
  selectionne, bouton "Lister les groupes effectifs".
- Defense en profondeur cote service : meme apres la requete LDAP, on
  filtre les groupes hors `CustomerReference` de l'utilisateur cible.

**Ecart vs cadrage initial** : la route est `customer-scoped`
(`/customers/{ref}/ad/users/{sam}/groups`) au lieu de la forme
`GUID-scoped` proposee (`/ad/users/{guid}/groups`). Cohesion avec les
endpoints existants de V0.18/V0.19 (disable, move-to-disabled, create,
membership). Le cas cross-customer (en B2c) acceptera un
`targetCustomerReference` dans le body.

### Sous-brique 2b livree 2026-06-30 — renommage utilisateur

- `IActiveDirectoryService.RenameUserAsync(customerRef, currentSam, request)`
  + impls Mock / Ldap / Disabled. Contrat `RenameAdUserRequest { NewSamAccountName, NewDisplayName, NewUserPrincipalName }`.
- Ldap : `DirectoryEntry.MoveTo(parent, "CN=<newSam>")` pour renommer le
  CN (si change), puis update des proprietes `sAMAccountName`,
  `displayName`, `userPrincipalName`.
- Mock : mutation atomique du `_objectsByDn` + migration des references
  dans `_groupMembers`.
- Codes : `AD_USER_RENAMED` (succes), `AD_USER_RENAME_NOOP` (deja a
  jour), `AD_OBJECT_ALREADY_EXISTS` (conflit nouveau SAM).
- Endpoint API-INTERNAL :
  `POST /internal/admin/customers/{ref}/ad/users/{sam}/rename` (audit
  `admin.customers.ad_users.rename`). Si succes + changement effectif,
  appelle `IActiveDirectoryLinkRepository.RefreshCustomerLinkAsync` pour
  mettre a jour la table `customer_ad_links` (sam, UPN, DN).
- Route BFF : `POST /api/admin/customers/{ref}/ad/users/{sam}/rename`,
  passe par `handleAdminMutation` + `parseAdUserRenamePayload`.
- UI : SectionCard "Renommer l'utilisateur" sous "Groupes effectifs",
  inputs newSam / newDisplayName / newUPN, bouton "Renommer" desactive
  si aucun utilisateur selectionne.

### Sous-brique 2c livree 2026-06-30 — deplacement utilisateur

- `IActiveDirectoryService.MoveUserAsync(currentCustomerRef, sam, request)`
  + impls Mock / Ldap / Disabled. Contrat
  `MoveAdUserRequest { TargetCustomerReference, TargetContainer }`.
- Cas couverts :
  - Users <-> Disabled du meme client (deplacement d'OU sans changement
    de scope client) ;
  - cross-customer A -> B (deplacement scope client, ex. erreur de
    saisie initiale).
- Ldap : `DirectoryEntry.MoveTo(targetParent)` (CN preserve, DN change).
- Mock : mutation atomique + migration references dans `_groupMembers`.
- Codes : `AD_USER_MOVED` (succes), `AD_USER_MOVE_NOOP` (deja a la
  bonne place), `AD_OBJECT_ALREADY_EXISTS` (conflit a la cible).
- Endpoint API-INTERNAL :
  `POST /internal/admin/customers/{ref}/ad/users/{sam}/move` (audit
  `admin.customers.ad_users.move`). Avant l'appel au service, le
  endpoint refuse early si `targetCustomerReference` est specifie mais
  inconnu dans `customers` (404 `PORTAL_DATA_NOT_FOUND`). Sur succes +
  changement, `RefreshCustomerLinkAsync(targetCustomerRef, ...)` migre
  le row `customer_ad_links` vers le nouveau client.
- Route BFF : `POST /api/admin/customers/{ref}/ad/users/{sam}/move`,
  passe par `handleAdminMutation` + `parseAdUserMovePayload`.
- UI : SectionCard "Deplacer l'utilisateur" a cote du renommage, input
  reference cible (defaut = client courant), radio Users/Disabled,
  bouton "Deplacer". `window.confirm()` obligatoire si cross-client.
