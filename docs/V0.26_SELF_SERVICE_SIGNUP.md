## Objet

Aujourd'hui, un compte client est cree manuellement par l'admin via
`/admin/customers/...`. La V0.26 ouvre une route publique d'inscription
self-service avec validation manuelle obligatoire de l'admin avant tout
acces aux services. **Aucun provisioning AD automatique** : la creation
AD reste un acte admin gere via V0.18 / V0.25.

## Architecture

```text
Visiteur anonyme
  -> WEBPORTAL/BFF (page publique /signup)
     -> API-INTERNAL POST /internal/signup
        -> MariaDB (table signup_pending)
        -> Email (token de verification, mode mock par defaut)

Apres clic sur lien email
  -> WEBPORTAL/BFF GET /signup/verify?token=...
     -> API-INTERNAL POST /internal/signup/verify
        -> bascule signup_pending.status = 'email_verified'

Admin
  -> /admin/signups (nouvelle page)
     -> approuver -> creer customer + portal_user (status active) +
                     email account_approved
     -> refuser   -> bascule status = 'rejected' + email account_rejected
```

Aucun acces au portail authentifie tant que l'admin n'a pas approuve.
Aucune creation AD avant V0.25 (ou plus tard sur demande admin).

## Modele MariaDB

### Migration `017_signup_pending.sql` (nouvelle)

```sql
CREATE TABLE signup_pending (
  id              CHAR(36) NOT NULL PRIMARY KEY,
  status          ENUM('email_pending','email_verified','approved','rejected','expired')
                  NOT NULL DEFAULT 'email_pending',
  company_name    VARCHAR(200) NOT NULL,
  contact_name    VARCHAR(200) NOT NULL,
  email           VARCHAR(320) NOT NULL,
  phone           VARCHAR(40) NULL,
  message         TEXT NULL,
  verification_token_hash CHAR(64) NOT NULL,
  verification_token_expires_at DATETIME NOT NULL,
  source_address  VARCHAR(45) NULL,
  user_agent      VARCHAR(500) NULL,
  approved_user_id CHAR(36) NULL,
  approved_at     DATETIME NULL,
  rejected_at     DATETIME NULL,
  rejected_reason VARCHAR(500) NULL,
  created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_signup_email_status (email, status),
  KEY idx_signup_status (status),
  KEY idx_signup_created (created_at)
);
```

Le token est stocke **uniquement en hash** (SHA-256 cote API-INTERNAL).
Aucune table `signup_pending` cote `customers` tant que l'admin n'a
pas approuve.

### Migration `018_email_log_signup_templates.sql` (legere)

Ajoute les 3 templates manquants a `email_messages.template_name` :

- `signup_verification` (lien de verification email)
- `account_approved` (admin a accepte)
- `account_rejected` (admin a refuse, avec motif optionnel)

Templates texte uniquement, suivent la convention V0.21.

## Contrats API

### Public BFF -> API-INTERNAL

- `POST /api/signup` (BFF) -> `POST /internal/signup` :
  - body : `{ companyName, contactName, email, phone?, message?,
    hcaptchaToken }`
  - **hCaptcha obligatoire** (decision 2026-06-28) : le BFF verifie le
    token cote serveur via `POST https://hcaptcha.com/siteverify` avec
    `HCAPTCHA_SECRET_KEY` avant tout traitement. Echec captcha = 400
    avec audit `signup.captcha_failed`, sans creation de
    `signup_pending` ;
  - rate limit : 3 signups par IP / heure, 1 signup par email / 24h ;
  - validation : email format strict, longueur 320 max, anti-bot en
    defense en profondeur (timing + honeypot field + hCaptcha) ;
  - genere token aleatoire (32 bytes), expire 24h, hashe SHA-256 ;
  - declenche email `signup_verification` en mode courant
    (`disabled`/`mock`/`live`) ;
  - reponse : `{ correlationId }`, **pas de leak** sur l'existence
    d'un email deja inscrit (toujours 200 OK meme si email pris).

- `GET /api/signup/verify?token=...` (BFF) -> `POST /internal/signup/verify` :
  - body : `{ token }`
  - bascule `signup_pending.status` de `email_pending` a
    `email_verified` si token valide et non expire ;
  - retourne page "Email confirme, en attente de validation par
    l'equipe" ;
  - token consomme (suppression de la colonne hash apres usage pour
    eviter rejeu).

### Admin BFF -> API-INTERNAL

- `GET /api/admin/signups` (BFF) -> `GET /internal/admin/signups` :
  - liste filtree par statut, pagination simple (limit 50) ;
- `POST /api/admin/signups/{id}/approve` (BFF, CSRF) ->
  `POST /internal/admin/signups/{id}/approve` :
  - cree `customers` + `portal_users` (sans AD) ;
  - genere mot de passe initial aleatoire OU lien d'init de mot de
    passe (a trancher pendant l'implementation — preference pour le
    lien de definition de mot de passe car plus securise) ;
  - declenche email `account_approved` ;
  - audit `signup.approved` ;
- `POST /api/admin/signups/{id}/reject` (BFF, CSRF) :
  - body : `{ reason? }`
  - audit `signup.rejected`, email `account_rejected`.

## Cote portail (UI)

### Pages publiques

- `/signup` : formulaire ; affiche message "Demande envoyee, verifiez
  votre boite mail" apres POST OK ;
- `/signup/verify` : confirme la verification email ; affiche message
  "En attente de validation par l'equipe".

### Pages admin (nouvelles)

- `/admin/signups` : liste des demandes en cours (status
  `email_pending`, `email_verified`) + recentes (rejected, approved) ;
- `/admin/signups/[id]` : detail (donnees du formulaire, statut email,
  date) + boutons Approuver / Refuser.

Sidebar admin : ajout d'un lien "Demandes d'inscription" sous la
section "Relation client".

## Securite

- formulaire CSRF token cote BFF (deja en place pour les mutations
  admin V0.19, etendre au POST `/api/signup` qui est public mais
  toujours soumis a une route POST avec referer check) ;
- **hCaptcha** verifie cote serveur a chaque POST `/api/signup` avant
  toute insertion DB ;
- rate limit IP + email (cf. ci-dessus) ;
- token verification : 32 bytes random (urlsafe base64), TTL 24h,
  one-shot (consomme apres verification reussie) ;
- pas de leak sur l'existence d'un email — reponse identique pour
  email connu ou inconnu (sauf erreur de format) ;
- audit ecrit a chaque etape (`signup.submit` / `.verify_success` /
  `.verify_failed` / `.approved` / `.rejected` / `.expired`) ;
- purge auto des `signup_pending` expires (> 30 jours en statut
  terminal) via job MariaDB ou script ops.

## Limites assumees

- **pas de creation AD automatique** : l'approbation cree seulement le
  `portal_user`. L'admin lance separement la creation AD
  (V0.18/V0.25) si necessaire. Decision 2026-06-28 : sequence
  "portail d'abord, AD ensuite" (cf. [`V0.25_AD_FINALISATION.md`](V0.25_AD_FINALISATION.md)
  section "Ordre compte portail / compte AD") ;
- pas de federation SSO (Google, Microsoft) ;
- pas de signup multi-utilisateurs pour un meme `customer` dans ce
  jalon — un signup = un customer + un portal_user. Les comptes
  utilisateurs supplementaires d'un meme client restent admin only ;
- pas de modification self-service apres signup (le client ne peut pas
  modifier `companyName` lui-meme).

## Implementation livree (2026-07-02)

### Decisions tranchees pendant l'implementation

- **Initialisation du mot de passe = lien de definition** (option
  recommandee du cadrage). L'e-mail `account_approved` contient un lien
  one-shot vers `/set-password?token=...`. Le token est un aleatoire de
  32 octets (base64url), **stocke uniquement en hash SHA-256** dans
  `signup_pending.password_setup_token_hash`, TTL 24h
  (`SIGNUP_PASSWORD_SETUP_TOKEN_TTL_HOURS`), consomme (hash efface) au
  premier usage. Aucun mot de passe en clair ne transite par e-mail ni
  n'est journalise. A l'approbation, le `portal_user` est cree avec
  `password_hash = NULL` : la connexion echoue contre le dummy hash tant
  que le mot de passe n'est pas defini. Longueur minimale du mot de passe
  portail : **12 caracteres** (validee BFF + API ; la politique AD reste
  distincte, cf. V0.25).
- **Numerotation de migration** : le cadrage citait `017_signup_pending`,
  mais `017`/`018`/`019` ont ete consommes par V0.29 (Stripe). La
  migration livree est **`020_signup_pending.sql`**. La migration
  `018_email_log_signup_templates` du cadrage est **abandonnee** :
  `email_messages.template` est un `VARCHAR(64)` libre, aucun changement
  de schema n'est requis pour ajouter des noms de template.
- **hCaptcha** : verifie cote serveur (`POST https://hcaptcha.com/siteverify`
  avec `HCAPTCHA_SECRET_KEY`) dans le BFF `/api/signup` avant tout appel a
  l'API-INTERNAL. En `Development` sans cle configuree, la verification est
  **sautee** pour rester testable en local ; en production, l'absence de
  `HCAPTCHA_SECRET_KEY` fait **echouer** chaque soumission (fail-closed).
  La cle site (`HCAPTCHA_SITE_KEY`) est injectee dans le formulaire ; si
  elle est absente, le widget n'est pas rendu.
- **hCaptcha derriere reverse-proxy (IIS/ARR)** : le parametre optionnel
  `remoteip` n'est transmis a hCaptcha **que si l'IP client est publique**.
  Derriere ARR sur SRV-01, le premier segment de `X-Forwarded-For` peut
  etre une IP privee (client LAN) ou celle du proxy ; l'envoyer comme
  `remoteip` la ferait diverger de l'IP vue par hCaptcha a la resolution
  du widget et rejetterait le siteverify (`CAPTCHA_FAILED`). Une IP non
  publique est donc omise, le token etant verifie seul. Tout echec est
  journalise sur `stderr` (`event: "hcaptcha_verify_failed"` avec
  `error_codes`, `hostname`, `remoteip_sent`, `correlation_id`), sans
  jamais journaliser le secret ni le token. Voir `DEPLOYMENT_WINDOWS.md`
  (pre-requis hCaptcha) pour le provisionnement des cles reelles.

### Chaine livree

- **Migration** `020_signup_pending.sql` : table `signup_pending` avec
  `status ENUM(email_pending|email_verified|approved|rejected|expired)`,
  colonnes hash `verification_token_hash` / `password_setup_token_hash`
  (CHAR(64), nullables, effacees a la consommation), `UNIQUE KEY
  (email, status)`, `approved_user_id` / `approved_customer_id`.
- **API-INTERNAL** : `SignupRuntimeConfiguration` (flags + TTL + rate
  limits), `ISignupRepository` + `MariaDbSignupRepository` (approbation
  transactionnelle : `INSERT customers` + `INSERT portal_users` + `UPDATE
  signup_pending`) + `MockSignupRepository`/`MockSignupStore` (mode dev
  sans MariaDB : cree l'utilisateur dans `MockAuthenticationStore` pour un
  parcours complet testable). `SignupService` porte la logique metier
  (non-leak via `HasRecentSignupOrUserAsync`, hash SHA-256, generation de
  reference client `CLI-XXXXXX`). Endpoints publics `POST /internal/signup`,
  `/internal/signup/verify`, `/internal/signup/set-password` (anonymes,
  `X-Service-Auth`) ; endpoints admin `GET /internal/admin/signups[/{id}]`,
  `POST .../approve`, `POST .../reject`. Templates
  `signup_verification` / `account_approved` / `account_rejected` via
  `EmailDispatchService.SendAdHocAsync` (journalises dans `email_messages`,
  `related_document_id = NULL`).
- **WEBPORTAL** : BFF `POST /api/signup` (gate `SIGNUP_ENABLED`, rate
  limit 3/IP/h, honeypot + timing anti-bot, hCaptcha, relais avec
  `sourceAddress`/`userAgent`), `POST /api/set-password`, et routes admin
  `/api/admin/signups[...]` (`handleAdminGet`/`handleAdminMutation`).
  Pages publiques `/signup`, `/signup/verify`, `/set-password` (chrome
  `PublicShell`) ; admin `/admin/signups` (liste + filtre statut) et
  `/admin/signups/[id]` (detail + approuver/refuser). Lien de nav admin
  « Demandes d'inscription » sous « Relation client ».
- **Audit** : `signup.submit`, `.verify_success`, `.verify_failed`,
  `.approved`, `.rejected`, `.password_set` / `.password_failed`.
- **Tests** : `npm run test:signup`
  (`scripts/verify-signup-contract.mjs`, 27 verifications de contrat).

### Ecarts vs cadrage

- Rate limit e-mail « 1/24h » implemente via une **fenetre de 24h** dans
  `HasRecentSignupOrUserAsync` (toute demande recente ou compte existant
  pour l'e-mail -> reponse d'acceptation identique, sans nouvelle
  insertion) plutot qu'un compteur dedie ; cela sert aussi l'idempotence
  et le non-leak.
- Purge auto des `signup_pending` expires : **non implementee** dans ce
  lot (a ajouter en script ops / job MariaDB avant l'ouverture publique).
- CSRF admin approve/reject : assure par le **meme mecanisme que toutes
  les mutations admin existantes** (`handleAdminMutation` + cookies
  SameSite), pas de traitement CSRF specifique ajoute.
- `SIGNUP_AUTO_APPROVE` : variable **reservee** (lue, defaut `false`),
  aucun code ne bascule dessus dans ce lot.

## Recette utilisateur (2026-07-02)

Parcours complet execute sur environnement local en mode mock puis en
mode `EMAIL_INTEGRATION_MODE=live` via SMTP OVH MX Plan reel
(`ssl0.ovh.net:587` STARTTLS, boite `support@zacharyhounsa.ovh`).

### Ce qui a ete valide

- **Formulaire public** `/signup` : soumission avec message
  « Demande envoyee » -> ligne `email_pending` en base.
- **Verification e-mail** : e-mail `signup_verification` recu (mock puis
  reel), lien one-shot consomme, bascule `email_verified`.
- **Approbation admin** : `/admin/signups` -> detail -> « Approuver et
  creer le compte » -> customer + portal_user crees, reference `CLI-XXXXXX`
  generee, e-mail `account_approved` envoye.
- **Definition mot de passe** : lien `/set-password?token=...` accepte,
  mot de passe (>=12 caracteres) accepte, compte activable.
- **Connexion reelle** : `/login` avec les identifiants nouvellement
  definis reussie (`auth.login success` sur la nouvelle reference
  client), acces au portail confirme.
- **Anti-rejeu** : rechargement d'un lien `/set-password` deja consomme
  -> `signup.password_failed refused TOKEN_INVALID`, aucun impact sur le
  compte deja actif. Comportement attendu.
- **Non-leak** : soumission d'un e-mail deja inscrit -> meme reponse
  « demande enregistree », aucune fuite.
- **Audit complet** : chaine `signup.submit` -> `.verify_success` ->
  `.approved` -> `.password_set` visible dans `/admin/audit-logs`,
  correlation_id propage.

### Ecarts observes non bloquants

- `ActorUserId` de `signup.password_set` reste `Systeme` (l'utilisateur
  n'est pas encore authentifie au moment de definir son mdp).
  `TargetReference` ne porte pas non plus le portal_user_id. Amelioration
  d'observabilite possible : passer le `portalUserId` en
  `TargetReference` pour tracer quel compte a active. Reporte sans
  gravite (l'`email_messages.correlation_id` permet le rapprochement).
