# V0.35.1 — correctif horodatages : NOW() locale → UTC_TIMESTAMP (3e recidive)

Date : 2026-07-09. Correctif applique suite au decalage +2h observe sur le
portail (document commercial « Mise a jour 03:25 » alors qu'il etait 01:28
heure de Paris).

## Symptome

Sur `/commercial-documents/<id>`, un document cree et emis a 01:25 (Paris)
affichait « Créé le 01:25 » (correct) mais « Mise à jour 03:25 » (futur,
+2h). Constate en recette locale (`localhost:3000` → API locale → MariaDB
KERMARIA-SRV-07 `test_web`).

## Diagnostic (preuve en base)

Ligne `commercial_documents` `c48c741a-843d-4379-888b-41ab6e996e4c` :

| colonne | valeur en base | interpretation |
| --- | --- | --- |
| `created_at` | `2026-07-08 23:25:54` | UTC correct (affiche 01:25 Paris) |
| `updated_at` | `2026-07-09 01:25:57` | **heure de Paris stockee brute** (affichee 03:25) |

Le serveur MariaDB est regle sur l'heure de Paris (`@@system_time_zone =
CEST`) : `NOW()` y renvoie l'heure locale, pas l'UTC. Or la convention du
projet (V0.23.2) est : **stockage UTC en base (DATETIME(6)), serialisation
API en ISO 8601 avec suffixe Z, affichage force Europe/Paris cote front**.
Toute ecriture `NOW()` stocke donc du +2h (ete) relu comme de l'UTC.

Origine : `MarkDocumentIssuedAsync` / `MarkDocumentPaidAsync` ecrivaient
`updated_at = NOW(6)` depuis la V0.20 slice C (0b9f2a8). Le fix V0.20
« BPCE timestamps stored in UTC » avait corrige les tables `bpce_*` mais
rate `commercial_documents`. C'est la **3e recidive** de cette famille de
bugs (V0.20 BPCE, V0.21 email log, V0.35.1 ici).

## Reparation des donnees (KERMARIA-SRV-07, base `test_web`)

Regle de decision : une ligne n'est reparee que si le biais +2h est
**prouve** par recoupement avec un horodatage UTC fiable de la meme
operation (`bpce_invoices.validated_at`, ecrit en `UTC_TIMESTAMP(6)`),
ou avec `schema_migrations.applied_at` (UTC, `DateTime.UtcNow`) pour les
seeds. Le recoupement a montre que les ecritures parametrees C#
(`DateTime.UtcNow`) ont toujours ete justes : seules les ecritures
`NOW(6)` (chemins V0.35 utilises depuis le ~2026-07-02 en dev, + seeds
006/009) etaient biaisees. Les documents plus anciens (delta 0 ou negatif
vs BPCE) n'ont **pas** ete touches.

- `commercial_documents` : **8 lignes** avec
  `updated_at = validated_at + 7200 s (±qq ms)` (dont `1c54b987…`,
  motif identique via `created_at`, re-validation BPCE ulterieure) →
  `updated_at = updated_at - INTERVAL 2 HOUR` ;
- `commercial_offers` : **17 seeds** de `009_catalog_articles`
  (`created_at = applied_at(009) + 2h` exactement) → `- INTERVAL 2 HOUR`
  sur `created_at`, et sur `updated_at` uniquement si jamais reedite
  (`updated_at = created_at`) — les editions admin (UTC correct) sont
  conservees. Le seed 006 « Sauvegarde dossier personnel » ne provenait
  pas de la migration (ligne pre-existante du 2026-06-15), non touche ;
- `cart_items` : table vide, rien a reparer ;
- migration `031` : appliquee automatiquement par le runner au prochain
  demarrage de l'API (deploiement SRV-02 / restart API locale).

Verification post-reparation : plus aucun `delta ≈ +7200 s` vs BPCE,
aucun horodatage dans le futur, catalogue realigne sur l'`applied_at`
de la migration. Dump de securite des deux tables pris avant reparation.
