## Cadrage initial (2026-06-28)

## Objet

Aujourd'hui, la route `/` du webportal redirige vers `/login` ou
`/dashboard` selon la session. La V0.27 ajoute une **vitrine publique**
en amont du backend authentifie pour :

1. presenter l'offre commerciale (catalogue V0.15) sans imposer de
   login ;
2. heberger les pages legales obligatoires (mentions legales,
   politique de confidentialite, CGV) — **prerequis V1.0 RC** ;
3. fournir un canal d'entree (`/signup` V0.26, `/login` pour les
   clients existants).

Le site vitrine n'expose **aucune donnee client**. Tout est statique
ou servi en lecture seule via le BFF (catalogue public).

## Architecture

```text
Visiteur anonyme
  -> WEBPORTAL (Next.js)
     /                 (landing)
     /offres           (catalogue public)
     /a-propos
     /contact
     /mentions-legales
     /politique-confidentialite
     /cgv

     /login            (clients existants)
     /signup           (V0.26)

     /dashboard        (auth requise, redirige vers /login si anonyme)
```

Pas d'appel `API-INTERNAL` depuis les pages vitrine **sauf** pour le
catalogue public (deja expose en `/internal/portal/catalog` en V0.15,
on reutilise le meme endpoint sans session).

Les pages legales sont des fichiers statiques (Next.js `app/...` avec
contenu MDX ou simple TSX). Aucune base de donnees.

## Pages a livrer

### `/` landing page

- bandeau hero (proposition de valeur, CTA "Decouvrir l'offre" /
  "Se connecter") ;
- 3 a 4 blocs de contenu (qui sommes-nous, services principaux,
  preuves/garanties, contact) ;
- footer avec liens legaux + lien `/login` + lien `/signup` (visible
  si `SIGNUP_ENABLED=true`) ;
- **pas de carrousel JS**, performances avant tout (LCP < 2s sur
  reseau 3G simule).

### `/offres` catalogue public

- liste lecture seule des offres `active` (cadence `one_time` et
  `monthly` melangees, tri par `displayOrder`) ;
- pas de prix dynamique dans le HTML cache plus de 5 min cote BFF
  (catalogue change peu, on accepte un cache court) ;
- bouton "Demander un devis" sur chaque carte -> `/contact` avec
  `?offer=...` prerempli, ou `/signup` si signup activee.

### `/a-propos`, `/contact`

- Statiques. `/contact` propose un formulaire qui POST vers
  `/api/contact` (nouveau, **pas** vers `/api/signup`) avec rate
  limit + envoi email a une adresse interne en mode mock par defaut.

### Pages legales

- `/mentions-legales`
- `/politique-confidentialite`
- `/cgv`

Contenu redige en interne et hardcode en TSX (decision 2026-06-28 :
pas de CMS). Pas de Sanity, Strapi, Contentful ou autre — toute mise
a jour passe par un commit dans le depot. Squelette TSX pose en
V0.27, contenu reel finalise avant V1.0 RC.

## Detection anonyme vs authentifie

Le `AppShell` actuel affiche la sidebar uniquement si une session
existe. Pour la V0.27 :

- routes publiques (`/`, `/offres`, `/a-propos`, `/contact`,
  `/mentions-legales`, `/politique-confidentialite`, `/cgv`,
  `/signup`, `/signup/verify`, `/login`) : **header public** ;
- routes authentifiees (`/dashboard`, `/services`, `/invoices`,
  `/support`, `/notifications`, `/profile`, `/password`, `/admin/*`) :
  sidebar (comportement V0.23 actuel) ;
- racine `/` change de comportement :
  - **anonyme** -> landing publique ;
  - **client_user** authentifie -> redirect `/dashboard` ;
  - **internal_admin** authentifie -> redirect `/admin`.

Cette logique vit dans `app/page.tsx` (deja present, a refondre).

## Confidentialite et cookies

Decision 2026-06-28 : **aucun analytics, aucun pixel de tracking,
aucun cookie tiers** sur les pages publiques. Position de principe :
on ne traque pas les visiteurs.

Consequences :

- **pas de banniere de consentement cookies** : pas de cookies non
  essentiels = pas d'obligation RGPD de bannière de consentement
  (seuls les cookies de session apres login sont emis, et uniquement
  sur les routes authentifiees) ;
- **pas de Google Analytics, Plausible, Matomo, Cloudflare Insights,
  Microsoft Clarity** ni equivalent ;
- **pas de pixel Meta/LinkedIn/X** ni autre script publicitaire ;
- les seuls cookies emis cote vitrine sont eventuellement le cookie de
  session hCaptcha (cookie tiers strictement necessaire au
  fonctionnement, mentionne dans `/politique-confidentialite`) et le
  cookie CSRF du formulaire `/contact` (si applicable) ;
- la page `/politique-confidentialite` doit clairement enumerer les
  (rares) cookies emis et expliquer qu'aucun traqueur n'est en place.

Si un besoin de mesure d'audience apparait plus tard, la solution
choisie devra etre serverless (logs Caddy/nginx anonymises) ou
self-hostee sans cookies (ex. Plausible self-hosted en mode
cookieless), et faire l'objet d'un nouveau jalon avec re-evaluation
explicite. Pas de bascule discrete depuis V1.0 RC.

## Limites assumees

- **pas de CMS** : contenu hardcode en TSX. Decision 2026-06-28 ;
- **pas de blog** ;
- **pas de multi-langue** : FR uniquement ;
- **pas d'analytics ni pixel tiers** (cf. section ci-dessus) ;
- formulaire `/contact` ne cree pas de ticket support — c'est juste un
  email entrant ;
- `/offres` reste public mais filtre les offres avec
  `billingCadence=monthly` qui n'ont **pas** de plan PayPal cote mode
  actif (sinon le visiteur voit une offre non commandable).
