# V0.27 Site vitrine public

Statut : **a cadrer, faisable sans la cible R740xd**. Doc de cadrage
ecrite le 2026-06-28.

## Objet

Aujourd'hui, la route `/` du webportal redirige vers `/login` ou
`/dashboard` selon la session. La V0.27 ajoute une **vitrine publique**
en amont du backend authentifie pour :

1. presenter l'offre commerciale (catalogue V0.15) sans imposer de
   login ;
2. heberger les pages legales obligatoires (mentions legales,
   politique de confidentialite, CGV) — **prerequis V1.0 RC** ;
3. fournir un canal d'entree (`/signup` V0.26, `/login` pour les
   clients existants).

Le site vitrine n'expose **aucune donnee client**. Tout est statique
ou servi en lecture seule via le BFF (catalogue public).

## Architecture

```text
Visiteur anonyme
  -> WEBPORTAL (Next.js)
     /                 (landing)
     /offres           (catalogue public)
     /a-propos
     /contact
     /mentions-legales
     /politique-confidentialite
     /cgv

     /login            (clients existants)
     /signup           (V0.26)

     /dashboard        (auth requise, redirige vers /login si anonyme)
```

Pas d'appel `API-INTERNAL` depuis les pages vitrine **sauf** pour le
catalogue public (deja expose en `/internal/portal/catalog` en V0.15,
on reutilise le meme endpoint sans session).

Les pages legales sont des fichiers statiques (Next.js `app/...` avec
contenu MDX ou simple TSX). Aucune base de donnees.

## Pages a livrer

### `/` landing page

- bandeau hero (proposition de valeur, CTA "Decouvrir l'offre" /
  "Se connecter") ;
- 3 a 4 blocs de contenu (qui sommes-nous, services principaux,
  preuves/garanties, contact) ;
- footer avec liens legaux + lien `/login` + lien `/signup` (visible
  si `SIGNUP_ENABLED=true`) ;
- **pas de carrousel JS**, performances avant tout (LCP < 2s sur
  reseau 3G simule).

### `/offres` catalogue public

- liste lecture seule des offres `active` (cadence `one_time` et
  `monthly` melangees, tri par `displayOrder`) ;
- pas de prix dynamique dans le HTML cache plus de 5 min cote BFF
  (catalogue change peu, on accepte un cache court) ;
- bouton "Demander un devis" sur chaque carte -> `/contact` avec
  `?offer=...` prerempli, ou `/signup` si signup activee.

### `/a-propos`, `/contact`

- Statiques. `/contact` propose un formulaire qui POST vers
  `/api/contact` (nouveau, **pas** vers `/api/signup`) avec rate
  limit + envoi email a une adresse interne en mode mock par defaut.

### Pages legales

- `/mentions-legales`
- `/politique-confidentialite`
- `/cgv`

Contenu redige par l'utilisateur (a confirmer pendant la phase de
livraison). Squelette TSX deja en place pour valider la structure.

## Navigation

- pas de sidebar (pas de session) ;
- header public minimal : logo `Zachary HOUNSA-HOUNKPA EI` + nav
  horizontale `Offres / A propos / Contact / Connexion` (et `Inscription`
  conditionnel) ;
- footer commun a toutes les pages publiques avec liens legaux ;
- separation visuelle nette avec le portail authentifie (palette ou
  spacing different) pour eviter toute confusion utilisateur.

## Detection anonyme vs authentifie

Le `AppShell` actuel affiche la sidebar uniquement si une session
existe. Pour la V0.27 :

- routes publiques (`/`, `/offres`, `/a-propos`, `/contact`,
  `/mentions-legales`, `/politique-confidentialite`, `/cgv`,
  `/signup`, `/signup/verify`, `/login`) : **header public** ;
- routes authentifiees (`/dashboard`, `/services`, `/invoices`,
  `/support`, `/notifications`, `/profile`, `/password`, `/admin/*`) :
  sidebar (comportement V0.23 actuel) ;
- racine `/` change de comportement :
  - **anonyme** -> landing publique ;
  - **client_user** authentifie -> redirect `/dashboard` ;
  - **internal_admin** authentifie -> redirect `/admin`.

Cette logique vit dans `app/page.tsx` (deja present, a refondre).

- structured data JSON-LD `Organization` sur `/` ;
- title `Zachary HOUNSA-HOUNKPA EI - Espace client professionnel` par

## Variables d'environnement nouvelles

- `PUBLIC_VITRINE_ENABLED=true|false` (defaut `false` pendant le
  developpement, `true` en preprod et V1.0 RC) ;
- `CONTACT_FORM_RECIPIENT=ops@...` (destinataire interne du formulaire
  `/contact`).

Si `PUBLIC_VITRINE_ENABLED=false`, la racine `/` garde le comportement
V0.23 (redirect /login ou /dashboard).

## Ordre de livraison propose

1. **Pages legales (squelette)** : `/mentions-legales`,
   `/politique-confidentialite`, `/cgv` en TSX statique avec contenu
   placeholder. Permet de valider le layout public.
2. **Header public + footer** : nouveau composant `PublicShell.tsx`
   parallele a `AppShell.tsx`, choix dans `layout.tsx` selon la route.
3. **Page `/`** : landing avec hero + 3 blocs, CTA conditionnels.
4. **Page `/offres`** : reutilise `getPublicCatalog` (existe deja pour
   client), lecture seule, sans auth.
5. **Page `/contact`** : formulaire + route BFF `POST /api/contact` +
   template email `contact_form` (mode `mock`).
6. **SEO** : sitemap, structured data, meta.
7. **Bascule `PUBLIC_VITRINE_ENABLED=true`** apres recette interne.

## Tests

- `npm run test:vitrine` (nouveau, `scripts/verify-vitrine-contract.mjs`) :
  - HTTP 200 sur toutes les routes publiques ;
  - pas d'appel `INTERNAL_API_URL` depuis les pages publiques sauf
    `/offres` ;
  - pas de fuite de donnees session (aucun cookie de session emis sur
    une route publique) ;
  - `robots.txt` interdit bien `/admin`, `/dashboard`, etc.
- Lighthouse local (objectif Performance >= 85, Accessibility >= 95) ;
- recette manuelle des pages legales avec contenu final.

## Limites assumees

- **pas de CMS** : contenu hardcode en TSX/MDX. Acceptable pour V0.27,
  a remplacer par un CMS headless si besoin V1.x ;
- **pas de blog** ;
- **pas de multi-langue** : FR uniquement ;
- **pas d'analytics tiers** dans ce jalon (a evaluer apres consentement
  RGPD avec banniere cookies, V1.x) ;
- formulaire `/contact` ne cree pas de ticket support — c'est juste un
  email entrant ;
- `/offres` reste public mais filtre les offres avec
  `billingCadence=monthly` qui n'ont **pas** de plan PayPal cote mode
  actif (sinon le visiteur voit une offre non commandable).
